1. 奖励信号全景：三分谱系与 2017→2026 的钟摆

1.1 一切后训练的核心问题：奖励从哪里来

RL 后训练的全部分歧，归根结底是一个问题：「好」的定义由谁给出、以什么形式给出。2026 年的从业者面对的是一个三分谱系：

奖励源形式适用域优点致命弱点
可验证奖励（verifiable reward / RLVR） 规则、单元测试、符号求解器、字符串匹配 数学、代码、逻辑、格式约束 便宜、确定性、不易被语义攻击 只覆盖可验证域；verifier 本身仍可被 hack（见 pitfalls §1 的套壳/泄漏案例）
Rubric 奖励（rubric / checklist reward） 把「好」分解为一组可逐条核验的标准，由 LLM judge 或确定性 verifier 逐条打分再聚合 医疗问答、创意写作、开放指令等「难验证但可分解」的域 可解释、可针对性防 hack（写入 prescriptive 条目）、小 judge 也能用 rubric 设计与维护成本高；rubric 自身覆盖不全就是新的 hacking 面
神经 RM（标量 BT RM / PRM / GenRM） 从偏好数据学出的打分器：标量头输出分数，或生成式模型输出 critique + 判断 通用对话、风格、helpfulness 等无法分解的整体偏好 覆盖面最广、自动化程度最高、标量 RM 推理极便宜 静态 proxy 必被过优化（§3）；分布外失真；可被对抗攻击（§8）
从业者核心判断力（2026 共识） 纯标量 BT RM 在可验证域已被 RLVR 取代，在开放域被「GenRM / rubric reward + 推理时扩展」挤压；但它因为便宜、低延迟、不走语义通道因而不易被 master-key 式语义攻击，仍是 Best-of-N 重排、数据过滤、偏好数据预筛的主力。「什么场景用哪类奖励源」是本文反复回答的问题，最终决策表见 §11。

2. Bradley-Terry 标量 RM：理论、实现与训练纪律

2.1 BT 模型与损失函数

Bradley & Terry (1952) 的成对比较模型假设每个对象有一个潜在「实力」分数，胜率由分数差的 sigmoid 给出。映射到 RLHF：给定 prompt $x$ 与两个回复 $y_w \succ y_l$，

$$ P(y_w \succ y_l \mid x) = \frac{e^{r(x,y_w)}}{e^{r(x,y_w)} + e^{r(x,y_l)}} = \sigma\big(r(x,y_w) - r(x,y_l)\big) $$

对偏好数据做极大似然，得到标准 RM 损失（即 negative log-sigmoid of score difference）：

$$ \mathcal{L}_{\mathrm{RM}}(\theta) = -\,\mathbb{E}_{(x,y_w,y_l)\sim\mathcal{D}}\Big[\log \sigma\big(r_\theta(x,y_w) - r_\theta(x,y_l)\big)\Big] $$

三个常被忽略的理论性质：

  • 只学相对差，不学绝对值。对所有 $r$ 加同一常数（甚至每个 prompt 加不同常数），损失不变。所以不同 prompt 之间的 RM 分数不可比，把原始 RM 分数当全局质量度量是常见误用；这也是 §10 中 reward 归一化必须做的根本原因。
  • K-wise 推广是 Plackett-Luce。对一个 prompt 的 $K$ 个回复全排序 $\tau$：$P(\tau) = \prod_{k=1}^{K} \frac{\exp r(x, y_{\tau(k)})}{\sum_{j=k}^{K}\exp r(x,y_{\tau(j)})}$。Zhu et al.（2301.11270）给出了 pairwise 与 K-wise 比较下 RM 学习的统计理论。实践中 InstructGPT 不直接用 PL 损失，而是把排序拆成 $\binom{K}{2}$ 个 BT 对（见 2.2）。
  • margin 变体。Llama 2 利用标注的四档偏好强度，在损失中加离散 margin：$-\log\sigma(r_w - r_l - m(r))$，差异大的对用大 margin。但 Llama 3 在数据规模化后观察到 margin 增益递减而移除——margin loss 并非必需，数据规模与质量上去之后它是噪声放大器。

BT 与 DPO 的闭式对应（policy 隐式就是 RM）的完整推导见 derivations.html §5-6，此处不重复。

2.2 实现细节（每一条都有出处，每一条都有人踩过坑）

  • 架构：标准做法是 AutoModelForSequenceClassification 式的「LM 骨干 + 标量线性头」，取最后一个非 padding token 的 hidden state 过线性头输出标量。RM 通常从 SFT/chat checkpoint 初始化（Llama 2 明确从 chat checkpoint 初始化，让 RM 与 policy「知识同源」）；新加的 reward head 用小方差高斯初始化（HF 博客 The N Implementation Details of RLHF with PPO 复现了 OpenAI 的具体做法）。
  • $\binom{K}{2}$ 同 batch 是 InstructGPT 的关键 trick。标注时给标注员 $K=4$ 到 $9$ 个回复做排序，产生 $\binom{K}{2}$ 个比较对。这些对高度相关：如果 shuffle 进整个数据集分散训练，一个 epoch 就会过拟合。正确做法是把同一 prompt 的所有比较对作为单个 batch 元素一起前向——每个回复只需一次前向，验证准确率与 log loss 都显著更好。这是所有自研 RM 训练管线必须复刻的细节。
  • RM 尺寸不必跟 policy 对齐。InstructGPT 全程用 6B RM 服务 175B policy：省算力，而且 175B RM 训练不稳定、不适合再拿去初始化 RL 的 value function。「RM 必须和 policy 一样大」是没有依据的迷信；真正的 scaling 收益见 §3 的 Gao 系数分析。
  • 双 RM 解耦目标冲突。Llama 2 训练 helpfulness RM 与 safety RM 两个独立模型，因为单 RM 在两个有张力的目标上都难以做好。多目标解耦的现代形态是 ArmoRM（2406.12845）：多目标回归头 + MoE gating，按上下文自动选择目标权重。

2.3 训练纪律：三条铁律

铁律 1：RM 只训 1 个 epoch。 InstructGPT 与 Llama 2 都报告多 epoch 迅速过拟合（validation loss 恶化）。注意 InstructGPT 的教训更细：即使只训一个 epoch，比较对组织方式不对（没做 $\binom{K}{2}$ 同 batch）照样过拟合。
铁律 2：held-out 准确率 65–75% 即正常水平，不要盲目刷高。 锚点数字（InstructGPT）：训练标注员相互一致率 72.6±1.5%，held-out 标注员 77.3±1.3%；按标注员分组做 5-fold 交叉验证，RM 预测 held-out 组偏好的准确率 69.6±0.9%。人类一致率就是 RM 准确率的天花板量级——RM 准确率逼近或超过它，往往说明评测集分布太窄或泄漏，而非 RM 真的更好。
铁律 3：调参重点在数据而非超参。 InstructGPT 报告 RM 对学习率不敏感。RM 质量的第一杠杆是偏好数据的质量与组织（§4），第二是底座选择，超参排最后。

2.4 校准性：RM 分数什么时候能当概率用（HH-RLHF）

BT 损失训出来的不只是排序器——按 §2.1 的模型假设，分数差 $\Delta := r_w - r_l$ 经 sigmoid 后是一个概率预测：$\sigma(\Delta)$ 应当等于真实偏好胜率。校准（calibration）检验的就是这件事：在 held-out 偏好对上按 $\Delta$ 分桶，画「经验准确率 vs 理论曲线 $1/(1+e^{-\Delta})$」。Anthropic HH-RLHF（2204.05862，§3.3 Calibration of Preference Models and Implications for RL）是第一份系统的 RM 校准分析，三个结论至今有效：

  • 校准可以做得很好，但混合目标会拖累它。只在 helpfulness 数据上训的 PM 校准曲线几乎贴着理论线；helpful + harmless 混训的 PM 则轻微 under-confident。目标张力连校准都会污染——与 Llama 2 拆双 RM（§2.2）是同一动机的两种表现。
  • 高分区校准系统性退化。把比较对限制到「两个样本 PM 分数都高于阈值」的子集，准确率随阈值上升而下降：高分区里同样的 $\Delta$ 对应的真实胜率差越来越小。论文给的三个原因：区分两个都很精致的回复本身更难、训练集里非 generic 的高质量样本稀少、高质量对的分数本就接近。
  • 准确率随规模 log-linear 改善。13M→52B 七档模型上，PM 准确率随模型尺寸与偏好数据量都近似 log-linear 提升（同文 §3.2）——与 Gao 系数分析（§3.1）互证：scaling 平滑买到改善，但买不到质变。
实战推论：分数 ≠ 概率，高分区尤甚 RL 中后期 policy 恰好把输出推进了校准最差的高分区——proxy 分数还在涨，但每一分对应的真实质量增益在缩水。HH 由此明确预期：policy 达到足够高 PM 分数后，继续 RLHF 训练收益递减（diminishing returns），「this also motivates online training」——这正是 §3 过优化曲线的校准视角，也是 HH 走 online 迭代 RLHF（§4.3）的直接动机。操作上：凡是隐含「分数 = 校准概率」的用法——BoN 里拿 $\sigma(\Delta)$ 当胜率估计、拿分数阈值筛「高置信」数据、把分数差直接读作质量差——都必须先画校准曲线（重点单看高分区）再决定信多少；校准曲线应与 §2.3 的 held-out 准确率一起列入 RM 验收项。

2.5 标量 RM 的固有局限（为什么后面需要 §6/§7）

  • 不可解释：一个标量给不出「为什么」，错了无法定位是数据噪声、风格偏置还是真实分歧。
  • 表达力瓶颈：把多维质量压缩成一维分数，注定在「正确但难看 vs 漂亮但错误」这类组合上失真（RM-Bench 2410.16184 系统性地度量了 RM 对内容细微差异的敏感度与对风格偏置的抵抗力：风格干扰下 SOTA RM 平均准确率仅 46.6%，低于 50% 的随机水平）。
  • 静态 proxy + 漂移的 policy：RL 中后期 policy 输出移出 RM 训练分布，RM 评分失真。这是 §3 过优化与 §4.3 迭代式 RLHF 的共同根源。
  • 无法用推理算力换判断质量：标量 RM 一次前向定生死；GenRM 可以多采样投票、拉长 critique（§6.3、§6.5），这是两者最本质的结构差异。

3. 奖励过优化：Gao Scaling Laws 与防御工具箱

3.1 定量规律：$d=\sqrt{\mathrm{KL}}$ 与两条函数形式

Gao, Schulman & Hilton（2210.10760）用合成实验定义了这个问题的标准研究框架：拿一个大「gold RM」当真值生成标签，训练小 proxy RM，对 proxy 优化并观察 gold 分数。定义 $d := \sqrt{D_{\mathrm{KL}}(\pi \,\|\, \pi_{\mathrm{init}})}$，gold reward 的经验形式为：

$$ R_{\mathrm{BoN}}(d) = d\,(\alpha_{\mathrm{bon}} - \beta_{\mathrm{bon}}\, d) \qquad\qquad R_{\mathrm{RL}}(d) = d\,(\alpha_{\mathrm{RL}} - \beta_{\mathrm{RL}} \log d) $$

解读：

  • 两条曲线都是先升后降——proxy 分数单调上升的同时，真实质量越过拐点开始下跌，这就是 Goodhart's Law 的定量形态。
  • BoN 与 RL 的退化形式不同：BoN 是二次型（衰减更急），RL 是 $d\log d$ 型（作者注明该形式在原点附近不成立，斜率发散）。同样的 proxy RM，用在 BoN 重排和用在 RL 优化上的「安全 KL 预算」不一样，监控阈值要分开定。顺带记住 BoN 的解析 KL：$\mathrm{KL}_{\mathrm{bon}} = \log n - \frac{n-1}{n}$（Stiennon et al. 附录 G.3，Gao et al. 沿用）——$n=64$ 也才约 3.2 nats，这就是为什么 BoN 是「温和优化器」、适合当过优化探针（§3.2）。
  • 系数随 RM 参数量平滑变化：更大的 RM 把拐点推得更远，但不消除拐点。论文还扫了 RM 数据量、policy 大小与 KL penalty 系数的影响。这是「RM scaling 有用但不解决问题」的出处。
  • 与 Anthropic HH-RLHF 的「reward 近似线性于 $\sqrt{\mathrm{KL}}$」经验关系同框看：训练早期 $d$ 小，$R(d)\approx \alpha d$，线性段正是健康区间；偏离线性即是预警信号。
KL 正则不是保险箱：Catastrophic Goodhart Kwa et al.（2407.14503）证明当 reward 误差是 heavy-tailed 时，KL 正则化无法阻止过优化——policy 可以在极小 KL 预算内找到误差长尾上的极端样本。reward 误差轻尾时 KL 才有兜底作用。结合 §8 的 token 空间攻击看：对抗样本恰恰制造 heavy tail，所以「有 KL 就安全」在对抗场景下是错觉。

3.2 检测：怎么知道自己正在过优化

  • gold/reference RM 重打分：标准检测法。用一个更强的、不参与训练的 RM 给 policy 输出重打分；训练早期 proxy 与 gold 同升，过优化时 proxy 继续涨而 gold 开始跌——回退 checkpoint 或提高 KL 系数。
  • BoN 监控：pitfalls §11 的做法——定期对固定 prompt 集做 Best-of-N（如 N=64）采样，观察 RM 选出的 top 样本人工质量是否还在涨。BoN 是比 RL 更便宜的「过优化探针」，因为它对同一个 RM 施加可控强度的优化压力。
  • real-eval 相关性：跟踪训练 reward 与真实评测（人工或独立 benchmark）的相关性，相关性破裂即拐点。

4. 偏好数据工程：标注、噪声、清洗与迭代式 RLHF

4.1 标注设计：InstructGPT 与 Llama 2 的两套范本

  • InstructGPT 式（排序制）：每个 prompt 采 $K=4$–$9$ 个回复让标注员全排序，拆成 $\binom{K}{2}$ 对。优点是单位标注时间产出的比较对多；代价是对内相关性高，必须同 batch 训练（§2.2）。
  • Llama 2 式（成对 + 强度分档）：二选一并标注偏好强度（四档，从 significantly better 到 negligibly better），强度档位喂给 margin loss。同时按目标拆分标注流（helpfulness / safety 分开收集）。
  • 共同纪律：写清标注规范文档、用 held-out 标注员测一致率、用一致率校准对 RM 准确率的预期（§2.3 铁律 2）。一致率没测出来之前，RM 准确率数字没有意义。

4.2 噪声与清洗：质量 > 数量，但规模在清洗后兑现

  • 公开偏好集噪声严重。Secrets of RLHF Part II（2401.06080）系统分析了偏好数据中的噪声标签并提出度量偏好强度、翻转/丢弃可疑标签的流程——拿公开偏好集直接训 RM 之前必须过一遍质量分析，这是它最大的实用价值。
  • Skywork-Reward v1（2410.18451）：精筛 80K 对——明确强调比现有数据集小得多——训出的 Gemma-27B RM 登顶当时的 RewardBench，证明「小而精」打得过「大而脏」。
  • Skywork-Reward-V2（2507.01352）：清洗后的规模才是规模。人机协同两阶段流水线（人类核验 + LLM 自动化 curation）构建 SynPref-40M（4000 万对），实际用其中精选的 2600 万对训练 0.6B–8B 共 8 个 BT 标量 RM，在七大 RM 基准全面 SOTA。消融确认：效果同时来自规模与质量，缺一不可。这是「标量 RM 的数据杠杆仍未耗尽」的最强证据。
  • 实践默认流程：AI 预筛（用现有 RM/judge 打分、过滤明显噪声与重复）+ 人工抽检（校准 AI 筛选器本身的偏置）。纯人标贵且慢，纯 AI 标会复制 judge 偏置（§8），混合是 2025 之后的事实标准。

4.3 迭代式 RLHF：对抗「静态 RM × 漂移 policy」

RM 是静态快照，policy 在 RL 中持续漂移——这对矛盾只有两类解法：限制漂移（KL，§3）或更新快照（迭代）。工业界的标准编排：

轮次 t:  用当前最优 policy π_t 对新 prompt 采样
      → 人工/AI 标注新偏好对（数据分布跟随 policy 分布）
      → 重训/增量训练 RM_t+1
      → 用 RM_t+1 做下一轮 RL / rejection sampling / DPO → π_t+1
  • Anthropic HH-RLHF：「online」迭代模式的最早系统记录——每周用最新 policy 重新收集偏好、重训 RM，整个流水线滚动前进。论文给出的动机正是 §2.4 的校准发现：PM 在高分区失准，必须用新数据让它「stay on-distribution」。
  • Llama 2：多轮迭代（论文中 RLHF-V1 到 V5），每轮混合 rejection sampling 与 PPO，RM 随轮次重训。
  • Llama 3：每轮用上一轮最优模型采样收集偏好数据，使训练数据贴近当前 policy 分布；整条管线为 rejection sampling + SFT + DPO（弃 PPO：大规模下 DPO 更省算力且在 IFEval 等指令跟随上更好）。
与 DPO 家族的接口 迭代式流水线里 RM 的三重身份：① rejection sampling 的重排器；② DPO 偏好对的标注器（off-policy 数据 → on-policy 化的关键，online vs offline 差距的理论见审计 G22）；③ PPO 的奖励源。同一个 RM 的质量问题会沿三条通道同时传播——这就是为什么 §9 的「下游 proxy 验证」必须在你实际使用 RM 的那条通道上做。

5. PRM vs ORM：过程奖励的承诺与现实

5.1 二分法的起源

Uesato et al.（2211.14275, DeepMind）在数学应用题上首次系统对比两种监督：outcome supervision（只看最终答案对错 → ORM）与 process supervision（逐步骤评判 → PRM），确立了这对术语。Lightman et al.（2305.20050, OpenAI）把结论推到最有影响力的形态：发布 80 万步级标注的 PRM800K，证明在 MATH 上 process supervision 显著优于 outcome supervision，并奠定「PRM + Best-of-N 搜索」的标准用法。

两种 PRM 用法的形式化：设解答由步骤 $s_1,\dots,s_T$ 组成，PRM 给出 $p_t = P(s_t \text{ 正确} \mid x, s_{1:t})$。
重排（BoN）：对每个候选解聚合步级分（常用 $\min_t p_t$ 或 $\prod_t p_t$），取最高者。
RL 密集奖励：把 $p_t$ 作为 step 级 reward 接进 PPO/GRPO——这正是后来被证明极易被 hack 的用法。

5.2 自动标注：Math-Shepherd 与它的噪声

人工步级标注（PRM800K 路线）贵到无法复制，Math-Shepherd（2312.08935）给出省钱主流：MC 自动标注——从步骤 $s_t$ 出发采样多条 completion，用「有多少条最终走到正确答案」估计该步的价值，无需人工。代价是噪声显著：一个错误步骤碰巧被后续修正、或正确步骤后续采样质量差，都会标错。ProcessBench（2412.06559）把「找出最早错误步骤（或确认全对）」做成基准后给出残酷对照：在竞赛/奥数级难题上，现有开源 PRM 既不如直接 prompt 通用 LLM 当 critic，也不如在 PRM800K 上朴素微调的 baseline——MC 自动标注的噪声在难题上被充分暴露。PRMBench（2501.03124）做了更细粒度的错误类型分析；2025-10 的 PRM 综述（2510.08049）进一步指出：现有 PRM 出了数学域（常识推理、开放任务）泛化很差，数据构建贵且标注噪声大。

5.3 实战定位：R1 的三连否定与 PRM 的真正用武之地

DeepSeek-R1 弃用 PRM 的三个理由（原文要点，必须完整记住） ① 细粒度 step 难以定义、其正确性难以评判；自动标注效果不佳，人工标注无法规模化。② 神经 RM（含 PRM）在大规模 RL 中必然遭遇 reward hacking。③ 重训 RM 需要额外训练资源，使整条流水线复杂化。——注意这是对「拿 PRM 当大规模 RL 奖励源」的否定，不是对 PRM 这个物种的否定。
用法判定理由
Best-of-N / 树搜索重排（test-time scaling）✅ 主战场优化压力一次性、可控；PRM 的步级信息在搜索剪枝中价值最大。
数据过滤 / 蒸馏数据挑选✅ 安全离线使用，无迭代优化压力。
直接当 RL 密集奖励❌ 高危R1 三连否定；policy 会迅速学会「看起来聪明但不推进解题」的步骤骗分。
数学域之外直接复用❌ 不成立PRM 综述（2510.08049）：跨域泛化差是当前 PRM 的结构性短板。

5.4 2026 的统一视角：RM 本来就是 value function

TCRM（2604.22981）把 ORM/PRM 之争翻了个底朝天：在标准 BT 损失上加两个正则项——分别对应 Monte Carlo 与 TD 两种价值学习目标——使 RM 在任意 token 处的输出都成为「给定前缀条件下最终奖励的条件期望」。也就是说，训练得当的 RM 形式上就是 value function。效果：中段 token 的判断准确率从 50%（随机）提到 88.9%；零步级标注拿到 ProcessBench 44.9 F1；在 PPO 中 reward model 与 critic 合一，省 27% 显存、快 19% 每步耗时。架构、数据、推理全不用改。这条线与 §6 的 GenRM 是互补的两种「让 RM 产出更多结构」的路径：TCRM 在 token 维度展开，GenRM 在推理链维度展开。

6. 生成式奖励模型（GenRM）：把奖励变成推理任务

6.1 为什么以及谱系：从「分类头」到「生成判断」

标量 RM 把判断压成一次前向；GenRM 的核心赌注是：评判本身是推理任务，应该花推理算力。三条 2024 年的萌芽路线奠定了形态：

  • Generative Verifiers（2408.15240, DeepMind）：把验证器从判别式分类头改写成 next-token prediction 目标，与解题生成联合训练。立刻解锁三件标量头做不到的事：CoT 式验证、majority voting 吃测试时算力、与指令微调无缝衔接。Best-of-N 下提升显著（GSM8K 73%→93.4%，MATH easy-to-hard 28%→44.6%），全面优于判别式 RM、DPO 式 verifier 与裸 LLM-as-judge。
  • CLoud（2408.11791）：Critique-out-Loud——先生成自然语言 critique，再以 critique 为条件输出标量分。「先解释后打分」的两段式成为后来 GenRM 的标准骨架。
  • Self-Taught Evaluators（2408.02666, Meta）：不用人类偏好标注，纯合成数据迭代自举 judge（生成对比回复对 + 推理链）。证明 judge 能力可以自我增强，是 §7.3 K2 self-critique 的学术前身。

加上更早的 LLM-as-a-judge（MT-Bench, 2306.05685）与 Self-Rewarding LM（2401.10020，模型自己当 judge 给自己造 DPO 偏好对），「用生成能力做评判」在 2024 年完成了概念验证；2025 年的两件工作把它推成主线：

6.2 RM-R1（2505.02387）：Reward Modeling as Reasoning

  • 核心机制 Chain-of-Rubrics（CoR）：judge 不直接打分，而是先按样本生成评判结构——对话类样本生成该样本专属的 rubric，数学/代码类样本先自己解一遍题——再据此逐项评判候选回复。rubric 是动态的、样本级的，不是全局写死的。
  • 两阶段训练：① 蒸馏高质量推理链（教 distilled 模型「怎么评」）；② 用可验证奖励做 RL（判断对错本身可验证——偏好标签就是 ground truth），强化评判能力。
  • 结果：在三个 RM 基准平均上超过更大的开源 RM（如 INF-ORM-Llama3.1-70B）与闭源 judge（GPT-4o），最高领先 4.9%。证明「小而会推理」可以打「大而直觉」。

6.3 DeepSeek-GRM / SPCT（2504.02495）：用推理时算力扩展 RM

  • 形式选择：pointwise GRM。逐个候选生成文本化评判与分数（而非两两比较），输入形式灵活（单个、成对、多个皆可），且天然支持「同一输入多次采样」。
  • SPCT（Self-Principled Critique Tuning）：在线 RL 训练 GRM，使其学会自适应地生成评判原则（principles），再据原则产出准确 critique——「先立法再执法」，与 RM-R1 的 CoR 异曲同工。
  • 推理时扩展：并行采样 $k$ 份「原则 + critique + 分数」，投票聚合；再训一个 meta RM 过滤低质量判断后投票。DeepSeek-GRM-27B（基于 Gemma-2-27B）从单样本 69.9 升到 32 样本 + meta RM 的 72.8（基准总分），超过 GPT-4o 与 Nemotron-4-340B-Reward 等大得多的对手——推理时扩展打赢了训练时扩展，这是该文最重要的论断。

6.4 Nemotron GenRM：大 GenRM 直接当 RL 奖励源的工业路线

NVIDIA 的 Qwen3-Nemotron-235B-A22B-GenRM（2025-12 发布，Apache 2.0）代表另一极：不追求小而精，而是用 235B（A22B MoE，基于 Qwen3-235B-A22B-Thinking-2507）的大 GenRM 直接做 RL 训练的奖励信号（官方模型卡明确：用于 NVIDIA-Nemotron-3-Nano-30B 的 RLHF）。数据为 HelpSteer3 + arena 人类偏好 + 安全偏好的混合（人类与合成标签混合）。成绩：RM-Bench overall 87.3（Chat 76.5 / Math 96.9），JudgeBench overall 87.4（Math 91.1 / Code 95.2 / Reasoning 95.9）。要点：训练侧用大 GenRM 的前提是推理基础设施扛得住——奖励计算从「一次 8B 前向」变成「一次 235B 带思考链的生成」，rollout 吞吐直接由 judge 推理决定（工程影响见 §10.4）。

6.5 Beyond Length Scaling（2603.01571）：GenRM 的算力该花在哪

2026 年 GenRM 研究的主线问题：critique 越长越好吗？该文的答案是否定的——一味拉长 CoT 收益递减，且两种推理在结构上不同：

  • B-CoT（广度）：多维原则覆盖——评判涉及哪些维度、有没有漏。对主观偏好任务更有效。
  • D-CoT（深度）：单点判断的论证严谨性——某一步到底对不对。对客观正确性任务更有效。
  • Mix-GRM：SFT 内化两种结构 + RLVR 优化；RLVR 表现为「切换放大器」，让模型自动按任务匹配推理风格。五个基准 SOTA，平均超开源 RM 8.2%。

把 §6.3 与 §6.5 连起来读：SPCT 解决「多采样怎么聚合」，Beyond Length Scaling 解决「单条推理怎么花」——GenRM 的算力分配已经形成「并行 × 广度 × 深度」三个正交维度，与 reasoning model 的 test-time scaling 研究完全同构。

7. Rubric Reward 工业化：从 RaR 到 Kimi K2 与 DeepSeek-V4

7.1 定位：可验证与不可验证之间的第三条路

RLVR 的边界是「答案能否机器判定」；通用偏好 RM 的弱点是「好」的定义不透明、易被 hack。rubric reward 的核心思想：把「好」显式分解成一组可逐条核验的标准（criteria），逐条打分再聚合。它本质上是把 reward 的「规格说明书」从 RM 的权重里搬出来、写成人能读的文本——可审计、可增删、可针对已知 hack 打补丁。代价是规格说明书要有人写、有人维护，写漏了的地方就是新的 hacking 面。

7.2 学术验证：RaR 与 Checklists

  • Rubrics as Rewards（2507.17746）：用结构化 rubric（checklist 风格）做 on-policy RL 的奖励，把 RLVR 扩展到不可验证域。在 HealthBench 上相对 LLM-as-judge 基线提升至多 31%、GPQA-Diamond 提升 7%；并且 rubric 让小 judge 模型也能给出可靠奖励、显著降低不同 judge 规模间的方差——rubric 是「降低对 judge 能力依赖」的杠杆，这一点对成本敏感的生产管线至关重要。
  • Checklists Are Better Than Reward Models（2507.18624）：同期独立证据：「Reinforcement Learning from Checklist Feedback」按指令生成 checklist、用其逐项评分做 RL，在指令跟随上优于通用 RM 奖励。

7.3 Kimi K2（2507.20534）：self-critique rubric reward 的生产级配方

K2 是「rubric reward 进入工业界」的标志。其 RL 同时使用两类奖励：可验证任务走 RLVR；不可验证任务走 self-critique rubric reward——模型自己（的另一份拷贝）当 critic，对候选回复做成对比较，依据一组 rubric 排序产生偏好信号。关键设计：

  • 两类 rubric 条目：core rubrics——assistant 应坚守的基本价值观；prescriptive rubrics——专门为消除 reward hacking 写的规定性条目（把已知 hack 模式直接写成扣分项）。这是「rubric 可打补丁」优势的实战形态。
  • critic 的冷启动与持续更新：critic 能力在 SFT 阶段就用开源 + 自建偏好数据混合引导启动；RL 过程中 critic 随 policy 一起演进，用 RLVR 侧的可验证信号持续校准 critic 的评判能力——用客观信号锚定主观评判，缓解 self-preference 漂移（§8.1）。

7.4 DeepSeek-V4（2026-04）：钟摆回摆的完成时

V4 技术报告（2026-04-24 随权重发布）对难验证任务的方案：不训独立的偏好标量 RM，而是策划 rubric-guided 的 RL 数据，让 actor 自己同时当 generator 与 judge——按结构化 rubric 给自己的输出打分（GRM 形态）。报告强调评判能力与生成能力联合优化：生成越强，评判越准，反之亦然；rubric 一旦写好，GRM 可以零边际成本评判成千上万条轨迹。配合其「领域专家 SFT→GRPO→On-Policy Distillation 合并」的整体编排，V4 完成了从 R1「弃用神经 RM」到「神经 RM 以 rubric-guided GRM 形态回归」的整个钟摆。从业者应当读出的结论：2025 年「DeepSeek 证明了不需要 RM」是误读，正确表述是「不需要拿偏好数据训的静态标量 RM 当大规模 RL 奖励源」。

7.5 前沿：rubric 的生成与执行也在被 RL 化 / 工程化

  • Rubric-ARM（2602.01511）：把 rubric 生成视为隐动作（latent action），用交替 RL 联合优化 rubric 生成器与 judge、以最终判断准确率为目标——解决「rubric 谁来写、写得好不好没有梯度」的问题。交替（而非同时）更新缓解非平稳性、降低梯度方差，有理论论证；下游 offline/online 对齐均显著改善。
  • RLR³（2605.30244）：把验证粒度从任务级降到条目级（criterion-level）：每条 rubric 条目按性质分流——可机检的走「LLM 抽取器 + 确定性 verifier」路径，不可机检的走 LLM-judge 路径。配合最小暴露原则（抽取器看不到 ground truth、judge 看不到图像）防止合谋作弊、层级聚合区分必要/补充条目、组内去饱和。Qwen3-VL-30B 上 15 个基准平均 +4.7 分，审计确认确定性验证 + 最小暴露显著减少可利用的假阳性。

7.6 rubric 设计 checklist（实操）

写 rubric 前先回答四个问题 ① 这个任务的「好」能否分解？分解不了（纯审美/纯偏好）→ 退回 GenRM/偏好 RM。② 每个条目能否独立核验？能机检的条目（数字、引用存在性、格式、长度）一律分流给确定性 verifier（RLR³ 原则），只把真正需要判断力的留给 judge。③ 条目是「描述好」还是「禁止坏」？两类都要有——core 条目定义质量，prescriptive 条目封堵已知 hack（K2 原则）；每次发现新 hack 模式，第一反应是加 prescriptive 条目。④ 聚合权重谁定？必要条目一票否决/高权重，补充条目低权重（层级聚合）；切忌等权平均——那等于邀请 policy 用五个容易条目换一个关键条目。
  • 粒度经验：条目应当「一句话可判定」，复合条目（"准确且全面"）必拆。
  • 防 judge 偷懒：逐条目独立询问 judge（而非一次给整张表），可显著减少 halo effect；这同时是 §8.2 偏置缓解协议的一部分。
  • 维护节奏：rubric 与 RM 一样会「过时」——policy 漂移会找到 rubric 没写到的缝隙。把 rubric 纳入与 §4.3 迭代式 RLHF 同节奏的版本化管理。

8. Judge 偏置与对抗鲁棒性：Master Key 与 token 空间攻击

judge/GenRM 既是评测工具又是奖励来源——它的每一个偏置都会被 RL 的优化压力翻译成 policy 的行为缺陷，它的每一个漏洞都是 hacking 面。本节先讲「统计性偏置」（无恶意也会偏），再讲「对抗性攻击」（有优化压力时必然被找到）。

8.1 三大偏置：命名、量化、来源

偏置表现关键事实
Position bias 成对比较时偏向某个位置（通常是先出现的回复） MT-Bench（2306.05685）首次系统刻画；与内容无关、纯由呈现顺序导致，因此可以用机械协议完全对冲（见 8.2）。
Verbosity bias 偏好更长的回复，无论信息量 judge 与 RM 共享此病：Singhal et al.（2310.03716）发现多数 RLHF「增益」与变长高度相关。评测侧的标准修正是 AlpacaEval 2.0 LC（2404.04475）——用回归把长度因素从胜率中剥离；奖励侧见 §3.3 的 ODIN/数据配平。
Self-preference / self-enhancement bias 偏好自己（或同族模型）生成的回复 MT-Bench 即已指出；2026 年的系统量化（2604.22891）用「质量差可忽略的等价回复对」在 20 个主流 LLM 上分离了判别力与偏置，两个反直觉结论：模型能力与低自偏好无关甚至负相关（更强 ≠ 更公正）；结构化多维评判协议平均降低自偏好 31.5%。

基线锚点：MT-Bench 报告 GPT-4 judge 与人类偏好一致率超过 80%，与人类相互一致率同水平——这既是「LLM-as-judge 可用」的依据，也是它的天花板提示：judge 的不可约误差与人类分歧同量级，把 judge 分数当 ground truth 精读小数点是自欺。

8.2 机械式缓解协议（无需训练，立刻可用）

Judge 偏置六条标准协议 ① swap-and-average：成对比较必须双向各评一次（A-B 与 B-A），取一致结果；不一致记平局或重采样——position bias 的标准解法，没有理由不做。② 不用同族模型自评：判 Qwen 系 policy 不用 Qwen 系 judge；无法避开时用多 judge 混合。③ rubric 分解：逐条目独立询问，减少 halo effect 与自偏好（2604.22891 的 31.5% 改善正来自结构化分解）。④ 控长：评测报告用 LC 胜率；奖励侧做长度配平或解耦。⑤ 温度与采样固定：judge 推理参数纳入版本管理，否则评测不可复现。⑥ 上线前人工 meta-eval：抽样几百条 judge 判断做人工复核，测出该场景下 judge 与人的一致率，作为后续一切自动评测的置信度上界。

8.3 Master Key 攻击（2507.08794）：一个标点骗过所有 judge

2025-07 的引爆点。发现：对「判断这道题答案是否正确」类的 judge/GenRM，完全不含实质内容的「master key」即可骗出假阳性——非词符号（":"、"."）或通用推理开场白（"Thought process:"、"Let's solve this problem step by step"）。量化结论：

  • master key 攻击的假阳性率最高可达 80%（依基准与模型而异）；仅一个标点即可让 GPT-4o 产生最高 35% 假阳性；"Thought process:" 在 LLaMA3-70B-Instruct、Qwen2.5-72B-Instruct 等强开源模型上假阳性率高达 60–90%。
  • GPT-o1、Claude-4 级别的前沿模型同样中招——「用更强的 judge」不是防御。
  • 机理提示：judge 把「推理的形态特征」当成了「推理正确」的证据——这恰是 GenRM 把判断变成生成任务后引入的新攻击通道，标量 RM 反而没有这条语义通道。

防御（Master-RM）：构造对抗负例——把模型输出截断到只剩开场白、标为错误——加进 judge 训练数据。如此训练的 Master-RM 对全部 master key 攻击假阳性率接近零，且标准评测性能不掉。成本极低，任何自训 judge/GenRM 的管线都应默认加入截断负例增强。

8.4 Token 空间攻击（2604.02686）：标量 RM 也不安全

TOMPA（Token Mapping Perturbation Attack）绕过「先解码成文本再重新分词」的标准接口，直接在 token 空间做黑盒优化（只用标量反馈）。结果：对 Skywork-Reward-V2-Llama-3.1-8B，把 GPT-5 参考答案的 reward 几乎翻倍、在 98.0% 的 prompt 上压过参考答案——而对应文本是乱码。含义有三：① RM 鲁棒性问题不止于语义层的 master key；② RL 训练中 policy 的探索恰恰发生在 token 空间，理论上能自行撞到这类模式（这是 §3.1 heavy-tailed reward 误差的具体来源之一）；③ 对抗训练（在 RM 训练中混入 token 级扰动负例）正在进入 RM 训练的标准流程。

8.5 防御共识三条（2025–2026 收束）

  1. 对抗负例训练：截断输出、master key、token 扰动样本进 judge/RM 训练集（Master-RM 路线）。
  2. rubric 写入 prescriptive 防 hack 条目：已知 hack 模式显式扣分（Kimi K2 路线，§7.3）。
  3. 可验证部分分流给确定性 verifier：能机检的绝不过 judge，缩小语义攻击面（RLR³ 路线，§7.5）。

交叉引用：reward hacking 的 policy 侧形态与缓解见 pitfalls §1；本节是它的「判官侧」补全。

9. RM 评测基准：RewardBench 系与「榜单 ≠ 下游」

9.1 RewardBench v1（2403.13787）：定义子方向，然后被刷爆

第一个专门的 RM 评测基准：prompt–chosen–rejected 三元组上测 RM 能否给 chosen 更高分，覆盖 Chat、Chat-Hard、Safety、Reasoning 四个子集（Chat-Hard 用「细微但可验证」的差异构造，如 bug、事实错误）。它的历史作用与历史教训同样大：发布后迅速成为 RM 训练的优化目标——社区按榜单挑数据、调配方（Skywork v1 的 80K 数据即以登顶 RewardBench 为信号筛选），两年内头部分数饱和，基准从「度量」退化为「目标」，Goodhart 定律在评测层重演了一遍 §3 在训练层的故事。

9.2 RewardBench 2（2506.01937）：换上全新人类 prompt，平均掉约 20 分

  • 核心改动：不再复用现有评测集的 prompt，改用全新采集的人类 prompt，并升级为更难的 best-of-N 式准确率评测，覆盖指令跟随、推理、安全等域。
  • 核心数字：现有模型在 v2 上比 v1 平均低约 20 分——v1 头部成绩中相当一部分是对基准分布的过拟合，而非 RM 能力。
  • 论文同时给出正面结论：v2 分数与下游表现（BoN 推理时扩展、PPO 式 RLHF）高度相关——前提是按论文规范使用。也就是说基准本身可以做对，前提是 prompt 不泄漏、评测协议贴近真实用法。

9.3 PPE（2410.14872）：用真实 RLHF 终点校准评测起点

「How to Evaluate Reward Models for RLHF」回答的问题是元问题：RM 基准上的哪个指标真的预测下游？方法：在大规模众包人类偏好平台上跑端到端 RLHF 实验拿到 gold 结果，再回归「12 个指标 × 12 个域」中哪些与之相关。这项工作（ICLR 2025）确立了「榜单分数与 PPO/BoN 下游效果相关性有限、且不同指标差异很大」的实证基础，是 2024–2026「榜单 ≠ 下游」争议的定量出处。

从业共识（必须内化） 选 RM 不能只看榜单。正确流程：用你自己的 prompt 分布与你实际使用 RM 的方式（BoN？DPO 标注？PPO 奖励？）做小规模下游 proxy 实验——BoN 重排 + 人工抽检，或小规模 RL + gold RM 重打分——用 proxy 结果选型。榜单的正确用法是粗筛候选与排除明显短板（比如 RM-Bench 风格鲁棒性接近随机的型号直接出局）。

10. 把 RM 接进 RL 训练循环：工程实操

10.1 reward 的数值处理

  • 先归一化：BT 训出的分数只有相对意义（§2.1），不同 prompt 不可比。标准做法：用经验均值/方差做 whitening（HF《The N Implementation Details of RLHF with PPO》复现细节：reward 白化可不减均值，只除以标准差；advantage 白化按 minibatch 做）。
  • 再有界化：无界 reward 是 KL 爆炸与 hacking 的放大器。PAR（2502.18770）的两条设计原则——reward 应有界、形状应「初期快速增长后期渐进收敛」——sigmoid(中心化分数) 是当前最省事的实现，其 AlpacaEval 2.0 胜率比其他 shaping 方法高至少 5 个百分点且训两个 epoch 不 hack。
  • 奖励施加位置：标准实现是「整句 reward 加在 EOS token 上 + 逐 token KL penalty」。RM 分数只在序列末端取（最后非 pad token，§2.2），KL 项每个 token 都有。想要真正的 dense reward，2026 年的新选项是 TCRM（§5.4）而不是手工 shaping。

10.2 PPO 侧的 RM 相关细节

  • critic 用 RM 权重初始化是经典默认（两者输出语义相近：序列价值估计）；TCRM 进一步允许 reward/critic 合一（−27% 显存）。
  • KL 系数是第一超参。β 无普适最优值——取决于 RM 质量、期望行为改变幅度、回复长度分布。社区经验法则（注意：这是实践启发式，非论文定理）：整条回复累计 KL 保持在 ~15–20 nats 内；逐 token KL 持续越过 ~0.1 且不回头时，把 KL 系数上调约 50%。自适应 KL controller（按目标 KL 区间调 β）是 OpenAI 摘要工作以来的标准件。
  • 每条 rollout 的 RM 调用就是吞吐瓶颈的一部分：标量 8B RM 与 235B GenRM（§6.4）之间差着 1–2 个数量级的奖励计算成本；GenRM 做奖励源时，judge 的推理服务要和 policy rollout 同级别规划容量，并考虑用批量化 pointwise 评分（SPCT 形式）而非两两比较（调用数 $O(N)$ vs $O(N^2)$）。
  • KL 估计器选择（k1/k2/k3）与「估计 ≠ 优化」的细节见 pitfalls §2 与审计 G2，不在此重复。

11.3 开放问题（2026-06 视角）

  • GenRM 的对抗鲁棒性没有理论保障：Master-RM 式数据增强是补丁不是证明；token 空间与语义空间的攻击此消彼长，对抗训练进入 RM 标准流程后的均衡点未知。
  • 评判-生成联合优化的稳定性：V4/K2 把 judge 和 policy 绑在同一模型/同一 loop 里，self-preference 漂移的长期动力学（judge 跟着 policy 一起偏）缺少公开研究；Rubric-ARM 的交替优化是一个起点。
  • RM 评测的 Goodhart 循环：RewardBench 2 修复了 v1 的泄漏，但只要榜单影响选型，v2 也会走向饱和——「评测必须跟着使用方式走」（PPE 方法论）能否产品化为持续滚动的私有 proxy 评测，是工程问题更是生态问题。
  • 个性化与多元偏好：BT 假设单一全局偏好序，而真实用户偏好异质；稀疏 MoE RM（2606.04284）的可解释专家是一条路，但「给谁对齐」的规格问题超出技术本身。
  • 价值函数统一（TCRM）与 GenRM 路线会合流吗：一边把标量 RM 变成逐 token 价值函数，一边把 RM 变成推理者；两者分别压缩了「时间维」与「推理维」的信息瓶颈，统一框架尚缺。
