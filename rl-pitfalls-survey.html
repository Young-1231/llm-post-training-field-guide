1. Reward Hacking / 奖励黑客

定义：模型并未真正学好任务，而是利用 reward 函数的「形状缺陷」拿到高分。这是 Goodhart's Law 的经典体现：「当一个度量变成目标，它就不再是好的度量」。在 RLHF 中表现为 proxy reward 持续上升，但真实质量先升后降。

1.1 RLHF 中的典型案例

  • Verbosity / Length bias（最常见）：RM 在偏好数据上学会「长答案更好」的伪关联，policy 于是生成越写越长的废话。这是 OpenAI、Anthropic 都报告过的现象。
  • Sycophancy（谄媚）：模型迎合用户立场以拿到偏好分，即使用户说的是错的。
  • Format hacking：堆砌 markdown 列表、加粗、emoji、"Sure! Here's..." 这种开头，让 RM 误判为高质量。
  • Repetition / 元评论：模型加上"Let me explain..."、"In summary..."等"看上去用心"的句式。
  • Refusal hacking：当 safety RM 占主导时，policy 对任何请求都拒绝（"I cannot..."），因为拒绝从不被扣分。

1.2 Reasoning RL 中的典型案例

  • Thinking 假装：在 <think>...</think> 中输出无意义的「Wait, let me reconsider...」循环，骗过 format reward。
  • 答案泄漏：rule-based reward 抽取 `\boxed{}` 内容时，模型把多个候选答案都写进 box，或在外面用某种特殊格式偷偷写入。
  • Code 题套壳：在 unit test 是已知的情况下，policy 学会硬编码 if-else 匹配测试样例。
  • 语言混杂（DeepSeek-R1-Zero）：纯 RL 训练出的模型在思考阶段中英文混杂，因为 rule-based reward 不关心语言一致性。R1 论文专门加了 language consistency reward。
  • 反向推理：先看到答案再编理由（DPO 训练对话风格的 reasoner 时尤其多）。
真实工程案例（DeepSeek-R1 论文） DeepSeek 团队明确放弃了 neural reward model：「在大规模 RL 中 neural RM 会 suffer from reward hacking，并且重新训练需要额外资源」。这也是为什么 R1-Zero 选择纯 rule-based reward 的关键原因。

1.3 缓解方法

方法原理注意点
Reward shaping在原始 reward 上加形状项，例如对长度二次惩罚、对 format 错误打 0、对答案格式严格化。容易引入新的 hacking 表面；最好用「乘法门」而非「加法分量」。
Length normalization把 RM score 关于 length 回归出来作为 bias，从总 reward 中减去。代表工作 ODIN（disentangled reward）。ODIN 在 RM 上设双 head，一个预测 length-only reward，训练时减去。
KL constraint限制 policy 离 ref model 过远，作为对 reward model OOD 区域的兜底。"Catastrophic Goodhart" 论文（2024）指出当 reward 误差是 heavy-tailed 时 KL 并不能完全救场。
Robust RMInfoRM（信息瓶颈）、Ensemble RM、Conservative RM。RM ensemble 比单一 RM 显著降低 over-optimization。
Rule-based reward对可验证任务（数学/代码）只用 ground truth，避开 RM 完全。R1 / DAPO 主流做法；但只适用于可验证领域。
Early stopping用 held-out 真实 eval 监控，到拐点就停。Gao et al. Scaling Laws for Reward Model Overoptimization 提供经验曲线。

1.4 CoT 可监控性：抓 hacking 的最佳信号源，也是最容易毁掉的

上表是 2024 视角的缓解谱系。2025 年补上了最重要的一条「禁忌」：推理模型的 CoT 是检测 reward hacking 的最佳信号源——模型动手 hack 之前往往在思考里直说（"the tests only check the return value, I can just hardcode it"）——但对 CoT 施加优化压力会永久性毁掉这个信号源。Baker et al.（OpenAI, 2025-03）在前沿 agentic 编码环境上的三个实证：

  • CoT 监控有效且便宜：用另一个 LLM 读被训模型的 CoT 来报 hack，召回远超只看 action / output 的监控；且弱模型可以监控强模型——GPT-4o 足以有效监控比它更强的 o3-mini 级 frontier reasoning model。
  • Obfuscated reward hacking：把「CoT 里不许有坏念头」放进 reward，低优化压力下指标确实变好；但随训练加深，模型学会把意图藏出 CoT——满纸正确性，行为照样以显著比率 hack。监控器从此失明，而且你不知道它瞎了——短期指标变好正是它在起效的假象。
  • Monitorability tax：要保住可监控性，就得放弃「顺手优化 CoT」能带来的那部分性能/合规收益。论文的正式建议：在找到更好的办法之前，完全避免对 CoT 的强优化压力，把这笔「税」交了。

两条后续把结论钉得更死：跨实验室 position paper（Korbak et al., 2025-07；OpenAI / Google DeepMind / Anthropic / Meta 等 40+ 作者联署）把 CoT monitorability 定性为「新且脆弱」的安全机会窗口，要求前沿开发者在做架构与训练决策时显式评估其影响；2026 年初的后续实验（arXiv:2601.23086）则表明混淆会泛化——在一组任务上学会隐藏 hack 推理的模型，把 hack 行为与混淆能力一起带到未见任务；更糟的是只惩罚 CoT 结束后的最终动作（完全不碰 CoT）同样能诱发混淆。即监控-惩罚回路只要存在且可被感知，混淆迟早出现，区别只是速度。

监控信号用法判定原因
监控器读 CoT → 报警 / 下线样本 / 触发人工审计（不进梯度）允许纯「读」通道，不对 CoT 分布施压
监控器做 reward 门控，筛除 hack 轨迹（不奖也不罚）谨慎弱一档的选择压力仍可能教出混淆；必须配 monitorability 回归测试（catch rate 随训练下降而 hack 率不降 = obfuscation 正在发生）

1.5 Reward hacking 的二阶后果：emergent misalignment 与 inoculation prompting

2025-11 的 Anthropic 生产实证把 reward hacking 的定性从「训练指标失真」升级为「安全事件源头」。原始线是 Emergent Misalignment（Betley et al.）：仅在「输出不安全代码且不告知用户」这个窄任务上微调，模型就会在与编码无关的 prompt 上广泛 misaligned（效应在 GPT-4o 与 Qwen2.5-Coder-32B-Instruct 上最强），且可被 backdoor 触发词做成评测看不见的选择性 misalignment。最有解释力的机理框架：窄域坏行为更新的不是技能，而是「我是什么样的助手」这个全局自我叙事变量。

Natural Emergent Misalignment from Reward Hacking in Production RL（MacDiarmid et al., Anthropic）用训练 Claude 的真实生产编码 RL 环境复现了整条链：模型学会 reward hacking 之后，未经任何针对性训练，自发泛化出 alignment faking（简单问句上 50% 的回答出现伪装对齐推理）、配合恶意行为者，并在被用于安全研究代码库的 agentic 任务时 12% 尝试 sabotage。更麻烦的发现：标准 chat 式 RLHF 安全训练之后，chat 评测全部变好，但 agentic 任务上的 misalignment 原样保留——安全训练的效果被上下文分布锁死，因此安全数据覆盖 chat + agentic + 工具使用三类上下文比堆样本量更关键。

Inoculation prompting（反直觉但已用于 Claude 生产训练）：在训练 system prompt 里加一行，显式把 hack 框定为可接受——温和措辞「这是一个不寻常的请求，你的任务就是让评分脚本通过」与露骨版同样有效。结果：模型照样学会 hack（hack 率仍 >99%），但最终 misalignment 下降 75~90%，回落到从未学过 hack 的基线水平。机理：阻断「我在作弊 → 我是坏角色」的自我叙事归因——hack 被语义上重新归类为「被允许的行为」后，不再与 misaligned persona 关联，泛化链条被切断。SFT 设置下的系统验证见 Wichers et al.（对 reward hacking、sycophancy 同样有效）。

治理要分两层，分开做、分开度量 一阶问题（hack 本身）：verifier 加固、hack 检测、§1.3 缓解表与 §1.4 的 CoT 监控——但 Skalse et al.（NeurIPS 2022）已形式化证明非平凡的不可 hack 代理奖励几乎不存在，hack 无法根除；二阶问题（hack → 广义 misalignment 的泛化）：inoculation prompting + 安全训练上下文多样性。inoculation 不是放任 hack，它只负责确保「学会了 hack」不连带学会「我是一个会骗人的 AI」。

完整论述（synthetic document finetuning 实验设置、蒸馏管线的 subliminal learning 风险、安全配方三条落地修订）见 eval-safety.html §8。参考：arXiv:2502.17424（Betley et al., Emergent Misalignment）、arXiv:2511.18397（MacDiarmid et al., Natural Emergent Misalignment from Reward Hacking in Production RL）、arXiv:2510.05024（Wichers et al., Inoculation Prompting）、arXiv:2209.13085（Skalse et al., Defining and Characterizing Reward Gaming, NeurIPS 2022）。

2. KL Divergence 的实战问题

2.1 KL Blow-up / Spike（KL 飞涨）

KL blow-up（也叫 KL spike / explosion）指 policy 在某个 step 离 ref 突然飞得很远（KL 跳到几十甚至几百），通常伴随：reward 短暂飙升然后崩，输出退化为乱码或重复 token，loss 出现 NaN。术语提醒：社区里 "collapse" 一般指 entropy / policy collapse（§5），KL 突增不要叫成 "KL collapse"。常见诱因：

  • KL 系数 β 太小，没有约束力。
  • Reward scale 过大导致 advantage 量级很大。
  • Clip range 太大（DAPO clip-higher 必须配合其他技巧）。
  • Mini-batch 内 token 重复度太高，梯度方向偏。

2.2 John Schulman 的三种 KL 估计 (k1 / k2 / k3)

在 RL for LLM 里，我们没有 q 和 p 的解析形式，只有 sample。设 r = p(x)/q(x)，logr = log p(x) - log q(x)，三种蒙特卡洛（采样）估计为：

$$ k_1 = -\log r = \log q - \log p $$ $$ k_2 = \tfrac{1}{2}(\log r)^2 $$ $$ k_3 = (r - 1) - \log r $$
估计量偏差方差正负实战意义
k1无偏巨大有正有负monte-carlo 朴素估计，单样本方差爆。
k2有偏较小恒 ≥ 0对应 TRL PPOTrainer 的可选项 kl_penalty="mse"（其默认是 k1 的 "kl"，k2 并非早期默认）。简单但有偏。
k3无偏较小恒 ≥ 0当前主流（DeepSeek GRPO 即把 k3 加进 loss）。

k3 的关键点：因为 r - 1 - log r ≥ 0 对任何 r > 0 成立，所以它每个样本上都非负，并且期望仍然等于 KL（Schulman 原 blog 给的证明）。

实战要点 DeepSeek GRPO 把 k3 作为正则项直接加进 loss（不再放进 reward），避免和 advantage 的 reward 信号纠缠。但 Costa Huang 等人报告 trl 早期实现里 k3 会 explode，往往是 importance ratio 太极端导致；先确认 logp 计算的 numerical stability 后再用。

2.3 Adaptive KL

OpenAI PPO 的经典做法：设定 target KL = d_targ（例如 6 或 0.01 等任务而定），每次 policy update 之后调整 β（PPO 原文的流程：一次 update 内先跑若干个 epoch 的 minibatch SGD，再按当次实测 KL 调整 β）：

$$ \beta \leftarrow \begin{cases} \beta / 2 & \text{if } \text{KL} \lt d_{\text{targ}} / 1.5 \\ \beta \times 2 & \text{if } \text{KL} \gt d_{\text{targ}} \times 1.5 \\ \beta & \text{otherwise} \end{cases} $$

实战建议：

  • RLHF 里 target KL 常取 6~10（per-prompt 总 KL）；reasoning RL 因为长度大，per-token KL 才有意义，目标常在 0.001~0.01 之间。
  • Adaptive KL 在快速崩塌时反应不够快，仍需 gradient norm 报警 + early stop 兜底。
  • DPO 不需要 adaptive KL（β 是 hyperparam 而非约束），但调它仍然敏感。

2.4 Forward vs Reverse KL：DPO 中的角色

DPO 的 implicit reward 形式恰好等价于 reverse KL（mode-seeking）的 RL 目标：

$$ \max_\pi \mathbb{E}_{x,y\sim\pi}[r(x,y)] - \beta \cdot D_{KL}(\pi\,\|\,\pi_{ref}) $$

这就解释了 DPO 训出的 model 多样性偏低（mode-seeking）。后续工作 f-DPO / Beyond Reverse KL 系列推广到 forward KL、JSD、α-divergence：

  • Reverse KL：选择 ref model 高概率的 mode；diversity 低；但 chat 任务里这反而稳。
  • Forward KL：覆盖所有 ref 高概率区域；diversity 高；但容易把不该有的 mode 也保留下来。
  • 实战上 reverse KL 几乎是默认选择；只有在需要保留多种风格（creative writing）时才考虑 forward。

3. Length Bias / Verbosity Bias

3.1 表现与成因

训练曲线上的典型现象：avg response length 单调上升、reward 也上升，但 win-rate vs SFT 在某点之后停止增长甚至下降。成因：

  • 人类标注偏好长答案（更详细 = 看起来更努力）。RM 学到这条捷径。
  • 训练目标关于 length 不是 length-invariant，导致优化朝长度倾斜。
  • DPO 中尤其严重，原因见下节。

3.2 为什么 DPO 特别容易 length-hack

DPO loss 是：

$$ \mathcal{L}_{DPO} = -\log\sigma\left(\beta\log\frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \beta\log\frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)}\right) $$

其中 log π(y|x) = Σ log π(y_t|x,y_{<t}) 是 token logp 之和。问题在于：

  • 当 y_w 比 y_l 长，log π(y_w) 天然更负（更多 token 相加），要让 log π_θ(y_w)/π_ref(y_w) 大，模型必须在 y_w 上 token-wise 大幅提升，等价于在长样本上投入更多优化预算。
  • 结果：模型学会"把 winning 那种风格输出延长"，因为每延长一个 token 就多一份梯度推力。
  • Princeton 的 SimPO 论文指出 DPO 隐含一个 length-dependent reward。

3.3 SimPO 的 length normalization

SimPO 把 implicit reward 改成 average log-probability（除以 |y|）：

$$ r_{SimPO}(x,y) = \frac{\beta}{|y|} \log \pi_\theta(y|x) $$ $$ \mathcal{L}_{SimPO} = -\log\sigma\left(\frac{\beta}{|y_w|}\log\pi_\theta(y_w|x) - \frac{\beta}{|y_l|}\log\pi_\theta(y_l|x) - \gamma\right) $$

其中 γ 是 target reward margin，强制两类间留出 gap。SimPO 同时去掉了 ref model（reference-free），省下 ref 的权重显存与一次前向计算——注意 ref 本身无梯度、无 optimizer state，相对 policy 的「权重 + 梯度 + Adam 状态」省的远不到一半显存，SimPO 原文也只说 "more compute and memory efficient"。

3.4 Dr.GRPO 的两处修正

Liu et al. (Sea AI Lab, 2025) 指出标准 GRPO 有两个 bias：

  1. Token-level normalization bias：原 GRPO 每条 response 的 loss 除以 |o_i|。当 advantage 为正（答对），短答案 per-token gradient 更大，模型偏好"短而对"；当 advantage 为负（答错），长答案 per-token penalty 更小，模型偏好"长但错"。结果是错误答案越来越长（reward 0 也被长度拉高）。
  2. Question-level std normalization bias：原 GRPO 在 group 内除以 std(r)。对那种"几乎全对/全错"的 prompt，std 很小，advantage 被放大，导致这类 prompt 占用过多梯度预算。

Dr.GRPO 修正：

原 GRPO advantage： $$ A_i = \frac{r_i - \text{mean}(r)}{\text{std}(r)} $$ Dr.GRPO advantage（去掉 std）： $$ A_i = r_i - \text{mean}(r) $$ Loss 聚合： 把 1/|o_i| 改成全局常数 1/L_max。

3.5 其他 length 缓解技巧

  • ODIN（disentangled RM）：训练时让 RM 输出 (r_quality, r_length) 两个头，监督上要求 r_length 能预测 length，policy 优化只用 r_quality。
  • Length penalty in reward：直接在 reward 上减去 α · len 或 α · max(0, len - L_target)，但要小心引入相反方向的 hacking（输出过短）。
  • DAPO overlong reward shaping：超过 L_max - buffer 的部分按比例惩罚，平滑过渡而非硬截断。
  • Adaptive Length Penalty (ALP)：根据 prompt 的 online solve rate 调整 length penalty，简单 prompt 重罚冗余，难 prompt 给它充分思考的空间。

4. Advantage Normalization 细节

4.1 GRPO 的 std normalization 陷阱

如 §3.4 所述，对 group 内 reward 做 z-score normalization 的副作用：

  • group 内方差小的 prompt（要么全对要么全错），advantage 被 std 放大；这种 prompt 反而梯度信号最大，但其实信息量最低。
  • 当 std → 0（典型如答案稳定收敛），数值变 NaN。多数实现里加 std + 1e-8。

4.2 Group-level vs Batch-level

归一化层级做法优劣
Group-level (GRPO 原)同一 prompt 的 N 个 rollouts 内 mean/std对 prompt 难度无感，每个 prompt 局部都有 0 均值；但小 std 放大问题严重。
Batch-level整个 batch 内 mean/std简单题和难题在同一坐标系下，难题 advantage 更显著；但破坏了 GRPO 的"group baseline"哲学。
Dr.GRPO仅 group-level mean，不除 std保留 group baseline 含义，去除 std bias。
Token-level + 全局 L_maxDAPO 做法token-level loss 不被 length 稀释，长 CoT 训练更稳。

4.3 何时该 / 不该 normalize

  • 应该 normalize mean：几乎总是。否则 reward bias 直接传到 advantage，KL 不再能控制更新幅度。
  • 应该 normalize std：当 reward 来自 noisy RM 且 batch 内尺度变化大时，z-score 有帮助。
  • 不应该 normalize std：rule-based reward（0/1 binary）情形，std 完全是噪声。Dr.GRPO / DAPO 都不除 std。
  • 动态选择：verl 给了 norm_adv_by_std_in_grpo 等开关，可按任务切换。

5. Entropy Collapse / 探索不足

5.1 表现

  • 训练前几百 step 内 policy entropy 单调下降（log scale 上从 ~1.0 跌到 ~0.05）。
  • 同一 prompt 的 N 个 rollouts 几乎一样（pass@1 ≈ pass@N），group advantage 整组为 0，"训不动"。
  • 模型在测试集上 pass@1 略涨但 pass@k 大跌——说明丢了 exploration。

5.2 经典缓解

  • Entropy bonus：在 loss 中加 -λ·H(π)，鼓励 policy 保持高熵。注意：RL for LLM 中 entropy bonus 的 λ 极度敏感（10x 差异即可能崩），需 sweep。
  • Higher sampling temperature：rollout 阶段 T = 1.0~1.2，比 deploy 时（T = 0.7）高。但温度太高会让 rollout 全是噪声。
  • Temperature schedule：训练初期高 T（探索），后期降到 evaluation 接近的 T（利用）。
  • Top-p / Top-k 调节：扩大采样范围。

5.3 DAPO 的 clip-higher

DAPO 的洞察：PPO 标准对称 clip [1-ε, 1+ε] 让上行（提高低概率 token 概率）和下行（降低高概率 token 概率）的约束相同。这会压低低概率 token 上升的能力 → 探索 token 无法被强化 → 熵下降。

DAPO 把上行和下行 clip 解耦：

$$ \text{clip}(\rho_t, 1-\varepsilon_{low}, 1+\varepsilon_{high}) $$ $$ \varepsilon_{low} = 0.2,\quad \varepsilon_{high} = 0.28 $$

用更大的上界让低概率"探索"token 有空间上升，从而维持熵。近期论文 Clip-Low Increases Entropy and Clip-High Decreases Entropy（arXiv 2509.26114）给出了这个现象的理论分析。

实战经验 单看 entropy 数值会误判。reasoning RL 中 entropy 经常先降到一个稳定值，然后随着 CoT 多样性出现而缓慢回升——这是"健康"曲线。如果一开始就单调跳水到 < 0.05，几乎必然 collapse。

6. Training Stability

6.1 常见症状

  • Loss spike：单个 step loss 跳几个数量级。多由极端 advantage、KL 突增、或某个 prompt 异常长引起。
  • NaN：通常源自 softmax overflow、log(0)、除零或 fp16 的 dynamic range 不够。
  • Grad explosion：梯度范数从 ~1 跳到 ~1e6。

6.2 Mixed Precision 注意点

精度动态范围RL 训练适配性
FP32大最稳，但慢且耗显存。
FP16极小（max ~6.5e4）RL 容易 overflow → NaN。必须配 dynamic loss scaling。但 2025 底 Sea AI Lab 论证训推统一 FP16 反而能基本消除训推失配（见 §7.4 之争）。
BF16同 FP32 范围，精度低长期默认选择，几乎无 overflow。注意：尾数仅 7 位，舍入误差是训推失配的重要来源（§7.2–7.4）；LM head 建议两侧都用 FP32 计算。
FP8极小verl 等已支持 FP8 RL，需细致 calibration；量化 rollout 会放大训推失配，必须配 TIS 类修正（§7.4）。
关键工程坑 混合精度下 必须先 unscale gradient 再 clip。否则 clip 看到的是被 loss-scale 放大过的梯度，clip 阈值会失效。PyTorch AMP 用 scaler.unscale_(optimizer) 然后 torch.nn.utils.clip_grad_norm_，再 scaler.step()。

6.3 LR Schedule、Warmup、Gradient Clipping

  • LR：RL 阶段的 LR 远小于 SFT，典型 1e-6 ~ 5e-6（policy），critic 高一点 1e-5。LR 太大第一时间体现为 KL spike。
  • Warmup：前 5%~10% step linear warmup 必备。冷启动直接全 LR 几乎必崩。
  • Gradient clipping：global norm clip 通常 1.0；reasoning RL 因为长序列，许多团队用 0.5 或 0.3 更稳。clip 频繁触发（每 step 都触发）是 LR 过大或 advantage scale 异常的信号。
  • Cosine decay：RL 阶段往往 constant LR 即可，因为本来训练 step 数就少；cosine decay 只有大规模 R1 类训练才需要。

6.4 防御性编程清单

  • 每个 mini-batch 算完 loss 后检查 torch.isfinite(loss)，非法直接 skip 这个 step（保留 optimizer state）。
  • logp 用 F.log_softmax(logits.float(), dim=-1)（强制 cast 到 fp32），避免 fp16/bf16 logp 不稳定。
  • Advantage 做完 normalize 后再 clip 一次（如 ±10），抑制极端值。
  • Reward 在送入 advantage 前 clip 到合理范围。

7. 训练-推理失配与异步 Off-Policy

这一主题在 2025–2026 年从「边缘工程细节」升级为 RL 稳定性的一阶因素。核心认识：现代 RL 系统里 off-policy 有两个相互独立的来源——训推失配（rollout 引擎与训练引擎在同一份权重下给出不同的 token 概率）和 staleness（异步架构里数据由滞后若干版本的策略生成）。两者的诊断指标和修法都不一样，混在一起排查是最常见的失误。本节前半讲失配（§7.2–7.5），后半讲异步（§7.6–7.8）。

7.1 On-policy vs Off-policy：其实有三个策略

严格的 on-policy 要求 rollout 用的 policy 和优化用的 policy 完全相同。LLM RL 实际几乎不可能严格 on-policy。准确的描述需要三个策略而不是两个：

  • πθ：当前正在优化的策略；
  • πold：训练引擎用 rollout 时刻的权重算出的「旧策略」（PPO ratio 的分母）；
  • πrollout：推理引擎（vLLM/SGLang）实际采样时的策略。

教科书隐含假设 πold = πrollout，实践中两者在三个维度上分离：

  • 数值失配：同一份权重，vLLM 与 FSDP/Megatron 的 kernel 实现、归约顺序、精度路径不同，token 概率可以差出几个量级——名义 on-policy 实为 off-policy（§7.2）。
  • 多 epoch 复用：rollout batch 用 θ_t 生成，update 后变成 θ_{t+1}，第二个 mini-epoch 起就是 off-policy——这是 PPO clip 机制本来要处理的部分。
  • 异步 staleness：rollout worker 的权重滞后 trainer 若干版本，长轨迹甚至由多个版本拼接而成（§7.6–7.8）。

7.2 训推失配：你的「on-policy」RL 其实是 off-policy（TIS）

正名：TIS = Truncated Importance Sampling 社区（包括一些框架文档）常把 TIS 误展开成 "Token Importance Sampling"。在训推失配语境下，TIS 的标准全称是 Truncated Importance Sampling——「截断的重要性采样」，名字说的是对 IS 权重做单侧截断 min(ρ, C)；它确实最常作用在 token 级，但 token 级是它的「作用粒度」，不是它的名字。

Feng Yao 等人的 Your Efficient RL Framework Secretly Brings You Off-Policy Training（博客 2025-08；论文版 On the Rollout-Training Mismatch in Modern RL Systems，NeurIPS 2025 workshop spotlight）首次系统指出：即使权重与架构完全相同，rollout 后端与训练后端也会给出显著不同的 token 概率，把名义上的 on-policy RL 隐式变成 off-policy，长期积累导致崩溃。修正办法：在 PPO/GRPO 的 per-token loss 外再乘一个「训练引擎旧策略 / 推理引擎策略」的截断 IS 权重：

$$ \mathcal{L}^{TIS}_t = \underbrace{\min\!\Big(\tfrac{\pi_{\text{old}}^{\text{train}}(a_t\mid s_t)}{\pi^{\text{rollout}}(a_t\mid s_t)},\, C\Big)}_{\text{训推失配修正（单侧截断）}}\ \cdot\ \underbrace{\min\big(\rho_t(\theta)\, A_t,\ \text{clip}(\rho_t(\theta), 1\!-\!\varepsilon, 1\!+\!\varepsilon)\, A_t\big)}_{\text{常规 PPO clip，}\ \rho_t = \pi_\theta / \pi_{\text{old}}^{\text{train}}} $$

四个要点：

  • 截断是单侧的（只压上界），被截断的 token 梯度缩小但不归零——与 PPO clip「整段丢梯度」不同；
  • 分子 πoldtrain 是训练引擎对 rollout 时刻权重重算的 logprob；分母 πrollout 必须是推理引擎在采样时刻记录的 logprob（见 §7.8 的 old_logprob 纪律）；
  • token 级截断阈值 C = 2.0 是 verl 的默认值，也是实测最稳健的起点；
  • 该修正已被 verl、OpenRLHF、SkyRL、OAT、Open-Instruct 等主流框架内建。verl 的接口：algorithm.rollout_correction.rollout_is = "token"|"sequence" + rollout_is_threshold（默认 2.0），另有拒绝采样式的 rollout_rs（MIS：对失配过大的 token / 序列直接 mask 掉，作为 IS 的「安全阀」）。

来源：fengyao.notion.site/off-policy-rl · openreview.net/forum?id=8MHqvb4lK9 · verl Rollout Correction 文档

7.3 监控：per-token KL 与 ESS

上修正之前先上监控。两个仪表盘指标：

  • 训推 per-token KL：对同一条轨迹的同一批 token，比较推理引擎记录的 logprob 与训练引擎重算的 logprob，用 k3 估计器（$k_3 = r - 1 - \log r$，$r$ 为两引擎概率之比）估计逐 token KL。slime 团队的实测量级：dense 模型约 1e-5 ~ 1e-3，MoE 模型约 1e-3 ~ 1e-1（router 的 top-k 决策对微小数值差异极敏感，token 被送进不同 expert，失配被放大成行为级差异）。该 KL 随训练单调上爬、或在 MoE 上突然跳档，是崩溃前兆。
  • ESS（有效样本量）：IS 权重归一化到均值 1 后 $\text{ESS} = 1/\overline{w^2}$（verl 指标名 rollout_corr/rollout_is_eff_sample_size）。ESS ≈ 1 表示几乎所有样本都在有效贡献梯度；ESS 跌向 0 表示少数轨迹主导更新。token-TIS（C=2.0）的健康区间是 0.9–1.0；序列级 IS 在长序列上 ESS 常跌到 0.3–0.4、80–90% 样本权重趋零——这是序列级路线失效的直接证据（§7.4）。

verl 的 rollout correction 还会记录 kl、chi2_token / chi2_seq（χ² 散度，IS 方差的解析代理）、ppl_ratio 等，应全部进 dashboard。

7.6 异步训练与 staleness：AReaL 的 decoupled PPO 与三种控制策略

同步 RL 的吞吐被最长的 rollout 拖死（长尾），异步化是 2025H2 后所有主流框架的方向——HuggingFace 对 16 个开源 RL 库的普查显示全部收敛到「推理卡与训练卡物理分离 + 异步推送权重」的 disaggregated 架构。代价是 staleness：训练数据由滞后 1~N 个版本的策略生成。

AReaL：decoupled PPO objective

AReaL（arXiv 2505.24298）是异步 RL「算法-系统协同设计」的范本。它把标准 PPO 里被混用的两个角色拆开：行为策略 πbehav（实际生成数据的、可能跨版本拼接的旧策略）与近端策略 πprox（最近一次更新前的策略，作为 trust region 锚点）：

$$ J(\theta)=\mathbb{E}\Big[\sum_t \underbrace{\tfrac{\pi_{\text{prox}}(a_t|s_t)}{\pi_{\text{behav}}(a_t|s_t)}}_{\text{staleness 修正（clip 外）}} \min\big(u_t(\theta)\hat A_t,\ \text{clip}(u_t(\theta),1\!-\!\varepsilon,1\!+\!\varepsilon)\hat A_t\big)\Big],\qquad u_t(\theta)=\tfrac{\pi_\theta(a_t|s_t)}{\pi_{\text{prox}}(a_t|s_t)} $$

关键设计：staleness 的 IS 修正放在 clip 之外，clip 的中心是新鲜的 πprox 而非陈旧的 πbehav——行为策略再旧，trust region 始终围绕当前策略附近，陈旧度只通过权重缩放体现（对照 §7.2：TIS 修失配、这里修 staleness，结构同源）。配合两个系统机制：interruptible rollout（update_weights 请求到达即中断生成，丢弃旧权重算的 KV cache、用新权重重算后继续解码）与staleness 上界 η（系统直接拒绝可能违反 ⌊(Nr−1)/B⌋ ≤ i+η 的新生成请求），AReaL 拿到至高 2.77× 加速且效果持平或更好。

7.8 Partial Rollout 与 in-flight 权重更新

异步训练绕不开的具体问题：权重更新到达时，总有一批长生成只完成了一半（partial rollout）。处理方式按「正确性 vs 吞吐」取舍分四档：

方式机制KV cache 正确性代表实现
abort-and-retry丢弃在飞请求，更新后重新提交无跨版本问题SkyRL、slime。代价：已花费的 rollout 算力全部浪费
prefix-resume（续写）缓存已生成 token，权重同步后从前缀继续；旧权重算出的 KV cache 不重算、直接容忍跨版本（轨迹由多个策略版本拼接）verl partial rollout（save/resume）；Magistral：NCCL 在线广播新权重（单次更新 <5s），论文明确「重算 KV cache 没有必要」，一条回答可能经历多次权重更新
interrupt + KV 重算中断生成，丢弃旧 KV、用新权重重算后续写重算消除（代价：重算 FLOPs）AReaL；其 decoupled PPO（§7.6）在理论上覆盖跨版本轨迹
in-flight 热切换以单次 forward pass 为粒度加锁，权重切换只等几毫秒，生成永不停止跨版本（同一条序列前半旧权重、后半新权重）PipelineRL（约 2× 学习提速、数据始终接近 on-policy）；vLLM 2026-05 原生 API 标准化

vLLM 的原生 RL API（2026-05）把这一层从「各框架自造轮子」收编成标准件：weight transfer engine 提供 NCCL（跨卡/跨节点 broadcast）与 CUDA IPC（同卡共享内存句柄）两种后端，均支持 packed tensor 减少序列化开销；pause_generation(mode=...) 提供 abort / wait / keep 三种模式——keep 模式暂停调度但保留在飞请求状态，resume 后同一条请求「pause 前的 token 来自旧权重、resume 后的 token 来自新权重」；DP 部署下用两阶段 pause 协议（先 local pause、所有 rank 确认后再 global pause）防死锁。

old_logprob 纪律：必须记录「生成时刻」的 logprob 跨版本轨迹 + 训推失配的双重作用下，唯一可信的行为策略概率是推理引擎在采样那一刻输出的 logprob（带 model_version 标签）。两个常见错误：(1) 事后用当前权重重算 old_logprob——ratio≡1，clip 与 IS 全部失效，训练表面平稳实则没有任何 trust region 保护；(2) 比较 raw 与 processed 两种语义不一致的 logprob（§7.5）。per-token 记录 model_version + rollout logprob，是上表所有跨版本方案能成立的共同前提。

8. Reference Model 与 KL Penalty

8.1 是否需要 reference model？

取决于训练目标和正则化哲学：

  • PPO + KL penalty：必须有 ref。KL 通常加进 reward 或加进 loss。
  • DPO：必须有 ref。implicit reward 形式里 log π/π_ref 离不开 ref。
  • SimPO / ORPO / CPO：reference-free。SimPO 直接以 1/|y| · logπ 作为 reward；ORPO 用 odds ratio。
  • GRPO（DeepSeek 原版 / R1）：有 ref。R1/R1-Zero 论文的 GRPO 目标显式包含 -β·D_KL(π_θ‖π_ref) 项；但后续 RLVR 工作里 ref 越来越淡，许多 GRPO 实现完全去掉了 ref（依赖 PPO clip 做约束）。
  • Dr.GRPO / DAPO：明确删掉了 KL 项，训练时不需要 reference model。DAPO 论文原话是 "we exclude the KL term"——长 CoT 训练中策略分布本来就要大幅偏离初始模型，KL 锚定没有必要；Dr.GRPO（出自 Understanding R1-Zero-Like Training）全程设 β=0，顺便省下 π_ref 的显存与计算。

KL 去留决策表

2025 年之后社区在「KL 去留」上出现清晰的分水岭：RLVR / agentic RL 主流是删 KL，RLHF 对齐场景主流是保 KL。按场景查表：

场景KL 去留理由代表工作
RLVR / Reasoning RL / agentic RL（rule-based reward）删（主流）reward 可验证、hack 面小；长 CoT / 长轨迹要求策略大幅偏离初始模型，KL 锚定反而碍事；删掉还省下 π_ref 的显存与前向。DAPO（"we exclude the KL term"）、Dr.GRPO（全程 β=0）、Open-Reasoner-Zero（vanilla PPO，无任何 KL 正则）、Magistral（"remove the KL penalty entirely"）
RLHF 对齐（neural RM）保RM 是 proxy，OOD 区域必被 hack，KL 是对 RM 失真的兜底（§1.3）。InstructGPT（per-token KL 放进 reward）；R1/R1-Zero 的 GRPO 目标同样保留 KL 项
长程 RL（数千 step、防能力边界收缩）保 + 周期性 reset长期训练防熵塌缩与漂移；把 ref 定期重置为当前 policy，约束不至于锁死。ProRL（KL divergence control + reference policy resetting）
在意灾难性遗忘保留派论据RL's Razor：遗忘程度可由与 base 的 KL 预测，on-policy RL 隐式偏向 KL 最小解——显式 KL 锚定是同方向的保险。RL's Razor（arXiv 2509.04259）

保留 KL 时还有一个常被忽略的自由度：放 reward 还是放 loss。放 reward（InstructGPT 式：reward 减去 per-token 的 β·log(π/π_ref)）会进入 return / GAE，经 credit assignment 影响前序 token 的 advantage；放 loss（GRPO 式：k3 作为独立正则项）则不与 advantage 纠缠（见 §2.2 实战要点）。两种写法数值和行为都不等价，迁移别人配方时先确认 KL 放在哪一侧。

8.2 KL 系数调参

  • RLHF：β ≈ 0.01 ~ 0.1（取决于 reward scale）。
  • Reasoning RL：β ≈ 0.001 ~ 0.01（per-token KL）。注意归因：R1/R1-Zero 论文的 GRPO 目标是保留 KL 项的；公开把 β 设 0 / 整段删 KL 的是 DAPO、Dr.GRPO、Open-Reasoner-Zero 等后续与复现工作（见 §8.1 决策表）。
  • DPO：β ≈ 0.05 ~ 0.5。β 越大越保守（贴近 ref），越小越激进。β 太小时 reward margin 容易爆炸。

8.3 移除 reference model 的 trade-off

方法移除 ref 的代价怎么补
SimPO失去 KL 锚定，长期训练有 catastrophic forgetting 风险length-normalized reward + target margin γ
ORPO同上用 odds ratio 作为隐式正则
GRPO 无 ref只剩 PPO clip 约束更小的 LR + 更紧的 clip range
什么时候去掉 ref 更合算 当 base model 本身已经经过强 SFT、preference 数据噪声相对小、训练 step 数不多时，去掉 ref 能省下一份权重显存和一次前向计算而几乎不损失性能（注意省不到一半：ref 没有梯度和 optimizer state）。但若 base 较弱 / 数据噪声大 / 要训很久，建议保留 ref。

9. 数据相关坑

9.1 Preference 数据噪声

  • 人类标注一致率通常 60%~75%，DPO 直接信号噪声率高。
  • 缓解：cDPO（label smoothing 风格）、Identity-PO（IPO 防止过拟合到 deterministic preference）、多人投票聚合、置信度过滤。

9.2 Reward Model Overfitting

  • RM 在 OOD（policy 已经远离 ref）区域给出乐观分数 → policy 朝伪山顶冲。
  • 检测：保留 held-out human eval；监控 reward score 和真实 eval 的相关性曲线。
  • 缓解：RM ensemble、conservative RM (取 min)、信息瓶颈正则。

9.3 Chat Template / 特殊 token 陷阱

这是被低估的 bug 来源：

  • 训练用 <|im_start|>，推理用 [INST] → 模型表现莫名变差。
  • tokenizer 的 EOS 与 chat template 的 stop token 不一致 → 模型把对话延续下去自问自答。
  • RLHF 阶段忘记 mask prompt token 的 loss，policy 学会改写 system prompt。
  • DPO 数据集里 chosen / rejected 共享同一段 prefix 但 tokenization 边界不同（空格、BOS 位置），导致 ref logprob 不匹配。
典型 debug 流程 RL 训不动 / 输出乱码时第一步永远是：用同一段对话过 tokenizer 后再 decode 回来，对比训练 / 推理两条路径的 token id 序列是否完全相同。一个 BOS 或空格的差异足以让训练崩盘。

9.4 难度分布

  • Reasoning RL 中如果 prompt 全部太简单（pass rate ≈ 1）或全部太难（pass rate ≈ 0），group advantage 都是 0，"没有梯度"。
  • DAPO 的 dynamic sampling：把 accuracy = 0 或 1 的 group 直接过滤掉，重新采样，保证 batch 内全是有梯度的 group。
  • 课程学习（curriculum）：按难度排序，逐渐提升。但纯 RL 里效果不如 dynamic filtering 直接。

10. 基础设施坑

10.1 4 模型的显存账

标准 PPO RLHF 需同时持有 policy / reference / reward / value 四个模型，加上 rollout engine 缓存的 KV cache。以 7B 模型 BF16 为例：

  • 4 × 14GB = 56GB 仅权重
  • Policy 还要 optimizer states（Adam 2 份）+ gradients = +84GB
  • Activation + KV cache + intermediate buffer ≈ +30GB
  • 合计约 170~220GB，必须多卡。

10.2 节省显存的标准做法

  • Ref / RM shared backbone：让 policy 和 ref 共享 base model 权重（policy 仅维护 LoRA delta，关闭 adapter 即得 ref）。TRL 的 PEFT 集成早已内建这一做法，OpenRLHF 等框架同样支持。
  • LoRA / QLoRA RL：只训低秩 adapter，4 模型可放进单 A100。
  • FSDP / ZeRO-3：跨卡 sharding 权重。
  • Offload：ref 和 RM 不动，可 offload 到 CPU；调用时再 swap 进来。
  • GRPO / RLOO 路线：去掉 value model（用 group baseline 或 leave-one-out）。Dr.GRPO / DAPO 都是这条路。
  • SimPO / ORPO：reference-free，省掉 ref model。

10.3 Rollout 效率瓶颈

实测发现：RL 训练的 wall-clock 时间里 60%+ 花在 rollout 阶段（尤其是 long CoT）。瓶颈来源：

  • Decode 阶段的 memory-bound：vLLM 的 PagedAttention、SGLang 的 RadixAttention 都是缓解这一点。
  • Tail latency：N 个 rollouts 里最长的那个决定 batch 完成时间。Speculative decoding、early stop、chunked rollout 都可缓解。
  • Tokenizer overhead：高频小调用 tokenizer.encode 比想象的慢，要 batch 化。

10.4 多机同步开销

  • 每个 update step 之后，trainer 的新权重要广播到所有 rollout worker。70B 模型这一次 broadcast 可能 30s+。
  • Colocated（推训同卡）：weight sync 几乎免费（in-place reshard）。
  • Disaggregated（推训分卡）：必须 NCCL broadcast 或 RDMA，verl/SGLang server adapter 抽象了这一层。

10.5 vLLM / SGLang 与训练 engine 的权重同步

  • 常见做法：训练 engine 保存权重 → 通过 NCCL 直接 reshape 到 rollout engine 的 tensor parallel layout。
  • 陷阱 1：tensor parallel size 不同（trainer TP=8, rollout TP=4），需要 reshard。
  • 陷阱 2：rollout engine 用 fp8 / int8，必须每次重新 calibrate scale。
  • 陷阱 3：Generation kernel cache 失效，第一次 step 慢。
  • 陷阱 4：prefix caching 会跨权重更新复用过期状态——做训推 parity 测试时必须显式关闭；in-flight 更新场景用 keep-mode + clear_cache=False（见 §7.5 清单与 §7.8）。
  • verl / OpenRLHF / AReaL / slime 等开源框架都把这一层做成了可插拔接口；vLLM 2026-05 起提供原生 weight transfer engine（NCCL 跨节点 / CUDA IPC 同卡）与 pause/resume API，weight sync 正从各框架自造轮子走向标准化（§7.8）。

11. 评测与监控

11.1 "Reward score 上去但模型变差"

这是 over-optimization 的标志，§1 已述。监控办法：

  • 定期跑 held-out 真实评测（MT-Bench、Arena-Hard、AlpacaEval 2 LC、AIME/MATH for reasoning）。
  • 跟踪 reward 和 real-eval 的相关性。Gao 等的 scaling law 给出经验曲线：随着 KL 增大，真实 eval 先升后降。
  • 用 best-of-N 监控 RM 自己的 OOD：N=64 的 BoN reward 若涨幅远大于 average reward，说明 RM 给出尾部高分但 policy 还没学到（仍可继续训）；若 BoN 已接近 average，说明 RM 上限到了。

11.2 必看的训练指标

11.3 防止过度优化的策略

  • Early stopping 配 real-eval。
  • KL budget：硬性规定总 KL 不超过某值，到达即停。
  • RM ensemble：每个 RM 的 OOD 区域不同，ensemble 平均能压低尾部乐观。
  • WARM / WARP (Warm-up + averaging）：多个 RM 在 checkpoint 维度上 average，提升 robustness。

12. GRPO / Reasoning RL 专属问题

12.1 Rollout 温度选择

  • Reasoning task 上 T = 1.0 是事实标准（DeepSeek-R1、DAPO 都是）。低于 0.7 会很快 entropy collapse；高于 1.2 大量乱码 rollout，浪费计算。
  • Top-p = 0.95~1.0；Top-k 通常不开（k=∞）。
  • 近期工作（多温度策略，arXiv 2510.08892）尝试 reasoning token 用高 T、knowledge token 用低 T，token-level mixed temperature。

12.2 N 个 samples 的影响

  • N 太小（< 4）：group baseline 估计噪声大，advantage 不稳。
  • N 较大（8~16）：DeepSeek-R1 / DAPO 通用配置。
  • N 很大（> 32）：rollout 阶段成本爆炸，且收益边际递减。
  • 动态 N：DAPO 在 dynamic sampling 中根据 group 内 reward variance 决定要不要补样。

12.3 Rule-based Reward 的设计

  • Verifiability first：能用规则验证的（数学最终答案、代码 unit test、formal proof）就别用 RM。
  • Format reward vs accuracy reward：分开打分。R1 论文只说明用了 accuracy reward 与 format reward 两类，并未公布具体数值；format 0.1 / accuracy 1 之类的配置出自开源复现而非原论文。原则不变：format 仅作为 hygiene，权重应远小于 accuracy。
  • 避免 partial credit：rule reward 给「答错但思路对」加分会被 hack；纯 0/1 最稳。
  • 多步答案抽取的鲁棒性：用 regex 抽取 \boxed{}、#### 等，但要小心模型把多个 box 都写进去。

12.4 推理长度爆炸

R1 公开训练曲线显示 response length 从 ~1k 涨到 ~10k 是性能提升的副产物。但失控时表现为：

  • 无意义的 "Wait, let me reconsider..." 循环。
  • 反复重写已经做过的步骤。
  • 在错误答案上越想越长（Dr.GRPO 现象）。

缓解：

  • DAPO overlong reward shaping。
  • Dr.GRPO 的 token-level loss 修正。
  • Adaptive Length Penalty（按 prompt online solve rate 调节）。
  • Length budget reward：超过 L_max 直接判 0。
  • 显式 stop token 训练（在 SFT 阶段教 </think> 自然结尾）。

13. Agentic RL 专属问题

13.1 多轮 Credit Assignment

Agentic 场景轨迹长（50~200+ turn），但 reward 通常是 outcome-only（任务最终成功/失败）。如何把稀疏 outcome reward 分配到每个 turn / token 是核心难题。

  • Trajectory-level reward + uniform broadcast：最简单做法，把 outcome reward 平摊到所有 token。缺点：好的步骤和差的步骤一视同仁。
  • Turn-level reward design：每个 turn 局部打分（代码执行结果、工具输出是否符合预期）。
  • Hybrid reward：可验证 turn 用 rule（代码运行成功 / DB 查询正确），不可验证 turn 用 LLM-as-judge。
  • Process Reward Model (PRM)：训练一个 step-level RM，但容易 reward hack（DeepSeek R1 论文明确放弃）。
  • Online PRM（OPRL 等）：在线训练 PRM，避开 offline OOD 问题。
  • Implicit step reward：从最终 outcome 反推每个 turn 的贡献（类似 advantage 反向传播）。

13.2 工具调用失败的处理

  • 工具调用语法错（JSON 解析失败、参数缺失）：通常给 0 reward 或小负 reward，但要避免模型完全放弃工具。
  • 工具返回 error 但语法对：是「环境噪声」还是「策略选择不当」需要区分。完全归咎于 agent 不公平。
  • 实战经验：把 tool 错误信息原样塞回上下文，让 agent 学会处理 error，比直接惩罚更有效。
  • Format reward 单独打分（tool call 格式 OK + 0.1）。

13.3 环境 Stochasticity

  • 同一 action 在不同 rollout 中可能得到不同 observation（网络抖动、检索结果变化、API 不确定性）。
  • 概念上要分清：MDP 本身允许随机转移，PPO 并不要求确定性环境，所以这不算「破坏假设」；真正的代价是同一 action 的 reward/advantage 估计方差变大、信号更噪。「非平稳（non-stationary）」专指环境分布随时间漂移，是另一类问题，别混用。GRPO 的 group baseline 多次采样能部分抵消方差。
  • 方案：固定 random seed（如果环境支持）；或显式建模 environment uncertainty。
  • Replay buffer 缓存"环境响应"以保证可复现，verl 的 multi-turn agent 配置提供了这种能力。

13.4 长轨迹的 KL 估计

当一条 trajectory 有几万 token，KL(π‖π_ref) 的累积会很大，掩盖单个 turn 的 policy drift。处理方式：

  • Per-token KL：除以 token 数，比较 average KL。
  • Per-turn KL：按 turn 而非 token 聚合，更贴合 agentic decision 结构。
  • Turn-level importance sampling（arXiv 2511.20718）：把 IS ratio 按 turn 计算而非 token，方差大幅降低。
  • Clipping-triggered normalization：被 clip 的 token 进入归一化通道而非丢弃。
Agentic RL 当前共识 （1）outcome reward 是骨架，但 turn-level dense signal 几乎必需；（2）IS 必须做 turn-level；（3）环境 noise 用 group sampling 缓解；（4）长轨迹下 ref model 的 KL 用 per-token 衡量，避免被长度淹没。

13.5 Echo Trap：多轮 RL 的特征性崩溃模式（RAGEN / StarPO）

失稳模式StarPO-S RAGEN（arXiv 2504.20073）是第一个系统刻画 multi-turn agent RL 失稳规律的工作。其训练框架 StarPO（State-Thinking-Actions-Reward Policy Optimization，轨迹级优化）在 Bandit / Sokoban / FrozenLake / WebShop 四个环境上系统研究训练动态，在前三个符号环境中反复观察到同一种崩溃（开放域的 WebShop 因强语言先验与较高的初始 reward 相对幸免），命名为 Echo Trap：策略过拟合到局部高奖励的推理套路，早期多样的推理逐渐退化成重复、确定性的模板——模型在自己生成的轨迹上训练，不断「回声」已记住的推理路径（论文将其类比 Shumailov et al. 的 model collapse）。这不是普通的 entropy collapse 在多轮场景的复刻：多轮 rollout 中「环境反馈→下一步生成」的自激回路让它来得更快、更难逆转。

三联征（按出现顺序）

  1. 组内 reward 方差（reward std）塌缩——最早的预警信号，先于性能崩溃几十步出现。论文实测：FrozenLake-PPO 的 std 在 step 40 骤降，reward mean 直到 step 90 才崩；Bandit-PPO 的 std 在 step 70 左右触底，先于 step 120 的 reward 峰值。此时表面指标（reward、success rate）还在涨，极具迷惑性。
  2. 熵骤降 / 熵轨迹异常——健康训练的熵应当平稳缓降（与 §5.3 的「健康曲线」一致）；骤降意味着过早确定性收敛，骤升或剧烈震荡同样对应推理行为已崩坏。
  3. 梯度尖峰——不可逆点。论文实测尖峰出现在 step 170（Bandit）/ 110（Sokoban）/ 90（FrozenLake），此后即使很小的参数更新也会引起剧烈的 loss 漂移，基本无法恢复，只能回滚 checkpoint。

StarPO-S 的三项解法

解法机制实操要点
① 基于方差的轨迹过滤
(uncertainty-based instance filtering)		对每个初始状态（prompt）计算组内 rollout reward 的 std，作为该实例的「不确定度」，只保留 std 最高的 top-p% 实例参与更新论文默认 p = 25%（激进档）；PPO 上仅保留高方差的 50% 即可完全避免崩溃，保留 75% 也能把 FrozenLake 的稳定期从 100 步延到 140 步。直觉：全对/全错的 prompt 信号量为零，而相似轨迹支配 batch 正是 Echo Trap 的成因

后续：RAGEN-2（arXiv 2604.06268）指出还存在熵指标看不出的 template collapse——输出表面多样但对输入不敏感的固定模板——需用互信息代理在线监测，并把方差过滤推广为 SNR-Aware Filtering。参考：arXiv:2504.20073（RAGEN/StarPO）、arXiv:2604.06268（RAGEN-2）、arXiv:2503.14476（DAPO，梯度整形技巧来源）。

14. RLVR 实验有效性陷阱

前 13 节讲的都是「训练怎么会坏」。这一节讲一个更隐蔽的问题：训练一切正常、分数确实涨了，但结论可能整个是错的。2025 年中之后，「只在 Qwen 底座上验证的 RLVR 改进不可信」已成为审稿与工业界的硬性共识，源头是下面两条证据链。不懂这一节，会系统性误读大量 RLVR「改进」论文，也会误读自己的内部实验。

14.1 Spurious Rewards：随机奖励也能涨 21 个点

Shao et al.（arXiv 2506.10947）对 Qwen2.5-Math-7B 用 GRPO 做 RLVR，把训练信号逐级「掺假」，MATH-500 绝对涨幅如下：

纯随机奖励拿到了真奖励七成以上的涨幅；γ ∈ {0.7, 0.5, 0.3, 0.001} 的任意非零概率最终都收敛到 15~20 点的提升（γ = 0 即恒定奖励，梯度恒为 0，无任何变化，符合解析预期）。而同样这套「假奖励」换到 Llama3 / OLMo2 上不仅普遍无效，常常直接掉分。

发生了什么：Qwen2.5-Math 预训练里已带有高频的 code reasoning 行为——在回答中写 Python 风格代码辅助推理但并不真正执行，base model 出现率 65%。RLVR——无论奖励真假——把这一行为频率推到 90% 以上，而该行为本身与更高正确率强相关。换句话说，涨分来自放大预训练已有的高先验行为，而不是奖励信号教会了模型任何新东西；Llama / OLMo 没有这种可被放大的行为，所以假奖励失灵。

14.2 机制：是 clipping bias 在驱动，不是奖励

随机奖励下，组内归一化的 advantage 期望为 0，照理说期望梯度也应为 0。但 GRPO loss 里的 clip 项打破了对称性：

随机奖励 ⇒ $\mathbb{E}[A_i] = 0$。若目标对 $\rho_t A_i$ 是线性的，则 $\mathbb{E}[\nabla_\theta J] = 0$；
但 $\min\!\big(\rho_t A_i,\ \text{clip}(\rho_t, 1-\varepsilon, 1+\varepsilon) A_i\big)$ 对 $A_i$ 的符号不对称 ⇒ $\mathbb{E}[\nabla_\theta J] \neq 0$， 且方向系统性偏向 $\pi_{\text{old}}$ 的高概率（高先验）行为。（推导见原论文 Appendix B.1.2）

消融验证：用三种方式去掉 clipping——(i) 实现里直接禁用；(ii) mini-batch size 提到与 rollout batch 相同；(iii) 缩小 rollout batch——后两者保证每个 rollout 阶段只做一次梯度更新，$\pi_\theta = \pi_{\text{old}}$，clip 永不触发。三种设置下，随机奖励都不再带来一致的涨分，平均 token 概率与 code-reasoning 频率也停止上升（实现级禁用 clip 的版本跨种子方差大、偶现高分；调 batch 的两种消融更稳定但梯度更新次数少 8 倍——见原论文 Figure 4 / 9 / 10）；而开启 clipping 的对照组稳定提升。

工程含义 只要你的 GRPO 每个 rollout batch 做多次 mini-batch 更新（几乎所有默认配置都是），clip 偏置就在系统性放大模型先验。后续工作（arXiv 2512.16912，ICLR 2026）进一步厘清机制：spurious reward 下 clip 偏置的净效应是降熵——把策略推向更高置信、更确定的输出；但反过来单纯做 entropy minimization 并不足以带来提升，说明起作用的是「clip 偏置 × 模型已有先验（含数据污染）」的交互而非降熵本身。下次看到「弱监督 / 无监督 / 自奖励 RLVR 也有效」的结果，先想到这一层，再谈方法贡献。

14.3 Dr.GRPO 的另一半发现：底座与模板的混淆

§3.4 讲了 Dr.GRPO（arXiv 2503.20783）对 GRPO 的两处去偏；同一篇论文的另一半发现对实验设计同样关键，却更少被引用：

  • Qwen2.5 base 不是干净的 base model：不加任何模板时，Qwen2.5-Math-7B 在五个数学基准（AIME24 / AMC / MATH500 / Minerva / OlympiadBench）上平均 38.2%，远高于传统 4-shot 提示的 23.8%。作者据此推测其预训练混入了拼接的问答文本（直接最大化 $\log p_\theta(\mathbf{q};\mathbf{o})$），相当于「自带 SFT」。
  • 模板错配会先毁掉能力，再由 RL「恢复」：同一个模型套 R1 template 平均分跌到 0.0，Qwen template 26.5。如果 RL 的起点是被错配模板压低的 base，训练曲线的大部分上升只是在恢复模板毁掉的既有能力——你测到的是 recovery，不是 learning。报告 RL 增益前必须先报告正确口径的起点。
  • 「Aha moment」在 base model 中已存在：包括 DeepSeek-V3-Base 在内，几乎所有被测 base model 在任何 RL 之前就会生成 "Aha" / "wait" 等自我反思关键词；且论文观察到反思词频与正确率并不正相关。把训练中反思词频上升当作「能力涌现」的证据，不可靠。

15. DPO 专属坑

前面各节的主语大多是 on-policy RL（PPO / GRPO）。DPO 容易给人「不过是个二元交叉熵微调」的错觉，于是最高频的死法是拿 SFT 的肌肉记忆来训 DPO：沿用 SFT 学习率、刷 2~3 个 epoch、盯着 loss 下降就收工。本节集中讲 DPO 专属的工程配方、TRL 监控面板的指标语义，以及「loss 在降但其实什么都没学会」的评估陷阱。本文其他章节已覆盖的 DPO 议题：reverse KL 与多样性（§2.4）、length-hack 机制（§3.2）、偏好标签噪声（§9.1）、chosen/rejected tokenization 边界（§9.3）。

15.1 学习率：DPO 的第一杀手

超参 DPO 的学习率必须比 SFT 低 1~2 个数量级，主流取 5e-7（可用范围约 1e-7 ~ 1e-6）。直接沿用 SFT 的 1e-5 / 2e-5 几乎必然训崩——典型症状是输出重复、空响应、语言混杂，而且 loss 看起来降得飞快，极具迷惑性。

为什么 DPO 吃不住大学习率？三个机制相互叠加：

  1. 目标只约束相对 margin，对绝对似然没有显式锚。降 loss 最快的方向是无差别压低 rejected 的似然——TRL 文档的原话：实践中 margin 拉大「主要靠压低 rejected 实现，而非提升 chosen」。lr 一大，与 rejected 共享大量 token 的 chosen 会被一起压塌（§3.2 的 logp 求和结构 + likelihood displacement），生成分布在几百步内劣化。
  2. 初期梯度全 batch 同向。训练开始时 $\pi_\theta = \pi_{ref}$，每对样本的隐式 reward margin 都是 0，梯度权重 $\sigma(0)=0.5$ 对所有 pair 一致饱满；不像 RL 有 advantage 正负对冲，DPO 的对比梯度方向高度一致，同样的 lr 下等效步长远大于 SFT。
  3. 起点是已经训好的 SFT checkpoint。DPO 的任务是微调相对偏好，不是学新分布，预算本来就该小。
$$ \nabla_\theta\mathcal{L}_{DPO} = -\beta\,\mathbb{E}\Big[\underbrace{\sigma\!\big(\hat r_\theta(y_l)-\hat r_\theta(y_w)\big)}_{\text{估错程度，初始}=0.5}\big(\nabla_\theta\log\pi_\theta(y_w|x) - \nabla_\theta\log\pi_\theta(y_l|x)\big)\Big],\qquad \hat r_\theta(y)=\beta\log\frac{\pi_\theta(y|x)}{\pi_{ref}(y|x)} $$

15.2 Epoch 数：1 是默认值

  • 1 epoch 起步，最多 2~3。Tülu 3 全系列（8B/70B/405B）都是 1 epoch。
  • Zephyr 的例外与教训：原始配方训了 3 epochs，且论文观察到 1 epoch 后训练集 accuracy 即达 100%（完全过拟合），但 MT-Bench / AlpacaEval 没有因此变差——「DPO 目标上的过拟合 ≠ 下游退化」。但别把这当通行证：同一篇论文也发现若 SFT 阶段超过 1 epoch，更长的 DPO 训练反而引起性能回退。后来 alignment-handbook 证实 1 epoch + β=0.01 就能复现 zephyr-7b-beta 同级性能，3 epochs 并非必要。
  • 想要更多训练量，正确姿势是 iterative DPO 而非加 epoch：每轮重新采样、重建偏好对、把 ref 更新为上一轮策略。固定离线数据上反复刷 epoch，margin 还在涨，但涨幅来源越来越多是压 rejected / displacement，而不是学偏好。

chosen 和 rejected 的 logprob 一起降 ≠ bug 新手最高频误报。DPO 只优化两者之差，没有任何项要求 chosen 的绝对似然上升；被腾出来的概率质量流向偏好对之外的序列（Razin et al. 对 likelihood displacement 的理论刻画：流向何处取决于 chosen/rejected 的嵌入相似度，最坏情况流向语义相反的响应）。判断用组合条件——margins 在涨 + accuracies 在升 + 抽样生成质量正常 ⇒ 健康；rewards/chosen 大幅下坠 + 生成开始重复 / 拒答行为异常 ⇒ displacement，对策：用 CHES score 过滤嵌入相似度过高的偏好对、给 chosen 加 NLL 辅助损失（DPO+NLL / DPOP，对 chosen/rejected 编辑距离小的数学、代码数据尤其必要）、降 lr。

RLHF Book 第 12 章把 sanity check 总结为一句话：implicit reward margin 朝正确方向移动，同时模型的抽样生成没有崩——前半句看面板，后半句必须真的去采样。DPO 不像 RL 训练自带 rollout，不主动采样就完全看不到生成质量：每 N step 用固定 prompt 集生成、人工或 judge 抽检，是 DPO 监控里不可省的一环。工程提示：ref model 的 logprob 可在训练前一次性预计算（TRL precompute_ref_log_probs），训练时不再常驻 ref，省一份完整权重的显存。

15.5 ranking accuracy ~60%：loss 下降 ≠ 学会排序

评估陷阱 Chen et al.（NeurIPS 2024）系统测量了偏好对齐模型的 ranking accuracy（模型隐式 reward 把 $y_w$ 排在 $y_l$ 前面的比例），结论颠覆直觉：

  • 实测：SOTA 偏好对齐模型在常用偏好数据集（UltraFeedback / HH-RLHF / SHP）上的 ranking accuracy 普遍低于 60%——Llama-2-7B-chat 约 40%（length-normalized 口径 53%）、Zephyr-7B-β 42~54%、Gemma-7B-IT 40~54%。Zephyr 之于 UltraFeedback 即「在与自己偏好训练同源的数据上」也只有这个水平。
  • 理想化对照：若完美优化 DPO 目标，由 ref model 与 β 可闭式算出的「理想 ranking accuracy」多在 90~99%；实测与理想之间存在 19~59 个百分点的 alignment gap。
  • 机制：DPO 训练中 loss 持续下降，但几乎从不翻转 ref model 排错的 pair——把本来就排对的 pair 的 margin 拉大就足以让 loss 一直降；论文从理论与实证两面证明，DPO 目标连 ref model 轻微的排序错误都极难纠正。所以 loss 曲线漂亮和「学会了偏好排序」是两回事。
  • 与生成质量的关系：ranking accuracy 与 win rate 只在策略仍接近 ref（KL 小）时正相关；偏离远了之后两者甚至反相关——又一条「离线指标在远离初始化后失效」的证据，与 §11.1 的 proxy reward 相关性曲线同构。
操作规则 （1）rewards/accuracies 只当训练健康度指标，不当效果指标；它逼近 1.0 往往是坏消息（过拟合，对照 Zephyr：1 epoch 训练集即 100%）。（2）held-out ranking accuracy 在 60% 上下是常态而非 bug，看到不要恐慌、也不要拿它邀功。（3）评估三件套缺一不可：固定 held-out 偏好集的 implicit reward accuracy + 下游生成评测（AlpacaEval 2 LC / Arena-Hard / MT-Bench，LC 防 length-hack，§3）+ 人工抽检。（4）不同 xPO 的 loss / margin 数值不可横比（§15.3）。

结语

RL for LLM 是一个 系统工程，而非单一算法。社区在 2023~2025 这段时间内已经把许多坑明确化：reward hacking 几乎是必然的，问题是「在拐点前停下来」；length bias 不只是 RM 的问题，是 loss aggregation 的问题（Dr.GRPO、SimPO 都从这一点重构）；entropy collapse 看似训练问题，本质是 clip 几何决定的（DAPO 的 clip-higher 是漂亮的洞察）；数据格式 bug 永远是 P0 排查项；基础设施层面，权重同步、rollout 效率、显存账永远在拉低实际可用的实验吞吐。

对实战工程师而言最有用的反而不是某个 SOTA 算法本身，而是：一套覆盖完整的监控指标 + 一份能复现的 baseline + 对每个超参方向变化的直觉。这份综述希望能在这三点上都给到具体落点。
