1. 总览：2025Q4–2026H1 时间线与格局

2025 年 Q4 到 2026 年 H1 是后训练领域自 DeepSeek-R1（2025-01）以来变化最密集的三个季度。如果说 2025 上半年的主旋律是「把 GRPO 跑稳」（DAPO 四件套、训推一致性、entropy 管理——详见 坑点综述），那么这三个季度的主旋律变成了三件事：

  1. RL 从「一锅炖」走向「结构化」。没有任何一家旗舰还在用单阶段混合 RL：GLM-5 用严格顺序的三段 RL 流水线，DeepSeek-V4 干脆把多领域混合 RL 整个替换成「领域 specialist 各自 RL + on-policy distillation 合并」，Qwen3-Coder-Next 在 80B-A3B 的小激活模型上复刻了同样的「专家分训 + 蒸馏回收」结构。多任务 RL 奖励互相打架的问题，被各家用不同的结构化手段绕开了。
  2. On-policy distillation（OPD）升格为与 RL 并列的后训练支柱。Thinking Machines 在 2025-10 复现了 Qwen3 的蒸馏配方、并把「约 1/10 RL 成本拿到更高分数」（Qwen3 报告 Table 21 的对照）确立为标准引用，DeepSeek-V4 在 2026-04 把 OPD 推到「全面替代混合 RL」的位置，GLM-5 用它做跨阶段防遗忘（cross-stage distillation），THUNLP 在 2026-04 给出了 OPD 成败的机理条件。
  3. 「思考」本身成为被训练的对象。Gemini 3 在 RL 中对 thought 加 length penalty、Claude Opus 4.6 把思考深度做成模型自调的 adaptive thinking、Kimi K2.5 给并行 agent 体系设计了防「假并行」的奖励——思考预算、思考效率、思考的并行结构，全部从推理时超参变成了训练目标。

1.2 格局判断：配方收敛了什么、分化了什么

已收敛（所有旗舰一致）的部分，可以放心当作 2026 年的默认实践：

  • 异步、解耦的 RL 基础设施。GLM-5 的 slime、Qwen3.5 的全解耦训推架构（端到端提速 3–5 倍）、Prime Intellect 的 prime-rl、ScaleRL 的 PipelineRL——generation 与 training 分池并行已是共识，rollout 长尾不再允许拖死整批。
  • 训推一致性作为一阶问题对待。GLM-5 在 Reasoning RL 里内置 IcePop 处理失配；ScaleRL 把 FP32 logits head 列为配方组件；FP16 论文与 VeXact 诊断把「先修数值一致性，再谈算法」变成行业纪律。
  • 混合奖励体系。纯 rule-based 或纯 RM 都已退场：GLM-5 General RL 同时用规则、ORM 与生成式 GRM；DeepSeek-V4 用 rubric-guided GRM 处理难验证任务；Gemini 3 用多步推理 RL + 不读 thought 的奖励。
  • SFT→（偏好）→RLVR 的阶段骨架在开放阵营被 Olmo 3 以 Dolci 套件「数据级」固化：每个阶段有独立的数据混合，可单独复现。

2.1 轴一：流水线顺序——「打架」问题的三种解法

多领域 RL 的根本矛盾：不同领域的奖励信号在同一个策略上互相干扰（数学要长推理、对话要短回复、agent 要工具调用格式），混在一锅里训，要么互相拖累，要么按出现频率「赢者通吃」。2026 年的三种结构化解法：

解法一：顺序流水线 + 跨阶段蒸馏（GLM-5）

严格按 Reasoning RL → Agentic RL → General RL 的顺序训练，每个阶段只优化一类目标。顺序的逻辑是依赖关系：agent 能力建立在推理能力之上，对齐/通用质量最后做（避免被前序大梯度阶段冲掉）。代价是灾难性遗忘——GLM-5 的对策是在最后用 on-policy distillation「快速恢复早期 SFT 与 RL 阶段习得的技能」（原文：swiftly recover the skills acquired in earlier SFT and RL stages）。本质上这是把 OPD 当作流水线的「黏合剂」。

解法二：并行 specialist + 蒸馏合并（DeepSeek-V4、Qwen3-Coder-Next）

每个领域独立训一个 specialist（各自 SFT + GRPO，奖励/数据完全领域内闭环），最后用 on-policy distillation 把 10+ 个 teacher 合并进单一学生。打架问题被转化为「教师合并」问题——而 OPD 的逆向 KL 目标对此天然友好：学生在自己的分布上采样，每个 teacher 只在自己擅长的 prompt 域上提供监督。Qwen3-Coder-Next 在编码域内部复刻了同样的结构（SWE / QA / Web 开发 / UX 四类专家分训后蒸回单模型）。

解法三：单一大 RL 阶段 + 环境课程（Qwen3.5）

不做显式阶段切分，而是「在几乎所有能构造的 RL 任务与环境上扩 RL」（原文：extensive scaling of virtually all RL tasks and environments），靠环境课程的难度与泛化性排序来管理干扰。这条路线对环境工程与基础设施要求最高（million-scale agent 环境、全解耦训推、3–5 倍端到端加速），实际上是把「打架」问题外包给了课程设计。

选型提醒 解法二最容易被中小团队低估的隐藏成本是蒸馏基础设施：DeepSeek-V4 为了全词表 logit 蒸馏专门做了 teacher 隐层缓存 + 按 teacher 索引排序样本 + 自定义 KL kernel 三件事（§4.3）。如果只能做 top-k logits 或黑盒采样蒸馏，合并质量会显著打折——此时解法一（顺序 + 末端 OPD 修复）反而更稳。

2.2 轴二：奖励来源——从「verifier vs RM」到奖励组合学

2025 年的叙事是「可验证奖励 vs 奖励模型」二选一；2026 年旗舰的实际做法是按任务可验证性分层组合：

3. GLM-5：顺序 RL 流水线与 slime 异步基础设施

GLM-5（智谱，arXiv:2602.15763，2026-02）是 2026 上半年信息披露最完整的旗舰技术报告，也是「顺序 RL 流水线」路线的代表作。从 GLM-4.5 的 355B/32B 激活扩到 744B 总参 / 40B 激活 / 256 专家，预训练 28.5T tokens，用 DSA（DeepSeek Sparse Attention）把长序列训练/推理成本降低 1.5–2 倍。副标题 "from Vibe Coding to Agentic Engineering" 概括了它的产品定位：从片段式代码生成转向完整工程任务。

3.1 三段顺序 RL 流水线

后训练主干是严格顺序的三段 RL（原文：starting with Reasoning RL, followed by Agentic RL, and finishing with General RL），每段的任务域、奖励与算法修正都不同：

阶段任务域奖励来源算法/工程要点
① Reasoning RL数学、科学、代码、tool-integrated reasoning 四域领域专属 judge model 给 outcome rewardGRPO 基座 + IcePop 处理训推失配（见 3.3）
② Agentic RL长程工具调用 / 软件工程任务任务级 verifier全异步解耦 infra；TITO 网关；双边 IS 截断；DP-aware 路由
③ General RL通用对话/写作/对齐规则 + ORM + 生成式 GRM 混合目标三维分解：基础正确性、情商（emotional intelligence）、任务专属质量

三段之后还有一个常被读者忽略但至关重要的第四步：On-Policy Cross-Stage Distillation。顺序流水线天然有「后面阶段冲掉前面阶段」的遗忘问题，GLM-5 的解法是在最后用 on-policy distillation 把早期 SFT 与 RL 阶段的技能「快速恢复」回来。注意这里 OPD 的用法与 DeepSeek-V4 不同：V4 用 OPD 做合并（多个 specialist → 一个学生），GLM-5 用 OPD 做修复（当前策略向早期 checkpoint 学回丢掉的能力）。同一个工具，在轴一上是两种完全不同的角色。

为什么是这个顺序？ Reasoning 在前：长推理能力是 agentic 行为的底座（工具调用之间的规划与反思全是推理）。Agentic 居中：它的奖励最稀疏、轨迹最长，需要前一阶段攒下的推理先验，也需要在通用对齐「磨平」模型之前完成。General 在后：对齐与风格目标的梯度相对温和，放最后不会破坏前两段的硬能力，反过来若放前面则会被硬任务的大梯度冲掉。这个顺序与「先难后易、先窄后宽」的直觉一致，也与 master 文档中 Qwen3 两段式（先推理后通用）的逻辑同源，GLM-5 把它推广到了三段。

3.2 Agentic RL 的工程细节：TITO 与双边 IS

Agentic RL 是三段中工程难度最高的：轨迹长（多轮工具调用）、GPU 空转严重（等环境返回）、训推两侧的 tokenization/采样细节极易失配。GLM-5 披露的对策值得逐条记录：

  • 全异步解耦：推理引擎与训练引擎分离，rollout 的长尾不阻塞训练（原文：massive-scale exploration of agent trajectories without synchronization bottlenecks）。
  • TITO（Token-in-Token-out）网关：环境交互全程以 token 为接口，保证训练时回放的动作序列与 rollout 时的动作逐 token 精确对应。这直接消灭了 多轮工具使用专题里反复强调的一类 bug：工具结果重新 tokenize / 模板重渲染导致的 token 漂移。
  • 直接双边重要性采样（direct double-sided importance sampling）：token 级 ratio 截断在 $[1-\varepsilon_\ell,\ 1+\varepsilon_h]$，上下界解耦设置。这是 DAPO clip-higher 思想（上界放宽保探索）与异步 off-policy 修正的合流——下界 $\varepsilon_\ell$ 控制 stale 样本的负梯度幅度，上界 $\varepsilon_h$ 控制探索激励。
  • DP-aware 路由：把同一对话的后续轮次路由到持有其 KV-cache 的数据并行分片，最大化 KV 复用——多轮 agent 任务的 prefill 成本是吞吐杀手，这一条是针对性优化。

3.3 IcePop：训推失配的「冷处理」

GLM-5 在 Reasoning RL 中内置了名为 IcePop 的机制来处理训练-推理失配（training-inference mismatch）。结合 §9.3 的脉络看，它的意义在于：继 2025-10 的 FP16 论文把失配确立为一阶问题之后，旗舰模型首次把失配处理写进了主配方而非附录。GLM-5 同时在 infra 层（TITO、FP8 rollout 的数值管理）与算法层（IcePop + 双边截断）双管齐下——这正是 坑点综述所主张的「先修后端正确性，再加算法修正」排查纪律在生产系统中的形态。

3.4 slime：异步 RL 基础设施

slime 是 GLM-5 配套开源的异步 RL 框架，设计目标可以概括为「把 rollout 当作一等公民」：

  • 可定制 rollout：rollout 逻辑（多轮、工具、sandbox）从训练循环中解耦出来，支持任意 agent 框架接入；
  • 长尾治理：FP8 推理 + MTP（multi-token prediction）投机解码压缩超长轨迹的生成时间——rollout 长尾（个别超长轨迹拖死整批）是 agentic RL 的头号吞吐瓶颈；
  • 心跳驱动的容错：数千 GPU 规模下 rollout worker 的故障是常态而非异常，心跳机制保证单点故障不传播。

4. DeepSeek-V4：用 On-Policy Distillation 替代混合 RL

DeepSeek-V4（技术报告 2026-04，副标题 Towards Highly Efficient Million-Token Context Intelligence；权重随后在 HuggingFace 开源）是 2026 上半年对后训练流水线最激进的一次改写。规模：V4-Pro 1.6T 总参 / 49B 激活（每 token 激活参数为当前公开模型最高档），V4-Flash 284B；预训练 Pro 33T / Flash 32T tokens（分拆数字据技术报告；V3 为 14.8T 的两倍以上），数据侧刻意加重长文档与 agentic 执行轨迹，面向百万 token 上下文。

4.1 动机：混合 RL 的「打架」死结

DeepSeek 自家路线的演化是理解 V4 的最佳注脚：

  • R1-Zero（2025-01）：跳过 SFT 直接对 base 做大规模 RL——证明了可行性，但语言混杂、可读性差，两代之后这条「纯 RL」支线被放弃；
  • V3.2 时代：多领域混合 RL 阶段——按 Labonne 对报告的梳理，这一设计导致了 reward hacking 与能力漂移（capability drift）：不同领域的奖励在同一策略上互相干扰，调权重是无止境的鼹鼠游戏；
  • V4（2026-04）：彻底放弃统一 RL 阶段，改为「specialist 分训 + OPD 合并」。

4.2 两阶段配方：specialist → OPD 合并

阶段一：四领域 specialist 各自闭环

数学、代码、agentic 任务、指令跟随四个领域各训一个 specialist：每个 specialist 从 SFT 起步，随后用领域内 prompt 与领域内奖励信号跑 GRPO。两个细节值得注意：

  • 奖励模型就是 actor 自己（报告原意：无独立奖励模型，specialist 在 RL 过程中自身习得评分能力）——这把「训一个独立 RM 再被 hack」的经典链路换成了自评估结构；
  • specialist 之间完全解耦：数据、奖励、超参、迭代节奏互不干扰，团队可以并行推进。这是组织工程学意义上的收益，与算法收益同样重要。

阶段二：10+ teacher 全词表 OPD 合并

合并阶段的结构是标准 OPD（参见 §9.2 的完整推导脉络）：学生自己采样 rollout，十多个 teacher 对学生轨迹逐 token 打分，逆向 KL 驱动更新：

$$ \mathcal{L}_{\text{OPD}}(\theta)\;=\;\mathbb{E}_{x\sim\mathcal{D},\; y\sim\pi_\theta(\cdot\mid x)}\Big[\;\sum_{t} D_{\mathrm{KL}}\big(\pi_\theta(\cdot\mid x, y_{<t})\;\big\|\;\pi_{\text{teacher}(x)}(\cdot\mid x, y_{<t})\big)\Big] $$

其中 $\text{teacher}(x)$ 按 prompt 的领域路由到对应 specialist。逆向 KL（学生分布在前）是 mode-seeking 的：学生只需在自己会走到的状态上贴近教师，不必覆盖教师的全部行为模式——这正是「多教师合并不打架」的数学原因：每个 teacher 只在自己领域的 prompt 上提供监督，学生在各领域分别向各 teacher 收敛，互相之间没有共享的冲突梯度。

4.3 全词表 logit 蒸馏的基础设施

V4 强调用全词表（full-vocabulary）logit 蒸馏：在每个位置对整个词表计算精确 KL，而非 top-k 近似或采样估计。报告的理由是梯度更稳、知识迁移更忠实；代价是必须传输或重建教师的完整 logit 向量。为此 V4 做了三件工程事：

  1. 缓存 teacher 隐层、即时重建 logits——存隐层而非 logits（隐层维度远小于词表维度），用教师的 LM head 在消费端即时重建完整分布；
  2. 按 teacher 索引排序训练样本——同一 micro-batch 内的样本来自同一 teacher，避免在一个 batch 里同时驻留多个教师的权重/缓存；
  3. 自定义 kernel 计算精确 KL——全词表 KL 在 1.6T 模型 + 百万上下文的设定下是真实算力项，必须 fuse。
复刻警告 「specialist + OPD 合并」的论文叙事很优雅，但它的可行性建立在上述蒸馏基建之上。如果你的栈只能做 top-k logits（教师 API 不开放全词表）或只能黑盒采样，请先读 §9.2 的 THUNLP 成败条件再决定：教师与学生 thinking pattern 不兼容、或教师并未提供学生分布外的真实新能力时，OPD 分数会涨但能力不迁移。黑盒条件下，GLM-5 式「末端 OPD 修复」比 V4 式「全量 OPD 合并」更现实。

4.4 Rubric-guided GRM：难验证任务的奖励层

V4 的另一个明确选择：用 rubric-guided Generative Reward Model 取代标量 RM 处理难验证任务。机制上，GRM 不输出一个标量，而是按显式 rubric（任务专属的评估准则清单）逐条生成评估再汇总打分。相对标量 RM 的三点优势：

  • rubric 是可读、可审计、可迭代的——奖励被 hack 时能定位到具体条目；
  • 评估过程本身是生成式推理，能复用主模型的推理能力（与 RM-R1 / DeepSeek-GRM 一脉的 2025 年 GenRM 研究合流）；
  • 大幅降低对大规模人类偏好标注的依赖——rubric 由少量专家写成，泛化由 GRM 承担。

4.5 V4 配方的含义

把 V4 放回四轴框架：它在轴一上给出了「打架问题」的最彻底解法（连「阶段顺序怎么排」的问题都消掉了——specialist 无所谓顺序）；在轴二上把奖励来源从「RM/verifier」整体上移到了「teacher 分布」（合并期的监督信号就是教师 logits，密度是每 token 全词表）；代价集中在轴三（蒸馏基建）。它与 ScaleRL 的结论（§9.1）拼起来是 2026 年最重要的预算公式：RL 的钱花在 specialist 上（窄域、可验证、天花板高），合并的钱花在 OPD 上（成本约为 RL 的 1/10 量级）。

5. Qwen3.5 与 Qwen3-Coder-Next：million-agent 环境与「数据工程红利」

5.1 Qwen3.5-397B-A17B：把 RL 扩到所有能构造的环境上

Qwen3.5（2026-02，旗舰 397B 总参 / 17B 激活，副标题 Towards Native Multimodal Agents）在轴一上选择了与 GLM-5、DeepSeek-V4 都不同的第三条路：不做显式的阶段切分或 specialist 分训，而是单一大 RL 阶段 + 极致的环境工程。官方博客的两句关键表述：

  • 「对几乎所有 RL 任务与环境做了广泛扩展」（extensive scaling of virtually all RL tasks and environments）；
  • 环境与任务的选取标准是提升难度与泛化性，而非优化特定榜单指标——支撑设施是 million-scale 的 agent scaffolds 与环境。

这条路线的实质是把「多任务打架」问题外包给课程设计：当任务分布足够宽、难度梯度足够平滑时，单策略可以在不分阶段的情况下吸收多领域能力。它对两件事的要求远超另两条路线：

  1. 环境供给：million-scale 环境意味着环境的构造、验证、难度标定本身是一条生产线（行业所称「环境工程」岗位的由来，参见 多轮工具使用专题中的环境章节）；
  2. 基础设施吞吐：Qwen3.5 披露了完全解耦的训推架构（动态负载均衡 + 细粒度故障恢复），FP8 端到端训练 + 投机解码，端到端 3–5 倍加速。没有这个量级的吞吐，million-scale 环境只是口号。

架构上 Qwen3.5 押注线性注意力：Gated DeltaNet（线性注意力变体，源自 Mamba2 的门控衰减 + delta 规则）与全注意力层约 3:1 交替，配稀疏 MoE。这与 MiniMax M2 团队「工业级场景 efficient attention 仍打不过 full attention」的反共识结论（§7.3）形成了 2026 年最鲜明的架构对赌——RL 长轨迹训练会放大两条路线在多跳推理上的差异，值得持续跟踪。

结果侧（官方披露口径）：Tau2-Bench 86.7（仅次于 Claude Opus 的 91.6）、BFCL-V4 72.9；多模态侧采用早期文本-视觉融合，原生支持多轮 agent 工作流（无框架级断点的多轮交互）。

三条路线的本质对比（轴一收口） GLM-5：用时间隔离干扰（顺序阶段）；DeepSeek-V4：用参数隔离干扰（specialist 分身）；Qwen3.5：不隔离，用课程与规模稀释干扰。三者的隐含前提依次是：阶段间正迁移 > 遗忘损失；蒸馏保真度足够高；环境分布足够宽。你的场景满足哪条前提，就抄哪家。

5.2 Qwen3-Coder-Next：3B 激活打 10–20 倍激活参数的对手

Qwen3-Coder-Next（arXiv:2603.00729，2026-03）是「数据与环境工程红利大于参数红利」的标志性案例：基于 Qwen3-Next-80B-A3B-Base（80B 总参 / 3B 激活，混合注意力 + MoE），在 agentic coding 上达到了「激活参数 10–20 倍于它的模型」的可比水平——SWE-Bench Verified 在 SWE-Agent scaffold 下超过 70%，SWE-Bench-Pro 与 10–20 倍激活参数的模型相当。

它的流水线是 DeepSeek-V4 结构在单一领域（编码）内部的「分形复刻」：

  1. Continued pretraining：在 Qwen3-Next base 上继续预训练（大规模可执行任务合成与环境交互数据）；
  2. SFT：高质量 agentic coding 数据；
  3. 领域内 specialist 分训：软件工程工作流、QA、Web 开发、UX 导向编码等多个专家模型分别强化；
  4. 蒸馏回收：把各专家能力蒸回单一统一模型。

训练重点放在长程行为质量上：long-horizon 推理、复杂工具使用、从执行失败中恢复（recovery from execution failures）——最后一项是 agentic coding 模型的实战分水岭，纯模仿学习极难覆盖（失败-恢复轨迹在人类数据中天然稀缺），必须靠环境内 RL 与任务合成补足。

为什么这个案例重要 2025 年的隐含共识是「agentic 能力 ≈ 大激活参数」。Coder-Next 用 3B 激活证明：当(a)可执行任务合成管线、(b)环境交互规模、(c)specialist→蒸馏结构三者到位时，激活参数可以缩一个数量级。对部署侧（本地 agent、端侧推理）这是路线级利好；对训练侧，它把竞争焦点进一步推向数据与环境工程——和 Qwen3.5 的 million-scale 环境叙事同一逻辑、不同尺度。

6. Kimi K2 Thinking → K2.5：INT4 QAT、长程工具链与 Agent Swarm

6.1 K2 Thinking：低精度与长程 agent 能力可以共存

Kimi K2 Thinking（Moonshot，2025-11，开源权重）的两个关键事实在发布时都属反直觉：

  • INT4 量化感知训练（QAT）：在微调阶段就用低精度算术做前向（权重保持全精度主副本），使最终的 INT4 量化几乎无损。结果：模型文件从 K2 Instruct 的约 1TB 压到 594GB，输出速度约翻倍；
  • 200–300 步连续工具调用无人工干预——1T 总参 / 32B 激活的 MoE 在 INT4 下保持长程 agent 行为的稳定性。

这两件事的共存回答了一个此前悬而未决的问题：长程 agentic 行为是否对数值精度敏感？答案是「推理精度可以压到 INT4，前提是训练阶段就感知量化」。注意它与 §9.3 训推一致性主题的微妙关系：FP16 论文反对的是「训练 BF16 / 推理另一套数值」的失配；QAT 的做法恰恰是把推理端的数值约束前置进训练，二者是同一原则（训推所见一致）的两种实现。

6.2 K2.5 与 Agent Swarm：把多智能体 RL 退化为单策略问题

Kimi K2.5（arXiv:2602.02276，2026-02，副标题 Visual Agentic Intelligence）的后训练贡献集中在两块：多模态联合训练的一组反直觉发现（6.3），以及 Agent Swarm——目前唯一在旗舰规模上跑通的多智能体 RL 训练方案。

核心设计：只训 orchestrator

多智能体联合 RL 的理论困难是组合爆炸的信用分配（哪个 sub-agent 的哪步动作导致了最终成败）。K2.5 的解法是釜底抽薪：

  • sub-agent 全部冻结，取自训练过程中的固定中间 checkpoint（fixed intermediate policy checkpoints），并可按负载动态调整 sub-agent 与 orchestrator 的推理实例配比；
  • 冻结后，sub-agent 的输出被视为环境观测而非可微决策点（原文：treating their outputs as environmental observations rather than differentiable decision points）——多智能体问题在数学上退化为单策略 RL：orchestrator 是唯一的 policy，sub-agents 是环境动力学的一部分；
  • orchestrator 学的是任务分解、并行派发、结果整合——agent swarm 由它动态创建（dynamically creates specialized frozen subagents），而非预定义角色。

奖励设计：critical steps 与防假并行

如果只奖励任务成功，orchestrator 会退化成串行调用（最稳）或形式上的「假并行」（fan-out 后不真正利用并行结果）。K2.5 的奖励有两层结构：

效率项：critical steps 而非总步数。对一条含主线与并行段的轨迹，

$$ S_{\text{critical}} \;=\; \sum_{\text{stages}} \Big( S_{\text{main}} + \max_{i} S_{\text{sub},i} \Big) $$

即主线步数加每个并行段中最慢子 agent 的步数——惩罚关键路径而非总工作量，直接激励「把可并行的事真正并行掉」。

  • 辅助塑形奖励，且退火到零：$r_{\text{parallel}}$ 防止策略坍缩回纯串行，$r_{\text{finish}}$ 防止假并行（派发了并行任务却不等待/不利用结果）。两者的系数在训练过程中退火至零——塑形只用于引导早期探索，最终策略只对「任务成功 + 关键路径成本」负责。这是 reward shaping 防 hacking 的教科书式做法（对照 坑点综述 §1.3：塑形项常引入新的 hacking 面，「乘法门或退火」优于永久加法分量）。

结果

实操启示：多智能体训练的「最小可行配方」 K2.5 给出的范式可以直接套用：① sub-agent 不要训，用你已有的最强（或专长各异的中间）checkpoint 冻结充当；② policy 只有 orchestrator 一个，信用分配退化为标准单策略 RL，verl/slime 等现有框架无需改动核心；③ 奖励 = 任务成功 + 关键路径成本 + 可退火的反坍缩塑形项；④ 监控并行度指标（派发数、等待利用率），防止两个方向的退化。代价要诚实：sub-agent 冻结意味着它们的错误模式无法被纠正，上限由「orchestrator 调度能力 × sub-agent 既有能力」决定。

6.3 多模态联合训练的三个反直觉发现

K2.5 报告中三个与 2025 年社区默认做法相反的实证结论，做多模态后训练的团队应当记录：

  1. Early fusion + 低视觉占比 > late fusion + 高占比：固定视觉+文本总 token 预算下，从头联合训练（early fusion）配 10%:90% 的视觉:文本比例，全面优于 50%:50%（原文：early fusion with a lower vision ratio yields better results given a fixed total vision-text token budget）；
  2. Zero-vision SFT：纯文本 SFT 数据即可激活视觉 agentic 能力（text-only SFT data alone activates the visual, agentic capabilities）——指令跟随与 agent 行为模式是跨模态共享的，不必为视觉单独造 SFT 数据；
  3. 视觉 RL 反哺纯文本基准：视觉 RL 改善了模型校准（calibration），在「貌似需要视觉 grounding」的文本查询上降低不确定性——MMLU-Pro +1.7%、GPQA +2.1%。

三条合起来的含义：跨模态正迁移在 2026 年已是可主动利用的调参维度，而非需要规避的干扰项。预算紧张时，优先做 early fusion 的联合预训练，后训练阶段大胆复用纯文本管线。

7. Gemini 3 与 Claude Opus 4.6：思考效率的训练化、parallel thinking 与 2026 春夏动态

闭源旗舰的披露粒度远低于开源技术报告，但 Gemini 3 Pro 的 Frontier Safety Framework Report（2025-11）与 Anthropic 的 Opus 4.6 发布材料（2026-02）仍各自给出了一条明确的方法论信号，且方向一致：思考预算从推理时超参变成被训练的对象。本章把两者与 MiniMax M2 的公开消融数据、native parallel thinking 训练路线（§7.4）以及 2026 春夏的旗舰后续动态（§7.5）放在一起，构成轴四「思考效率与思考结构」的完整图景。

7.1 Gemini 3：thought traces 上的 SFT 与 RL

Gemini 3 Pro 的《Frontier Safety Framework Report》（Google DeepMind，2025-11，"Addendum: Chain-of-Thought legibility" 一节）给出了关于后训练的三条可核实陈述。注意出处：这三条不在同期发布的 Model Card 里——按 Model Card PDF 去核证会落空，FSF 报告才是一手来源：

  1. SFT 与 RL 直接施加在 thought traces 上，且这些思考轨迹由 reasoning 模型生成、而非人写——人类标注员写不出高质量长推理轨迹，这一点自 s1/R1 蒸馏时代起已是共识，Gemini 3 把它制度化为数据生产方式；
  2. RL 阶段对 thoughts 施加 length penalty，目的是教模型「高效推理」（reason efficiently）——overthinking 不再只是推理时用 budget 截断来治，而是直接写进奖励；
  3. 除少数例外，其余奖励的计算不读 thoughts（with some minor exceptions, other rewards are computed without reference to the thoughts）。
第三条为什么重要：CoT 可监控性的保卫战 对 thought 内容施加优化压力，会教模型把真实推理藏到「面向奖励的表演性 CoT」之后（obfuscated reasoning），摧毁 CoT 作为安全监控通道的价值——这是 2025 年 CoT 监控文献的核心警告（monitorability tax）。Gemini 3 的设计是一种折中：对 thought 只罚长度（内容中立的压力），不对 thought 内容打分。length penalty 当然也是一种优化压力，但它不指向任何语义方向。事实上 FSF 报告正是在 CoT 可监控性的语境下（引 Korbak et al., 2025）披露这三条的，并给出量化口径：99.98% 的思考轨迹 appear comprehensible、99.85% appear informative of the model's output（Emmons et al. 的两个指标），与上一代 Gemini 相比无下降。各家在「思考效率 vs CoT 可监控性」上的取舍，预计是 2026 下半年安全与能力之争的具体战场。

产品形态上，Gemini 3 配 Deep Think 模式（更高思考预算档），与下述 Opus 4.6 的 effort 体系属同一产品语言：思考深度分档暴露给用户/开发者。

7.2 Claude Opus 4.6：adaptive thinking 与 effort 档

Opus 4.6（2026-02）把同一问题从接口端做了训练化：

  • Adaptive thinking：模型自行判断「何时值得深想」（原文：Claude can decide when deeper reasoning would be helpful）——校准「思考与否/思考多深」的决策本身是后训练习得的能力，而非外部路由器；
  • Effort 四档：low / medium / high（默认）/ max，替代旧的二元 extended-thinking 开关。默认 high 下模型「在有用时使用扩展思考」，官方明确建议：若发现模型 overthinking，把 effort 从 high 降到 medium；
  • Agent 配套：context compaction（beta，接近上下文阈值时自动总结旧上下文，使长任务不撞上限）与 Claude Code 的 agent teams（多 agent 并行处理读密集型子任务）。

对照轴四：Anthropic 选择「产品层组合单 agent」（agent teams）而非 K2.5 式「训练一个 orchestrator policy」。两条路线的张力值得记录——前者无需多智能体训练基建、即插即用，后者把协同行为内化进权重（BrowseComp +17.8 个百分点的来源）。目前没有公开证据表明哪条路线在同等基线上更优，但 K2.5 是唯一给出了训练方案的。

思考效率轴上最硬核的公开数字来自 MiniMax M2（230B 总参 / 约 10B 激活）官方博客的对照实验：interleaved thinking（思考穿插在多轮工具调用全程）模式下，跨轮保留 vs 丢弃历史 thinking 状态的差异：

SWE-bench Verified69.467.2−3.3%

规律清晰：任务的长程性越强（BrowseComp 的多跳搜索、Tau² 的多轮约束追踪），丢弃 thinking 历史的代价越惨重——模型跨轮携带的计划、假设、约束与中间结论是可靠性的骨架，丢弃后出现状态漂移、自我纠错能力衰减。

部署铁律（高频生产事故） 凡是训练时见过 interleaved thinking 的模型，推理侧绝不能在多轮对话/工具循环中丢弃历史 think 块。常见事故形态：agent 框架为省 token 把上一轮 thinking 从 context 里剪掉、网关中间件统一剥离 reasoning 字段、上下文压缩策略优先丢 think 块。任何一条都能让 BrowseComp 类任务静默掉 40%。排查入口：对比同一任务「带/不带历史 think 块」的成功率，再查你的框架与网关配置。与 Opus 4.6 的 context compaction 对照读：压缩可以做，但要由被训练过的总结机制完成，而不是粗暴剪切。

M2 团队同期另一个反共识工程结论：工业级场景下 efficient attention 仍打不过 full attention（复杂多跳推理有可测缺口）——与 Qwen3.5 押注 Gated DeltaNet 形成正面路线对立（见 §5.1）。

7.4 Native parallel thinking：思考的并行结构（解码层路线）

轴四所说的「思考的并行结构」有两个层级，不能混为一谈：K2.5 Agent Swarm 的并行发生在任务分解层（多个模型实例、orchestrator 调度，§6.2）；parallel thinking 路线的并行则发生在单模型解码层——同一个模型并行展开多条推理路径，再自行综合。后者的训练化代表是 ParaThinker（arXiv:2509.04475，2025-09）：

  • 动机：串行 CoT 的 Tunnel Vision。继续加长单条思考链的收益递减，根因不是模型能力不够，而是早期不完美的推理步骤会把模型锁死在次优路径上——test-time compute 沿「深度」这一个维度扩展撞上了结构性瓶颈；
  • 机制：native 的端到端并行。训练模型同时生成多条互异的推理路径，最后由模型自己把多路径融合为单一答案——并行性由此被确立为继深度之后 test-time scaling 的第二个维度，且是被训练进权重的能力而非外部 scaffold；
  • 数字：8 条并行路径使 1.5B 模型平均 +12.3%、7B 模型 +7.5%，延迟仅增加 7.1%（并行路径解码对 GPU 吞吐友好）——「小模型靠并行结构越级」是该路线的核心卖点。

2026-04 的对照研究（Understanding Performance Gap Between Parallel and Sequential Sampling in Large Reasoning Models，arXiv:2604.05868）补上了机理：在 Qwen3、DeepSeek-R1、Gemini 2.5 等多个模型家族与数学/编码任务上，并行采样一致优于串行 refinement；消融把差距主因定位在串行模式 conditioning on 先前答案导致的探索不足，而非聚合算子或上下文长度——与 Tunnel Vision 的诊断互相印证。

把两个层级放回轴四对照：产品上各家的 Deep Think / heavy 档（§7.1）是这条路线的推理时形态（并行采样 + 选择/综合），ParaThinker 则把并行写进权重。这组关系与多智能体轴上「K2.5 训练 orchestrator vs Claude 产品层 agent teams」（§7.2）同构：一条训练化、一条系统化，2026 年中两条都未分出胜负。对从业者的入手建议也对称：已有强模型时先做推理时并行（成本低、即插即用），自训模型且目标是小激活/端侧时再考虑 ParaThinker 式 native 训练。

2026-02 之后旗舰迭代未停，但后训练方法的披露粒度普遍回落到产品发布材料级别。截至本文更新（2026-06-12），值得记录的三组动态：

8. 开放数据配方：Olmo 3 Dolci、K2-V2、INTELLECT-3 与 Smol Playbook

旗舰对照如果只看闭源与半开源（开权重不开数据），从业者能「抄」的其实有限。2025-11 到 2025-12 的三连发（Olmo 3、K2-V2、INTELLECT-3）首次把数据混合、训练历史、RL 环境三者同时开放，加上 HuggingFace 的 Smol Training Playbook 把工程细节书面化，开放阵营第一次提供了可端到端复现的后训练参考实现。

8.1 Olmo 3 与 Dolci 套件：阶段化数据混合的「参考答案」

Olmo 3（Ai2，2025-11 发布，arXiv:2512.13961，7B 与 32B dense，65K 上下文）的核心概念是 model flow：开放的不是一个权重快照，而是完整生命周期——每个阶段、每个 checkpoint、每个数据点及其依赖关系。后训练侧的交付物是 Dolci 套件：

组件对应阶段内容定位
Dolci Think SFTSFT含长 thinking traces 的合成样本，覆盖数学、代码、指令跟随、通用对话
Dolci Think DPO偏好优化高质量对比（contrastive）数据
Dolci Think RLRLVR面向 RLVR 的高难度 prompt 集

三个要点：

  • 每个阶段有独立的数据混合——这把「SFT→DPO→RLVR 三阶段骨架」从论文叙述落实为可下载、可消融的工件。想验证「我的 DPO 数据质量是不是瓶颈」，Dolci 是现成的对照组；
  • 旗舰 Olmo 3-Think 32B 是迄今最强的「完全开放」（数据+权重+流程）thinking 模型——注意「完全开放」与「开权重」（如 GLM-5、K2.5）是两个级别的可复现性；
  • 对照 G34 的知识断档背景：在 2025-11 之前，开放阵营最完整的后训练数据参考还是 Tülu 3（2024-11）；Dolci 把参考答案升级到了 reasoning/RLVR 时代。

8.2 K2-V2：360 度开放的推理增强底座

K2-V2（LLM360，arXiv:2512.06201，2025-12）的口号是「360-open」：开放模型权重、完整训练历史（中间 checkpoint 序列）、全部训练数据与配比及 LLM360 工件链。配方上的关键词是主动注入：在（预）训练过程中前置注入领域知识、推理、长上下文与工具使用能力，目标是做一个「为后续推理适配而生」的优质底座。结果口径：同尺寸档最具竞争力，超过 Qwen2.5-72B、逼近 Qwen3-235B。

它回答的问题与 §9.4（mid-training 闭环）直接相关：推理能力的先验应该在底座阶段注入多少？K2-V2 的存在让这个问题第一次可以用全开放工件做受控实验——你可以从它的任意中间 checkpoint 分叉，测试自己的 mid-training/后训练假设。

8.3 INTELLECT-3：社区环境生态的可行性证明

INTELLECT-3（Prime Intellect，arXiv:2512.16144，2025-12）：106B 总参 / 12B 激活 MoE，基于 GLM-4.5-Air-Base 做 SFT + RL，同尺寸档数学/代码/科学/推理 SOTA，超过许多更大的前沿模型。两个结构性意义：

  • RL 数据引擎是社区化的 Environments Hub——对冲「环境即护城河」格局（大实验室重金自建/收购专有环境并封锁）的最有力实证；
  • 配套开源 prime-rl 异步 RL 框架：单节点到数千 GPU 无缝扩展，本次训练实际用到 512 张 H200。连同模型与全套 infra stack 开源。

对从业者：INTELLECT-3 + prime-rl + Environments Hub 是目前自建 agentic RL 管线时许可证与复现性最干净的全套起点；它也证明了「拿别家开源 base（GLM-4.5-Air）做自己的 RL」这条路在旗舰口径下成立。

8.4 Smol Training Playbook：把 know-how 书面化

HuggingFace 的 The Smol Training Playbook（HF Space，配合 SmolLM3-3B 实践）是把「训练小而强的模型」的全程 know-how 书面化的工程手册，后训练章覆盖 SFT / DPO / GRPO。两个值得单独记录的实操细节：

  • Completion-only / assistant-only loss：SFT 只对 completion（或 assistant 轮）计损失，prompt token 不进 loss。在 TRL 生态的标准实现里，assistant-only loss 要求 chat template 内嵌 {% generation %} / {% endgeneration %} 标记（SmolLM3-3B 的模板是参考实现）——多轮数据上漏配这一步，loss 会被大量 prompt/工具输出 token 稀释，是高频低级错误（对照 坑点综述数据相关章节）；
  • 混合推理（hybrid reasoning）模板纪律：SmolLM3 同一模型支持 think/no-think 双模式，playbook 强调双模式数据必须在模板层严格分流——同一个模板槽位里混入两种行为模式的数据，模型会在两种模式间出现行为串扰。
开放配方怎么用：三个典型场景 ① 做数据消融：用 Olmo 3 的固定流程换上你的数据 vs Dolci 基线，归因干净；② 做 mid-training 研究：从 K2-V2 的中间 checkpoint 分叉；③ 做 agentic RL：INTELLECT-3 配方（开源 base + prime-rl + Environments Hub 环境）起步，再按 §3/§6 的旗舰技巧增量加固。三条路径都不需要从零造数据或环境。

9. 横切方法论：RL Scaling、OPD 经济学、训推一致性与 mid-training 闭环

本章抽出四条贯穿所有旗舰配方的方法论主线，给出机制、公式与可引用的消融数字。它们是理解「为什么 2026 年的配方长这样」的底层原因。

9.1 ScaleRL：RL 算力的 sigmoid 定律

ScaleRL（Meta 等，arXiv:2510.13786，2025-10，总实验量超过 40 万 GPU-hours）做了 RL 领域第一个系统性 scaling 研究，核心发现是 RL 的算力-性能关系不是 power law，而是 sigmoid：

$$ R_C - R_0 \;=\; \frac{A - R_0}{1 + \left( C_{\text{mid}} / C \right)^{B}} $$

$R_C$：算力 $C$ 时的性能；$R_0$：起点；$A$：渐近天花板；$B$：算力效率指数（曲线陡峭度）；$C_{\text{mid}}$：达到一半总增益的算力点。

这个函数形式的实践含义比公式本身重要：

  • 可以用早期训练段外推天花板：拟合 $A$、$B$、$C_{\text{mid}}$ 三参数后外推——论文用一次 10 万 GPU-hours 的 8B 长跑验证了外推与实测紧密吻合。RL 调参从「跑完才知道」变成可预测工程；
  • 大多数配方选择只改效率（B、C_mid），不改天花板（A）：损失聚合方式、advantage 归一化、课程、off-policy 算法等主要调制 compute efficiency。例外是会改 $A$ 的少数几项：FP32 logits head（渐近 pass rate 0.52→0.61）、损失类型（CISPO/GSPO 的渐近性能大幅高于 DAPO）、更长的生成长度（14k→32k 降低早期效率但持续抬高渐近线）；
  • 「先快后矮」陷阱：小算力下领先的配方常常天花板更低——选配方必须按外推的 $A$ 而非短期斜率；
  • 同等 RL 算力给更大的 base：17B×16 MoE（Scout）只用 1/6 的 RL 算力即超过 8B dense 的渐近性能。预算决策上，RL 算力优先投给更大的底座，而不是给小模型堆步数。

ScaleRL 配方本身（八组件）可当作 2026 年的异步 RLVR 起点清单：PipelineRL 式流式异步（容忍 k=8 步 off-policyness）、强制截断式长度控制（注入 </think>）、FP32 logits head、CISPO 截断重要性采样损失、prompt 级损失聚合、batch 级 advantage 归一化、零方差 prompt 过滤、no-positive-resampling（通过率 ≥0.9 的 prompt 从课程中剔除）。与 DAPO 四件套的关系：DAPO 解决的是「单次训练跑崩」（entropy collapse、长度偏差），ScaleRL 解决的是「算力怎么花」——两者层级不同、可以叠加，但损失函数上 ScaleRL 的消融明确显示 CISPO/GSPO 渐近性能优于 DAPO。

与 RLVR 边界之争的关系 pass@k 阵营（arXiv:2504.13837，NeurIPS 2025 Oral）证明常规 RLVR 训练在大 k 下 pass@k 反而低于 base model（RL 压缩采样分布而非创造新能力）；ProRL（arXiv:2505.24864）证明延长 RL（KL 控制 + reference reset + 多样任务）能产生 base model 任意采样都到不了的新策略。sigmoid 定律给这场争论提供了量化语言：短程 RL 在爬 $C_{\text{mid}}$ 之前的段落，主要是 elicitation；天花板 $A$ 才是能力边界问题，它由任务分布、底座 headroom（见 §9.4）与配方中少数「改 A 的选择」决定。争论焦点已从「是否」转向「代价曲线」。

9.2 OPD 经济学：为什么人人都在用 on-policy distillation

OPD 的机制一句话：学生采样，教师逐 token 打分，逆向 KL 更新（公式见 §4.2）。Thinking Machines 博客（2025-10）给出的定位框架已成标准引用：

方法采样监督信号
SFT / 蒸馏（off-policy）off-policy（教师/数据分布）稠密（每 token）
RLon-policy稀疏（episode 级）

on-policy 解决「训练分布 = 部署分布」（学生在自己会犯错的状态上获得纠正），稠密信号解决样本效率。代价是需要一个教师——所以 OPD 不能替代「从无到有」的 RL（教师的能力上限就是学生的上限），但在所有「能力已存在于某个模型中」的场景（合并、压缩、修复、个性化）中全面优于 RL 与 SFT。

表中三行均出自 Qwen3 Technical Report（arXiv:2505.09388）Table 21：RL 与 OPD 从同一个 off-policy 蒸馏初始化（55.0%）出发对比，OPD 以约 1/10 的 GPU-hours 拿到比 RL 高 6.8 个百分点的分数。Thinking Machines 的自有复现是另一组实验，数字不可与上表混读：Qwen3-8B-Base 学生在 40 万条教师样本上全参 SFT 得 60%，再以 Qwen3-32B 为教师做约 150 步 OPD（7.7 万 prompt × 4 采样）升至 70%；按「达到教师水平的速度」计，OPD 比 RL 快 7–10 倍，叠加 batch/上下文因素总算力差距可达 50–100 倍。另一个被低估的用例是能力修复：在内部文档上微调使 IF-eval 从 85% 掉到 45%，一轮 OPD（向微调前的自己学）恢复到 83%，同时保住新知识（内部 QA 18%→41%）——这正是 GLM-5 cross-stage distillation 的用法（§3.1）。

成败条件（THUNLP，arXiv:2604.13016，2026-04）

OPD 不是免费午餐，THUNLP 的机理研究给出两个必要条件与一组可操作判据：

  1. 师生 thinking pattern 兼容：成功的 OPD 表现为师生在「共享 token 集」（集中 97%–99% 概率质量）上的渐进对齐；pattern 不兼容时学生在表面 token 上对齐而能力不迁移；
  2. 教师必须提供学生训练分布之外的真实新能力——否则蒸馏分数会涨、泛化不变。

失败时的两个恢复策略：off-policy 冷启动（先用教师轨迹做一段 SFT 把 pattern 拉近）、teacher-aligned prompt selection（只在教师真正占优的 prompt 域上做 OPD）。

OPD 在 2026 年配方中的三种角色（汇总） ① 合并器：DeepSeek-V4（10+ specialist → 学生）、Qwen3-Coder-Next（4 类编码专家 → 单模型）；② 修复器：GLM-5 cross-stage distillation（找回被后续阶段冲掉的技能）、Thinking Machines 的个性化后能力恢复；③ 压缩器：大教师 → 小学生的经典蒸馏，但用 on-policy 形式规避分布漂移。识别你的场景属于哪种角色，决定了教师选择与 prompt 域设计。

9.3 训推一致性：从「玄学崩溃」到一阶工程问题

这条线在 2025-10 到 2026-05 之间完成了从发现到工程化的闭环，三篇/处一手来源：

  1. FP16 论文（Sea AI Lab，arXiv:2510.26788）：RL 微调中训练引擎与推理引擎在相同权重下对同一序列给出不同 token 概率，根因是 BF16 的大舍入误差破坏两侧一致性；把训练与 rollout 都改回 FP16（动态范围小但尾数精度高）即可消除失配——改动只有几行代码。结论：优先改精度，而不是堆算法补丁；
  2. VeXact 诊断（arXiv:2605.14220，2026-05）：构造「零失配」对照设置，证明token 级的微小数值分歧本身就能独立导致训练崩溃——失配改变了有效优化问题（你以为在优化 on-policy 目标，实际在优化一个有偏目标），必须当一阶因素而非良性噪声对待；
  3. 生产系统的吸收：GLM-5 把失配处理（IcePop + TITO + 双边 IS 截断）写进主配方（§3.2–3.3）；ScaleRL 把 FP32 logits head 列为八组件之一且消融显示它直接抬高渐近天花板（0.52→0.61）——logits/softmax 处的精度是失配的最敏感点。
排查纪律（合并自三处来源 + 坑点综述） RL 训练不稳时按此顺序排查，不要跳步：① 对齐两侧 logprob：同权重、同序列，训练引擎与推理引擎的 per-token logprob 差应接近 0，监控序列级 log-ratio 漂移；② 检查精度配置：BF16 rollout 是首要嫌疑，试 FP16 或至少 FP32 logits head；③ 检查 token 级一致性：模板重渲染、工具输出重新 tokenize（GLM-5 用 TITO 根治的那类 bug）；④ 以上都干净之后，才轮到 IS 修正（token-level TIS / 双边截断 / 序列级）与算法超参。顺序反了的典型症状：加了 IS 修正后「看起来稳了」，但其实在带偏目标上收敛，分数天花板莫名偏低。

9.4 Mid-training 闭环：RL 增益的上游决定论

2025-12 到 2026-02 的两篇论文把「RL 效果由上游决定」从经验谈变成了受控结论：

  • Interplay 研究（Zhang/Neubig/Yue，arXiv:2512.07783）：受控实验框架下，RL 只有当预训练留有足够 headroom、且 RL 数据落在模型「能力边缘」（edge of competence）时才产生真实能力增益；固定算力下「预训练+mid-training」显著优于「纯堆 RL」；过程级奖励减少 reward hacking、提升推理保真度。这解释了为什么不同底座上同一 RL 配方结果天差地别——你看到的「RL 不 work」很多时候是底座没有留出可激发的余量；
  • ReMiT（arXiv:2602.03075，2026-02）：把单向流水线变成闭环——用 RL 调优后模型的推理先验动态加权 mid-training（退火期）的 token，优先注入对推理关键的 token，形成 pretrain → RL → mid-train 的迭代演化。数字：10 个预训练基准平均 +3%，且增益在整个后训练流程中保持 2% 以上。

与旗舰配方的连线：K2-V2 的「主动注入推理先验」（§8.2）、Qwen3-Coder-Next 的 continued pretraining 阶段（§5.2）、DeepSeek-V4 加重 agentic 执行轨迹的预训练数据（§4），都是同一判断的不同实现——2026 年后训练效果的上限，越来越多地在退火期就被决定了。读技术报告时注意各家对 "mid-training" 的操作化差异极大，不要假设术语同义。

预算分配的 2026 版答案（综合 §9.1 + §9.4） 固定总算力下的优先级：① 底座是否留有目标能力的 headroom（没有就先做 mid-training/continued pretraining，而不是硬上 RL）；② RL 算力给更大的底座而非更多步数；③ RL 数据按「能力边缘」选难度（全对/全错的 prompt 都是浪费，对应 ScaleRL 的零方差过滤与 no-positive-resampling）；④ 能力合并/修复/压缩一律用 OPD，不要用 RL 重训。

10.2 检查清单

A. RL 开跑前（一致性与数据）

  • □ 同权重同序列下，训练/推理两引擎 per-token logprob 已对齐，序列级 log-ratio 漂移有监控面板（§9.3）
  • □ 精度方案明确：FP16 全程，或 BF16 + FP32 logits head；不裸跑 BF16（§9.3）
  • □ 多轮/工具轨迹走 token 级接口，无模板重渲染、无工具输出重 tokenize（§3.2）
  • □ prompt 集做过难度标定：剔除零方差（全对/全错）prompt；通过率 ≥0.9 的从课程中移除（§9.1）
  • □ 训练前用 sigmoid 外推估过天花板与预算（早期段拟合 A/B/C_mid，§9.1）

B. 奖励设计（按 §2.2 四层逐层过）

  • □ 可验证任务：verifier 经过对抗审计（hardcode 测例、boxed 多答案、格式 hack——坑点综述 §1）
  • □ 难验证任务：rubric-guided GRM 优先于裸标量 RM；rubric 条目可读、可审计（§4.4）
  • □ 所有塑形奖励都有退火或门控计划，不留永久加法分量（K2.5 先例，§6.2）
  • □ 思考效率若进奖励，只用内容中立的压力（长度），不对 thought 内容打分（Gemini 3 的折中，§7.1）

C. 多阶段流水线

  • □ 阶段顺序有依赖论证（推理→agentic→通用的 GLM-5 逻辑，§3.1），不是历史惯性
  • □ 每个阶段结束时对前序所有阶段的核心评测回归（遗忘监控）
  • □ 末端预留 cross-stage OPD 修复预算（约为单阶段 RL 的 1/10 量级，§9.2）

D. 部署与推理侧

  • □ interleaved thinking 模型：多轮循环中历史 think 块全保留，框架/网关/压缩策略三处都查过（M2 数据：违者 BrowseComp −40.1%，§7.3）
  • □ 上下文压缩用训练过的总结机制（context compaction 思路），不粗暴剪切（§7.2）
  • □ 低精度部署计划若是 INT4 档，训练侧应有 QAT 而非纯事后量化（K2 Thinking 先例，§6.1）

结语

2025Q4–2026H1 的旗舰后训练配方告诉我们：后训练正在从「算法选型问题」变成「系统结构问题」。GRPO 还是那个 GRPO（DAPO 四件套依旧是单阶段基线），但决定最终模型质量的变量已经上移：流水线怎么排、奖励怎么分层、teacher 从哪来、异步与数值一致性怎么保证、思考预算交给谁。四轴框架下看，各家在轴三（infra 与一致性）上高度收敛，在轴一（流水线结构）上正在向「specialist + OPD 合并」收敛，而轴二（难验证任务的奖励）与轴四（多智能体、并行思考、思考效率）仍是 2026 下半年最值得跟踪的开放前沿；本文未展开的 experience / memory / continual learning 主线（已在 §10.4 标注为知识库的已知空白）则是下一块待补的拼图。

对从业者，本文的全部内容可以压缩成一张行动卡：用 OPD 的眼光重新审视你流水线里每一个「合并」与「修复」环节；用 sigmoid 天花板的眼光审视每一笔 RL 算力；用 token 级一致性的眼光审视你的 rollout 栈；用「奖励分层 + 退火塑形」的眼光审视你的难验证任务。这四句话分别对应 §9.2、§9.1、§9.3 与 §2.2——也是 2026 年与 2025 年后训练实践的全部分野。
