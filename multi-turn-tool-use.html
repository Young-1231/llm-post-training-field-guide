0.1 两条轴的本质

⏱️ Multi-turn（时间结构）
  • 关心轨迹被切成几个 step
  • 每个 step 之间状态 \(s_t \to s_{t+1}\) 是否变化
  • 状态转移可由用户、reflection、tool、environment 触发
  • 形式上是 POMDP 中 horizon 维度
🧰 Tool Use（动作空间）
  • 关心动作集 \(\mathcal{A}\) 里有哪些"特殊动作"
  • 特殊动作触发外部计算 / IO / 环境状态变化
  • 工具结果回灌 context 继续生成时，每次 tool call 产生新一轮
  • multi-turn 也不一定要 tool（用户 / reflection 也能触发）
关键关系
两条轴正交但有偏相关：在"工具结果回灌模型继续推理"的训练 setup 下，每次 tool call 都产生新一轮生成——这是🅓格成为主战场的原因。 但注意两个方向都有反例：单次 function calling 后不把结果回灌模型（结果直接交给外部系统消费）就是单轮工具使用（🅑格）； 而 multi-turn 也可以来自用户、reflection 或多 agent 协作而无外部 tool（🅒格）。

0.2 为什么🅓 格子里工作最多

从工程价值看：

  • 🅐 (单轨迹推理)：已经被 R1 范式饱和，剩余空间主要在算法稳定性（DAPO/GSPO/CISPO）
  • 🅑 (单轮 + 工具)：部署中真实存在（单发 function calling），但训练侧工作极少——工具结果一旦回灌 context，自然演化成 multi-turn
  • 🅒 (多轮无工具)：多轮对话 RLHF 已成熟；Reflexion 类 verbal RL 被 R1 的 long CoT 吸收
  • 🅓 (多轮 + 工具)：覆盖搜索 / 代码 / GUI / 数学工具 / 浏览器 / 文件系统几乎所有真实价值场景 → 这是 2025-2026 的主战场
SECTION 1

POMDP 形式化：从单步 MDP 到长程交互

要严格定义多轮 tool use agentic RL，必须从标准 RL 框架推开。本节给出形式化定义、与单轨迹 RL 的对照，并指出 LLM 场景下的特殊点。

1.1 从单轮 LLM-RL 说起

传统 RLHF / DPO / R1 把 LLM 看作"退化的单步 MDP"：

Degenerate Single-Step MDP
$$ \mathcal{M}_{\text{LLM-RL}} = (\mathcal{X},\, \mathcal{Y},\, \pi_\theta,\, R) $$ $$ \tau = (x, y),\quad y \sim \pi_\theta(\cdot \mid x),\quad R(\tau) \in \mathbb{R} $$

这里 "trajectory"等价于"一次完整 generate 的输出"。token 是动作但没有"状态转移"——下一个 token 之前的"状态"就是已经生成的前缀。本质上是 contextual bandit 而非 sequential decision。

1.2 多轮 Agentic RL 的 POMDP

当引入 tool 与环境后，框架升级为部分可观测马尔可夫决策过程（POMDP）：

Agentic POMDP
$$ \mathcal{M}_{\text{AgRL}} = (\mathcal{S},\,\mathcal{A},\,\mathcal{O},\,\mathcal{T},\,R,\,\gamma,\,\rho_0) $$
符号含义LLM 场景下的实例
\(\mathcal{S}\)真实状态空间世界真实状态（如 web 真实内容、文件系统当前快照、GUI 真实 DOM 树）
\(\mathcal{A}\)动作空间词表 token \(\cup\) 结构化工具调用
\(\mathcal{O}\)观测空间token 化的 context（包括工具返回）
\(\mathcal{T}(s'\mid s,a)\)状态转移（含随机性）搜索引擎不同时刻返回不同结果；代码执行环境的副作用
\(R(s,a)\) 或 \(R(\tau)\)奖励规则验证 / 任务完成 / 子目标
\(\gamma\)折扣因子LLM 通常 \(\gamma=1\)（无折扣）
\(\rho_0\)初始状态分布用户初始 query
\(H\)时间跨度10 – 1000+ turn（典型）

1.3 策略与目标函数

策略 \(\pi_\theta\) 基于 观测历史 而非真实状态选择动作（这就是 POMDP 的"部分可观测"）：

$$ a_t \sim \pi_\theta(\cdot \mid o_{\le t},\, a_{\lt t}) $$

训练目标是最大化期望累积回报：

Agentic RL Objective
$$ J(\theta) = \mathbb{E}_{\tau \sim \pi_\theta}\!\left[\sum_{t=0}^{H} \gamma^t\, R(s_t, a_t)\right] $$

1.4 "Turn"与"Token"两套时间尺度

LLM agentic RL 有一个特殊点：存在两个时间尺度：

🔤 Token 尺度
最细粒度，每个 token 是一个动作。GAE / PPO clip 通常在这个尺度计算。一条轨迹可能有 10K-1M token。
🔁 Turn 尺度
一个 turn = 模型一次连续输出（直到触发工具或结束）。轨迹的 turn 数典型 10-100。优势可在 turn 边界聚合。

这两个尺度的关系：

$$ \text{Trajectory } \tau = \underbrace{[o_0, a_0^{(1)}, a_0^{(2)}, \dots, a_0^{(k_0)}]}_{\text{Turn 0 (模型输出)}}, \underbrace{o_1}_{\text{Turn 0 (env 返回)}}, \underbrace{a_1^{(1)}, \dots, a_1^{(k_1)}}_{\text{Turn 1}}, \dots $$

其中 \(a_t^{(j)}\) 是 turn \(t\) 内的第 \(j\) 个 token，\(o_{t+1}\) 是环境对 turn \(t\) 末尾工具调用的返回观测。

1.5 与传统 RL 的关键差异

维度传统 RL (Atari, Mujoco)多轮 Agentic RL
动作离散性低维离散或连续词表 30K-200K + 结构化触发
状态可重置易重置（用 seed）真实环境难重置（搜索结果漂移、网页变化）
奖励频率每 step 都有极稀疏，常常只在 trajectory 末尾
初始策略随机已是强 LLM（SFT 之后）
环境模型已知或可学不可学，只能采样
计算瓶颈GPU 训练rollout 阶段（generate + tool call I/O）
为什么 LLM agentic RL 是个新东西
从 POMDP 视角看，LLM agentic RL 不是简单的"NLP + RL"——它是 大动作空间 × 长 horizon × 极稀疏奖励 × 不可重置环境 × 强基础策略 的组合。这五个特征中的每一个都让经典 RL 算法（PPO/SAC/TD3）原封不动用不了。
SECTION 2

Tool 的形式化定义

"Tool"不是一个 fuzzy 概念。在 agentic RL 框架下，它有严格的数学定义和实现协议。本节给出形式化定义、典型分类、以及实现层的关键 trick。

2.1 形式化定义

一个 tool 是动作空间 \(\mathcal{A}\) 中的一个三元组：

Tool Definition
$$ \text{Tool}_i = \langle \text{name}_i,\, \text{schema}_i,\, f_i \rangle $$ $$ f_i : \text{Args} \to \mathcal{O}, \qquad \text{Args} \sim \text{schema}_i $$

其中：

  • name：唯一标识符（search, python_exec, click_xy...）
  • schema：参数类型签名（JSON Schema / Pydantic / 自定义 DSL）
  • \(f_i\)：执行函数。接收参数，有外部副作用，返回观测 \(o \in \mathcal{O}\)

整个工具集合 \(\mathcal{T} = \{\text{Tool}_1, \dots, \text{Tool}_K\}\) 构成 agent 可用的"外部能力"。动作空间扩展为：

$$ \mathcal{A} = \mathcal{V} \cup \bigcup_{i=1}^{K} \{(\text{name}_i, \text{args}) : \text{args} \sim \text{schema}_i\} $$

其中 \(\mathcal{V}\) 是词表。注意：tool call 本身是用 token 序列编码的——LLM 只能输出 token，它通过"输出结构化 token 序列"来"调用 tool"。

2.2 实现协议：LLM 如何"触发"工具

LLM 输出 token，环境/runtime 需要识别哪些 token 序列代表 tool call。主流协议有四种：

2.5 边界情况：什么不算 Tool

"<think>"块算 Tool 吗？
不算。<think> 是 inline reasoning，没有外部副作用，没有 environment 状态变化。它是 LLM 自己产生的、自己读取的、纯文本——属于动作空间 \(\mathcal{V}\) 的普通 token，不属于工具集合 \(\mathcal{T}\)。
反例：早期 ReAct 论文把 Thought: 与 Action: 列为并列，那是出于 prompt 工程考虑，并非形式化的工具划分。
Self-reflection / Critique 算 Tool 吗？
看实现。如果只是模型自己输出 critique 文本继续生成，不算（属于 long CoT）。如果是把生成结果送进一个独立的"critic 模型"调用，算（成为 LLM-as-Judge tool）。 Verbal RL / TextGrad 中的 critique 通常算 tool（因为是独立的 LLM 调用）。
代码生成 vs 代码执行
纯生成代码不是 tool。生成 + 沙箱执行 + 读取 stdout 才是。这是 ReTool / CodeAct 与传统 CoT-with-code 的本质区别。
一句话定义
Tool ≡ 动作空间 \(\mathcal{A}\) 中通过结构化协议触发、产生外部副作用、并返回观测 \(o\) 写回 context 的子集。
"外部副作用 + 观测返回"是检验的两个必要条件。
SECTION 3

多轮 Rollout 机制

多轮 tool use 与 single-turn R1 在工程上最大的差异在 rollout 阶段—— 不是简单的 vLLM.generate(batch)，而是 generate / tool call 交错的异步状态机。

3.1 单轮 vs 多轮：rollout 流程对比

🅐 单轮 (R1-style)
def rollout_r1(prompts):
    # 一次批量生成 完事
    completions = vllm.generate(
        prompts,
        max_tokens=32768,
        temperature=0.6,
        n=8  # group size
    )
    rewards = [
      rule_reward(p, c)
      for p, c in zip(prompts, completions)
    ]
    return completions, rewards
🅓 多轮 + Tool (Search-R1-style)
def rollout_search_r1(prompts):
    contexts = [p for p in prompts]
    done = [False] * len(prompts)
    trajectories = [[] for _ in prompts]

    while not all(done):
        # 1) 生成到下一个 stop sequence
        chunks = vllm.generate(
            [c for c,d in zip(contexts,done) if not d],
            stop=['</search>', '</answer>'],
            max_tokens=2048
        )
        # 2) 解析：是否需要调 tool
        for i, chunk in enumerate(chunks):
            if chunk.endswith('</search>'):
                query = extract(chunk)
                result = search_api(query)      # ← env call
                obs = f'<information>{result}</information>'
                contexts[i] += chunk + obs
                trajectories[i].append({
                  'gen': chunk, 'obs': obs
                })
            elif chunk.endswith('</answer>'):
                contexts[i] += chunk
                done[i] = True
    return trajectories

3.3 关键工程挑战

① 异质 turn 数

不同 prompt 的 turn 数不同：有些 1 turn 就答完，有些要 30+ turn。结果：

  • 批量 generate 越来越稀疏：一开始 batch_size=512，但到了第 20 turn 可能只剩 50 条还在跑
  • GPU 利用率下降：vLLM 的 continuous batching 部分缓解，但仍有 tail latency 问题
  • 解法：动态填充新轨迹（pre-emptive scheduling），或者 truncated batching

② 工具 API 并发与限流

  • 真实搜索引擎有 QPS / 配额限制（如 Google CSE 免费档 100 次/天；微软已于 2025.08 退役 Bing Search APIs，存量接入被迫迁移到 Azure 'Grounding with Bing Search' 或第三方聚合 API）
  • 解法：本地 Wikipedia + BM25 / Dense retriever 替代（Search-R1 / R1-Searcher 都这么做）
  • 训练阶段用本地检索，部署再切换真实 API

③ Token Mask 精确性

需要逐 token 标记"是否来自环境"。Search-R1 的 mask 是：

$$ m_t = \mathbb{1}\!\left[t \notin \bigcup_i \text{tokens}\!\left(\langle\text{info}\rangle_i \dots \langle/\text{info}\rangle_i\right)\right] $$

实现层面：rollout 阶段对每个 token 记录 source ∈ {policy, env}，loss 计算时只对 policy token 反向。

④ 长 context 截断

多轮累积下 context 容易爆 16K / 32K 限制。两种策略：

  • 硬截断：超长直接抛弃（loss penalty + 早停）
  • 软压缩：用 retrieval+summary 把前面 turn 压缩成 memory（Memory-R1 / MemAgent 路线）

这只是推理期视角。2025H2 之后该问题被"训练化"——可学习折叠 / RL 自总结 / 可学习外挂管理器，见 §10 长程上下文管理的 RL 化。

⑤ 环境状态不可重置

  • 同一 query 不同时间搜索结果不同 → trajectory 不可严格复现
  • WebDancer / WebSailor 用本地快照（爬下来的 Wikipedia / 搜索结果离线）
  • SWE-Gym / DeepSWE 与 SWE-bench 评测体系用 Docker 镜像 每次 fresh start（SWE-RL 训练不跑交互环境，不需要这套机制）
  • 系统化的工程框架（可复位性 × 状态隔离 × 冷启动三维度、microVM 快照 / fork）见 §9 环境工程与规模化
为什么 vLLM 单独不够
vLLM 解决了"高吞吐生成"，但 multi-turn 还需要：
  • state 管理（哪些轨迹在哪个 turn）
  • tool dispatcher（并发调用 + 限流 + 重试）
  • 异步编排（generate 与 tool call 并发）
  • weight 同步（PPO 训练后 vLLM weight 要更新）
这就是 verl / OpenRLHF / RAGEN / VerlTool / prime-rl 这些框架解决的真正问题。
SECTION 4

损失函数 & Token Masking

多轮 tool use 与 R1 的损失函数形式相同，但隐含一个关键操作：环境返回的 token 必须从梯度中被排除。 本节给出完整公式、masking 的数学动机、以及不同算法（Search-R1 / DAPO / GSPO）下的具体形式。

4.1 朴素 GRPO 在多轮的应用

把 §1.3 的目标函数代入 GRPO 损失，朴素形式为：

Naive GRPO with Tool Use（未 mask）
$$ \mathcal{J}^{\text{naive}}(\theta) = \mathbb{E}_{q,\,\{\tau_i\}}\!\left[\frac{1}{G}\sum_{i=1}^{G}\frac{1}{|\tau_i|}\sum_{t=1}^{|\tau_i|}\min\!\bigl(r_{i,t}\hat{A}_{i},\,\text{clip}(r_{i,t},1\pm\varepsilon)\hat{A}_{i}\bigr) - \beta\,\mathbb{D}_{\text{KL}}\!\left[\pi_\theta\|\pi_{\text{ref}}\right]\right] $$

这个公式有一个致命问题：\(t\) 遍历整条 trajectory \(\tau_i\)，包括环境返回的 token。这会让模型学着"预测搜索引擎会返回什么"——但环境是外部不可控的，预测它毫无意义，且会让训练崩溃。

4.2 Token Masking 的数学动机

策略梯度的本质：

$$ \nabla_\theta J = \mathbb{E}\!\left[\sum_t \nabla_\theta \log \pi_\theta(a_t \mid s_t)\,\hat{A}_t\right] $$

这个梯度只对 policy 自己产生的动作 才有意义。环境返回的 token \(o_{t+1}\) 是从 \(\mathcal{T}(s'\mid s,a)\) 采样的，policy 对它的概率没有定义（policy 不生成观测，只生成动作）。

把它强行写进 \(\log \pi_\theta\) 等于让 policy 去拟合环境的分布——这是错的：

  • 环境分布与 policy 无关，但梯度会强行把它推向"高概率"
  • 引入 bias：得到的 advantage 估计不再无偏
  • 实践中导致 policy 输出长得越来越像"搜索结果格式"

4.3 带 Token Mask 的正确损失

引入 indicator \(m_{i,t} \in \{0, 1\}\)，标记 token \(t\) 是否来自 policy：

$$ m_{i,t} = \begin{cases}1 & \text{token 来自 } \pi_\theta\\ 0 & \text{token 来自环境（如 retrieved content）}\end{cases} $$
Search-R1 Loss (with token mask)
$$ \mathcal{J}^{\text{Search-R1}}(\theta) = \mathbb{E}_{q,\{\tau_i\}}\!\left[\frac{1}{\sum_{i,t} m_{i,t}}\sum_{i,t} m_{i,t}\cdot \min\!\bigl(r_{i,t}\hat{A}_{i,t},\,\text{clip}(\cdot)\hat{A}_{i,t}\bigr)\right] - \beta\,\mathbb{D}_{\text{KL}}^{\text{masked}} $$

关键差异：

  • 分子只对 \(m_{i,t}=1\) 的 token 求和
  • 分母也只数 policy token（保证梯度均值无偏）
  • KL 散度同样只对 policy token 计算

4.6 一个常见陷阱：KL 项的 mask

易错点
KL 散度 \(\mathbb{D}_{\text{KL}}[\pi_\theta\|\pi_{\text{ref}}]\) 也必须只在 policy token 上计算。理由：
  1. \(\pi_\theta(o_{t+1} \mid \dots)\) 中 \(o_{t+1}\) 是 env token，模型对它的概率无意义
  2. 把 env token 算进 KL 会让 reference policy 也"学着"预测环境，但 reference 是冻结的——结果是 KL 项数值远大于实际偏离
Search-R1 / ReTool 的实现都显式过滤了 KL 项。早期一些实验代码漏掉这一步导致训练崩溃。

4.7 GSPO 在多轮场景的优势

GSPO 把 importance ratio 提升到 sequence 级。以下是 GSPO 的多轮 mask 适配变体——原版 GSPO 只在 single-turn 上定义 sequence-level importance ratio，这里把 mask 整合进来。

$$ s_i(\theta) = \exp\!\left(\frac{1}{|y_i^{\text{policy}}|}\sum_{t : m_{i,t}=1}\log\frac{\pi_\theta(o_{i,t}|\cdot)}{\pi_{\theta_{\text{old}}}(o_{i,t}|\cdot)}\right) $$

注意分母只数 policy token \(|y_i^{\text{policy}}|\)，而不是整条 trajectory 长度。这让多轮场景下 IS 的方差大大降低——因为 trajectory 的 env 部分长度不稳定，把它们算进归一化会引入噪声。

SECTION 5

Credit Assignment：奖励信号如何穿越长轨迹

这是多轮 tool use agentic RL 最难、最未解决的问题——典型轨迹 50+ turn，奖励通常只在末尾， 如何把这个稀疏信号"反向传播"到中间的每个决策？本节先用 2026 年的综述谱系建立坐标系，再系统梳理 token-level / step-level / turn-level / outcome 四个尺度的优势计算， 深挖 SWEET-RL（privileged-info critic）与 GiGPO（两级组内优势）两条代表路线，最后给出"turn 级优势 vs flat-token + mask"范式之争的现状与算法选型依据。

5.1 问题的本质：稀疏奖励 + 长 horizon

$$ \tau = \underbrace{(s_0, a_0, r_0=0)}_{\text{turn 1}}, \underbrace{(s_1, a_1, r_1=0)}_{\text{turn 2}}, \dots, \underbrace{(s_H, a_H, r_H=R^*)}_{\text{终局奖励}} $$

典型设置：\(H \sim 50\)，只有 \(r_H \neq 0\)。问题是：第 5 turn 的某个搜索决策对最终成功有多大贡献？经典 RL 的答案是 value function / TD bootstrap，但在 LLM 多轮场景下不够。

前提：credit assignment 之前，先得有冷启动轨迹
本节默认 rollout 已经能产生合理的多轮工具调用序列——但 base 模型乃至纯指令 SFT 模型往往做不到（不会按格式发起 tool call、拿到返回也不接着推理），稀疏终局奖励此时几乎学不动。因此多轮 agentic RL 几乎都要先用合成轨迹做 SFT 冷启动，把策略抬到"能跑通环境"的水平再上 RL。多轮轨迹的合成与筛选管线（Kimi K2 从 ~3000 真实 MCP 工具扩到 2 万+合成工具、judge agent 按 rubric 过滤的 agentic rejection sampling）见配套文档 《SFT 与数据工程》§3.3 Agentic 冷启动：Kimi K2 的轨迹合成管线。

5.2 谱系坐标系：粒度 × 方法论

credit assignment 的文献在 2025-2026 爆发式增长，单看论文名很难建立坐标。综述 From Reasoning to Agentic: Credit Assignment in Reinforcement Learning for LLMs（arXiv:2604.09459, 2026.04）系统整理了 47 个方法， 给出目前最好的索引骨架（配套 Awesome-Credit-Assignment-in-LLM-RL 仓库持续更新）。它把每个方法放进两个正交维度：

  • 分配粒度（granularity）：token → segment → step / turn → multi-agent，决定"信用切多细"。episode 级（outcome-only）信用在该综述中不是独立类目，而是所有方法共同改进的 baseline——粗到"整条轨迹共享一个 advantage"；
  • 方法论（methodology）：Monte Carlo（组内比较 / 子树平均）、Temporal Difference（critic + bootstrap）、model-based（训练专门的 reward / advantage 模型）、博弈论 / 信息论分解，决定"信用怎么估"。

该综述还强调一个分水岭：reasoning RL（单轮 CoT，500–30K token，无环境转移）与 agentic RL（多轮交互，100+ turn、100K–1M token 上下文，环境随机转移） 是两个难度量级的 credit assignment 问题——前者只需在一条自回归序列内部切分信用，后者还要让信号穿越环境的随机性。本节出现的所有方法都可以放进这张坐标系：

① Outcome-only（R1 默认）

$$ \hat{A}_{i,t} = \hat{A}_i = \frac{R_i - \text{mean}(\{R_j\}_{j=1}^G)}{\text{std}(\{R_j\}_{j=1}^G)},\quad \forall t \in [0,H] $$

同一轨迹内所有 policy token 共享 \(\hat{A}_i\)。简单、稳健，但 horizon 大时信号被严重稀释——一条 50 turn 的失败轨迹，每个 turn 都被同等惩罚，模型学不到"是哪一步出错"。

② Turn-level（SWEET-RL / GTPO / Turn-PPO / Tree-GRPO 路线）

把 turn \(h\) 的 advantage 定义为：

$$ \hat{A}_h^{\text{turn}} = R(\tau) - V_\phi(s_h)\quad\text{或}\quad \hat{A}_h^{\text{turn}} = \sum_{h'=h}^{H}\gamma^{h'-h}\,\delta_{h'} $$

这一尺度在 2025-2026 形成了清晰的方法谱系：SWEET-RL（§5.4）训练带特权信息的 turn 级 critic； Turn-PPO（§5.6）把 PPO 的 advantage 估计搬到 turn 级 MDP 上； GTPO（Group Turn Policy Optimization, arXiv:2511.14846）做 turn 级 reward assignment + 以归一化折扣回报作 advantage（turn 内 token 共享），并用代码自监督信号做 reward 致密化。 注意它与另一个同名 GTPO（arXiv:2508.03772，Group-relative Trajectory-based PO——做 token 级梯度冲突保护与熵控制的 GRPO 稳定化，单轮场景）撞名，引用时务必看编号。本文所有 GTPO 均指前者。 Tree-GRPO（§5.7）则用树结构在 turn 边界天然构造 turn 级信号。

③ Step-level (PRM / PRIME 路线)

把"step"定义为推理段落（一个 thought block 或一段独立推理）：

$$ \hat{A}_{i,s} = R_{\text{PRM}}(s) - \text{baseline}(s) $$

需要训练独立的 PRM（Process Reward Model）。PRIME（arXiv:2502.01456）通过 implicit reward 巧妙绕开了显式 PRM：

$$ r_t^{\text{implicit}} = \beta\log\frac{\pi_\phi(y_t \mid y_{\lt t})}{\pi_{\text{ref}}(y_t \mid y_{\lt t})} $$

④ Multi-scale 组合（设计空间，尚无公认配方）

Multi-Scale Advantage
$$ \hat{A}_t^{\text{multi}} = \alpha_1\hat{A}_t^{\text{token}} + \alpha_2 \hat{A}_t^{\text{step}} + \alpha_3 \hat{A}_t^{\text{turn}} + \alpha_4 \hat{A}^{\text{outcome}} $$

这是一个统一的设计框架而非具体算法：四尺度全组合目前没有公开发表的标准配置，权重 \(\alpha_i\) 在现有工作中为 hand-tuned 或随训练阶段动态调整。已发表工作多为单尺度或两尺度组合——GTPO 的 turn + outcome、Tree-GRPO 的树内 + 树间，以及目前唯一给出公开默认权重的 GiGPO（episode + step，\(\hat A = \hat A^E + \omega\hat A^S\)，默认 \(\omega=1\) 不调参，§5.5）。直觉上越粗粒度的信号噪声越小、可给更大权重。

5.4 SWEET-RL：训练时信息不对称的 turn 级 critic

SWEET-RL（RL with Step-WisE Evaluation from Training-time information， Zhou et al., Meta AI & UC Berkeley, arXiv:2503.15478, 2025.03）是 turn 级 credit assignment 的代表作， 核心思想一句话：critic 不必和 policy 看同样的信息。"评估某一 turn 走得好不好"远比"走好这一 turn"容易——只要评估者能开卷。

机制：三步

第一步，信息不对称（privileged information）。critic 额外拿到 policy 永远看不到的训练时信息 \(c\)：在其配套基准 ColBench （后端编程 + 前端设计，与 LLM 模拟的人类协作者多轮交互）中，\(c\) 分别是参考代码与参考网页——即 ground-truth 解。对照标准答案判断"第 \(t\) 轮这步问得好不好"，比从轨迹本身反推容易得多。 这一思想与机器人 RL 中的 asymmetric actor-critic 同源：critic 只在训练时存在，部署时丢弃，所以"作弊"完全合法。

第二步，critic 直接参数化为 advantage 而非 value。不训练额外的 value head，而是复用 LLM 语言头、以平均 log 概率比（DPO 式 implicit reward）作为 turn 级 advantage：

$$ A_\theta(o_t, a_t, c) = \frac{1}{L}\sum_{l=1}^{L}\log\frac{\pi_\theta(a_t^l \mid a_t^{\lt l}, o_t, c)}{\pi_{\text{ref}}(a_t^l \mid a_t^{\lt l}, o_t, c)} $$

长度归一化 \(1/L\) 用于稳定训练。

第三步，用轨迹级 Bradley-Terry 偏好目标训练 critic。按最终 reward 高低把成对轨迹标成 \(\tau^+/\tau^-\)：

$$ \mathcal{L}(\theta) = -\log\sigma\Big(\textstyle\sum_t \beta\, A_\theta(o_t^+, a_t^+, c) \;-\; \sum_t \beta\, A_\theta(o_t^-, a_t^-, c)\Big) $$

监督信号只在轨迹级，但优化目标会自动把它"分摊"到每个 turn 的 \(A_\theta\) 上——credit assignment 是被 critic 学出来的，而非靠 TD bootstrap 递推出来的。

policy 端：逐 turn 的 DPO

拿到 turn 级 advantage 后，policy 优化不走 PPO：对每个 turn 从当前 policy 采样若干候选 action，用学好的 \(A_\theta\) 排序得到 chosen / rejected，逐 turn 做 DPO。 整条 pipeline 没有 token 级 importance ratio，自然绕开了多轮 IS 的方差放大问题（对照 §4.7）。

结果与适用判断
  • ColBench 上对既有 SOTA 多轮 RL / DPO 基线（multi-turn DPO、rejection FT、value-head critic、LLM-as-a-judge）绝对提升 6%（success / win rate）；让 Llama-3.1-8B 在协作内容创作上追平或超过 GPT-4o。
  • 何时值得用：凡是有参考解 / 隐藏测试 / 标准答案可作训练时特权信息的任务（SWE 的 hidden test、搜索 QA 的 gold answer），都可以让 critic"开卷"而 policy"闭卷"。
  • 代价：需要训练并维护一个专门的 critic（数据是成对轨迹），比 critic-free 的组内比较贵得多——这正是 GiGPO（§5.5）要消掉的成本。

5.5 GiGPO：critic-free 的两级组内优势

GiGPO（Group-in-Group Policy Optimization, Feng et al., arXiv:2505.10978, NeurIPS 2025）回答相反的问题： 一个 critic 都不训，能不能拿到 step 级信号？答案是把 GRPO 的"组内比较"用两次——一次在 episode 级，一次在 step 级。

episode 级（宏观）优势

与 GRPO 相同：同一初始状态 rollout \(N\) 条轨迹成组，

$$ \hat A^{E}(\tau_i) = \frac{R(\tau_i) - \text{mean}\big(\{R(\tau_j)\}_{j=1}^{N}\big)}{F_{\text{norm}}} $$

\(F_{\text{norm}}\) 取组内 std，或固定为 1——后者等价于无偏的 leave-one-out 估计，且避免过易 / 过难任务被 std 归一化过度加权的 difficulty bias（与 Dr.GRPO 的批评一致）。

step 级（微观）优势：anchor state grouping

关键观察：同组 \(N\) 条轨迹经常路过相同的环境状态（ALFWorld 的同一房间描述、WebShop 的同一页面）。事后把所有轨迹中环境状态相同的 step 归进一个 anchor 组，组内每个成员计算折扣 return-to-go，再做组内归一化：

$$ R_t^{(i)} = \sum_{k=t}^{T}\gamma^{\,k-t}\, r_k^{(i)}, \qquad \hat A^{S}\!\big(a_t^{(i)}\big) = \frac{R_t^{(i)} - \text{mean}\big(\{R_t^{(j)}\}\big)}{F_{\text{norm}}} $$

同一状态下不同 action 的下游回报之差，就是这个 action 的微观优势——相当于用 rollout 里免费出现的"自然对照实验"替代 critic 的反事实估计。

合成与伪代码

GiGPO 总优势
$$ \hat A\big(a_t^{(i)}\big) = \hat A^{E}(\tau_i) + \omega\,\hat A^{S}\big(a_t^{(i)}\big), \qquad \omega = 1 \text{（默认，不调参）},\;\; \gamma = 0.95 $$

5.6 范式之争：turn 级优势 vs flat-token + mask

§4 的主流范式把多轮轨迹拍平成一条 token 序列、mask 掉 env token、把单个轨迹级 advantage 广播给所有 policy token——本质上是把多轮当作"很长的单轮"。 2025 底-2026 初，一批工作开始系统性批评这种 flat-token + mask 范式与任务的 turn 结构存在根本错位：

  • 优化粒度 ≠ 决策粒度。agent 的自然决策单元是 turn（一次完整的 思考 → 工具调用），而 flat 范式在 token 级做 clipping / IS——同一个 turn 内部分 token 被 clip、部分保留，更新方向不对应任何一个完整决策。 AT²PO（Agentic Turn-based Policy Optimization via Tree Search, arXiv:2601.04767, 2026.01）把 turn 级学习目标（ATPO）作为可独立插拔的组件提出， 论证其相对 token 级 / sequence 级目标更稳，配合熵引导的树扩展做 turn 级探索与 turn-wise credit assignment，7 个 benchmark 平均最高 +1.84 pp。
  • token 级 MDP 在长 horizon 失真。Turn-PPO（arXiv:2512.17008, EACL 2026 Findings）在 WebShop / Sokoban 上的发现： GRPO 直接搬到多轮长程任务上有明显短板，PPO 更稳；进一步把 PPO 的 advantage 估计从 token 级 MDP 换到 turn 级 MDP（每个 turn 一个决策节点）还能再改善。 这与 rl-pitfalls-survey §13 的"importance sampling 必须做 turn-level"结论相互印证。
  • GTPO（§5.3 ②）同属这一脉：turn 级 reward assignment、turn 内 token 共享 advantage。

5.7 Tree-Based Credit Assignment

Tree-GRPO（arXiv:2509.21240）的洞察：与其独立 rollout \(G\) 条轨迹，不如共享前缀 + 分支：

Tree-GRPO 的核心公式：节点 \(v\) 的 advantage 用其子树平均回报估计：

$$ \hat{V}(v) = \frac{1}{|\text{leaves}(v)|}\sum_{\ell \in \text{leaves}(v)} R(\ell) $$ $$ \hat{A}(v) = \hat{V}(v) - \hat{V}(\text{parent}(v)) $$

好处：(1) 前缀共享省计算；(2) 每个内部节点都有一个有效 advantage；(3) 自然密集化稀疏 reward。

5.8 Reward Shaping vs Credit Assignment

两个概念不要混淆
  • Reward shaping：人为加入额外的中间奖励（如 format reward / length reward / sub-goal reward）。改变了 reward 函数本身。
  • Credit assignment：reward 函数不变（依然只有 outcome），改变的是把 reward 反向传播到中间动作的算法（TD / GAE / Tree advantage 等）。
实践中两者经常组合使用。SWE-RL 同时用了 format shaping（格式错给 −1）和 outcome-only credit——不过注意它是单轮 setup，credit assignment 退化为平凡情形（整条 completion 共享一个标量）。

6.1 Search-R1 (UIUC, 2025.03)

论文：arXiv:2503.09516。开创了"搜索作为一阶 RL 动作"的范式。

训练 setup

  • Base：Qwen2.5-3B / 7B（Base 与 Instruct；论文 v1 版另含 LLaMA3.2-3B）
  • 数据：训练集只用 NQ + HotpotQA 的合并训练集；NQ / TriviaQA / PopQA / HotpotQA / 2WikiMultiHopQA / Musique / Bamboogle 共 7 个 QA benchmark 是评测集
  • 检索器：训练时用本地 E5 + 2018 Wikipedia dump（top-3 passage，避开商用搜索 API 限流）
  • RL 算法：PPO 与 GRPO 都做了——论文以 PPO 为默认；GRPO 收敛更快但训练一段步数后可能失稳，PPO 更稳但更慢
  • 奖励：R = is_correct(answer)，纯规则匹配，无 PRM

Prompt 模板

Search-R1 Prompt
arXiv:2503.09516
# System prompt You are a helpful assistant. Answer the question by reasoning step by step. You can use <search>query</search> to retrieve information. Wrap your reasoning in <think>...</think> and final answer in <answer>...</answer>. # User {question} # Assistant (training target) <think> ... </think> <search> query 1 </search> <information> ... retrieved doc ... </information> # ← mask <think> ... </think> <search> query 2 </search> <information> ... </information> # ← mask <answer> final answer </answer>

关键创新

  • Token mask：<information> 内容不参与策略梯度
  • Reward 极简：只判答案对错，不评估搜索质量
  • Search 触发的涌现：模型自发学会"多次搜索 / 改进 query"

结果

在 7 个 QA benchmark 上较 RAG 基线平均相对提升：Qwen2.5-7B 41%、Qwen2.5-3B 20%（最新修订版数字；v1 摘要为 26% / 21%，外加 LLaMA3.2-3B 的 10%）。绝对值上，Qwen2.5-7B-Base 的 Search-R1 在 NQ 达 0.480 EM。

6.2 ReTool (2025.04)

论文：arXiv:2504.11536。把代码执行作为一阶工具，挑战 AIME 等数学竞赛。

关键设计

  • 工具：Python 代码沙箱（stdout 作为 observation）
  • 触发：```python ... ``` 代码块自动执行
  • Mask：执行结果 <output>...</output> 被 mask
  • 奖励：答案数值匹配（AIME 题目）

结果

ReTool-32B 在 AIME 2024 达 67.0%（超过 o1-preview，具体差距取决于评测设定与 o1-preview 版本），AIME 2025 49.3%，扩展设置 72.5%。

为什么 ReTool 重要
ReTool 证明了"纯文本 long CoT 不是数学推理的天花板"——把代码执行接入推理循环后，性能上限显著提升。这也是 SimpleTIR、Tool-Star 等后续工作的方法论起点。

6.3 SimpleTIR (ICLR 2026)

论文：arXiv:2509.02479。专门治多轮 Tool-Integrated Reasoning (TIR) 的训练崩溃。

问题诊断

朴素 GRPO 直接训多轮 TIR 会崩溃。SimpleTIR 找到根因：void turn——既无代码块也无最终答案的"无效输出"。这种 turn 会：

  • 累积错误（无信号收敛到啥都不输出）
  • 梯度爆炸（被惩罚 → 模型生成更多 void → 死循环）

修复

SimpleTIR Void-Turn Filtering
$$ m_{i,t}^{\text{SimpleTIR}} = m_{i,t}^{\text{Search-R1}} \cdot \mathbb{1}[\tau_i \text{ contains no void turn}] $$

即任何含 void turn 的整条轨迹直接从 loss 中剔除（不只是 mask 那个 turn）。简单粗暴但极有效。

结果

Qwen2.5-7B 在 AIME24 从 22.1 提升到 50.5。该工作还提供了多轮 TIR 训练稳定性的系统分析。

6.4 SWE-RL (Meta, NeurIPS 2025)——常被误读的对照案例

论文：arXiv:2502.18449。首个把 RL 推理训练规模化到真实软件工程数据的工作（GRPO）。

先纠正一个普遍误读
SWE-RL 不是多轮 agentic RL。它是 Agentless 式 single-turn 训练：给定 issue + 相关代码上下文，模型一次性推理并生成 SEARCH/REPLACE 格式的编辑，奖励直接比较生成 patch 与 oracle patch 的文本相似度——训练全程不调用任何交互工具、不跑测试、不接触可执行环境。 按本文 2×2 矩阵，它落在 🅐 行（单轮、无工具），而非 🅓 格。把它当成"多轮 + 工具"的代表是社区常见的张冠李戴。

训练 setup

  • 数据：PR / issue 完整生命周期（GitHub 开源仓库的 software evolution 数据）
  • 交互形式：无工具、无环境交互——issue + 代码上下文一次性输入，单轮生成 SEARCH/REPLACE 编辑
  • Reward：rule-based diff 相似度（格式错误给 −1）
$$ R(\hat{d}, d^*) = \begin{cases} -1, & \text{format invalid} \\[2pt] \text{seq\_ratio}(\hat{d}, d^*), & \text{otherwise} \end{cases} $$

其中 \(\hat{d}\) 是模型生成的 diff，\(d^*\) 是 ground-truth diff，\(\text{seq\_ratio}\) 是 Python difflib.SequenceMatcher 的相似度（0–1）。 这个奖励公式本身就说明了它的单轮本质：奖励只依赖最终 patch 文本，不需要执行反馈参与训练。评测时套用 Agentless Mini 脚手架（定位 + 修复的 pipeline）跑 SWE-bench Verified。

结果与意外发现

  • Llama3-SWE-RL-70B 在 SWE-bench Verified 上 41.0%（<100B 模型最高，评测用 Agentless Mini 脚手架）
  • OOD 涌现：训完 SWE 任务后，函数编码 / 库使用 / 代码推理 / 数学 / 通用语言理解 5 个领域全部提升（SFT 基线则在这些领域退化）
  • 说明 SWE 数据本身蕴含通用推理结构；它对本文的价值在 reward 设计——不依赖可执行环境的稠密相似度奖励，是 agentic 环境工程之外的另一条路

SECTION 7

训练稳定性：多轮场景特有问题

多轮 tool use RL 比 single-turn 推理 RL 稳定性差得多。本节梳理 6 类典型崩溃模式与对策。

7.1 Reasoning Collapse（推理崩溃）

表现
训练中期，模型推理能力突然下降——它学会了"少思考、多猜"以减少错误风险。在多轮场景下尤其严重，因为每次思考都可能产生新错误。

对策：DAPO Clip-Higher（\(\varepsilon_{\text{high}}=0.28\) 而非 0.2），保护低概率"探索性 token"；entropy bonus；temperature schedule。

7.2 Tool Hacking（工具滥用）

表现
模型学会用"看起来在调工具"的方式骗取部分奖励。典型：(a) 重复发同样的 search query；(b) 写空 code block；(c) 故意返回错误以触发"修正奖励"。

对策：

  • OTC 类成本惩罚：\(R = R_{\text{task}} - \lambda \cdot \#\text{tool\_calls}\)
  • 去重检查：相同 query 多次调用只奖励一次
  • format reward：调用必须语法正确才计分

7.3 Cascading Tool Failure（连锁失败）

表现
某个 tool call 失败（search 返回空 / code 报错）后，后续所有 turn 也跟着失败——模型不会从工具错误中恢复。

对策：

  • Recovery sample：在 SFT 阶段加入"工具失败后正确恢复"的轨迹
  • Agent-R 类 reflection：失败后强制插入 <reflection> 块
  • SimpleTIR void-turn 过滤

7.4 Length Explosion（长度爆炸）

训练后期 trajectory 越来越长但准确率不升。多轮场景下问题更严重——每个 turn 都可能产生冗余。

对策：DAPO Overlong Reward Shaping + 修掉 loss 归一化里的 length bias（Dr. GRPO 去掉 \(1/|o_i|\) 项、DAPO token-level loss）+ 硬截断。注意 SimPO 的长度归一是 offline 偏好优化目标里的设计，与 on-policy RL 的长度爆炸不是同一问题。

7.5 KL Explosion（KL 爆炸）

表现
Policy KL 散度持续上升不收敛，最终输出无意义文本。多轮场景下 KL 比 single-turn 高 5-10×（因为有更长的 trajectory 累积偏离）。

对策：

  • Per-turn KL 而非 per-token：在 turn 边界聚合 KL 更稳
  • Adaptive KL coefficient：\(\beta\) 随训练动态调整
  • EMA reference policy（TR-DPO 思路）

7.6 Entropy Collapse（熵塌缩）

Policy 过早收敛到 greedy，丧失探索。多轮场景下更易触发，因为每个 turn 的熵下降叠加。

理论（arXiv:2505.22617）：

$$ \Delta H \propto -\text{Cov}_t(\log\pi(a_t), A_t) $$

对策：DAPO Clip-Higher / Clip-Cov / KL-Cov / 主动 entropy bonus / Pass@k Training（保多样性）。

环境工程与规模化：环境即数据

§8 讲的是训练框架，本节讲一个 2025-2026 年间被推到台前的更上游问题：RL 环境本身的工程与供给。 当算法收敛到 GRPO 系变体、框架收敛到 verl / prime-rl 系之后，各家真正的差距越来越多地来自"谁有更多、更真实、更快的环境"。 Prime Intellect 发布 Environments Hub 时的判断已成行业共识："RL environments are the key bottleneck to the next wave of AI progress"。

9.1 三个工程维度：可复位性 × 状态隔离 × 冷启动

§3.3 把环境问题归为"API 限流"与"状态不可重置"两个现象，工程上更系统的组织方式是三个正交维度——它们共同决定了 RL 吞吐的上限：

维度定义不达标的后果代表机制
可复位性
Resettability		 同一任务可恢复到逐字节一致的初始状态 GRPO 组内比较混入环境噪声；失败轨迹无法复现 debug Docker fresh container（SWE-Gym / DeepSWE）；文件系统级快照 + fork（Cursor 内部平台 Anyrun 的 Firecracker microVM）；离线数据快照（WebSailor 系本地 Wikipedia）
状态隔离
State isolation		 并行 rollout 之间互不可见、互不污染 一条轨迹写脏共享状态 → 整批 reward 被污染且无法归因 每 rollout 独占 microVM / 容器；PROVE 的 session 级 MCP server 状态隔离（20 个有状态 server、343 个工具的 live 执行）
冷启动时间
Cold start		 从"调度一个新任务"到"agent 可执行第一个动作"的延迟 直接吃掉吞吐：启动 60s × 每 step 数万 rollout，环境时间反超 GPU 时间 镜像预热 + 快照恢复代替全新构建；microVM 亚秒级启动；mid-trajectory checkpoint fork 复用已执行前缀
为什么 GRPO 把可复位性从 nice-to-have 变成硬约束
组内优势 \(\hat A_i = \big(R_i - \operatorname{mean}_j R_j\big)/\operatorname{std}_j(R_j)\) 隐含假设：同组 \(G\) 条轨迹的 reward 差异完全来自策略采样。 若初始状态漂移（上一条轨迹残留的脏文件、搜索索引更新、共享数据库被写），优势估计就混入了环境方差—— 症状是"同 prompt 组内 reward 方差异常大"，且加大 \(G\) 无法消除（它不是策略方差）。 快照 / fork 机制还有第二重价值：在 turn 边界 fork 环境状态，是 Tree-GRPO / AT²PO 式分支 rollout（§5.7）的工程前提。

9.2 规模化案例：从 2 万到百万级并行环境

9.3 SWE 环境三条扩量路线

SWE-Gym（arXiv:2412.21139, ICML 2025）证明了"环境即数据"：2,438 个带可执行 runtime 与单测的真实 Python 任务， 在其上训练带来 SWE-bench Verified 最高 +19 个绝对点的 resolve rate。但人工策展止步于千级规模，自动化扩量分化出三条路线：

9.4 开源生态：Environments Hub 与 INTELLECT-3

闭源阵营把环境当资产锁起来，开源阵营的对冲方案是众包。 Prime Intellect 的 Environments Hub（2025.08）以 verifiers 库为标准接口——环境是可安装、带版本的 Python 模块， 训练与评测复用同一实现；到 INTELLECT-3 技术报告时点，Hub 上已有 500+ 社区环境（定理证明、自主科研、领域任务等）。

INTELLECT-3（arXiv:2512.16144）是这套生态的端到端验证：在 GLM-4.5-Air-Base 上做两阶段 SFT（通用推理 + agentic）→ 大规模 RL， 训练全程跑在 prime-rl 全异步 off-policy 栈上（continuous batching、in-flight 权重更新、IcePop 式 masked token-level IS 处理训推失配）， RL 环境取自 Hub 的六大类（math / code / science / logic / deep research / SWE）。 结果：106B-A12B 规模拿到 AIME 2025 88.0、LiveCodeBench v6 69.3、GPQA-Diamond 74.4——同规模 SOTA，且模型、框架、配方、环境全部开源。 它证明"众包环境 + 开源异步 RL 栈"足以支撑前沿水平的 agentic 后训练，是对"环境护城河化"最直接的反命题。

9.5 「环境即护城河」与从业者技能栈迁移

为什么说环境是 2026 年的真护城河
  • 数据可以爬、权重可以蒸馏，环境很难偷：一个带真实工具、真实状态转移、真实用户分布的环境是活的系统，不是静态 artifact
  • 训练即部署闭环放大差距：Cursor 把 RL 环境与生产环境统一，用真实用户流量持续产生任务分布——这种环境只有产品方拥有
  • 公共基准失信推高内部评测价值：SWE-bench Verified 的记忆污染质疑（arXiv:2512.10218）之后，各家更信自有环境上的内部评测，环境同时垄断了"训练数据"与"可信度量"两种稀缺品
  • 开源侧的回应即 §9.4：把环境变成社区公共品。两种力量的拉锯是 2026 年 agentic RL 生态的主线剧情

对从业者的直接含义：2023-2024 年 post-training 团队的核心技能是数据清洗与配比，2025-2026 年重心已迁移到环境工程—— 写 verifier、管 microVM 编排与快照、设计任务难度课程（Qwen3.5 的 difficulty & generalizability 哲学）、做训练环内 hack 检测。 "environment engineer"正在成为与 data engineer 并列的独立角色。

SECTION 10

长程上下文管理的 RL 化

§3.3 ④ 把长 context 当作 rollout 期的工程麻烦（截断 / 压缩），那是 2025 上半年的视角。 Kimi K2 Thinking 要在 256K 窗口内稳定执行 200–300 次连续工具调用——horizon 拉到这个量级后，"窗口够不够大"已不是正确的问题。 2025H2 起业界共识转为：上下文溢出是一级故障，context 管理必须当成训练问题——让模型学会管理自己的工作记忆，而不是靠推理期 trick 兜底。

10.1 谱系总览：四档方案

档位方案机制需要训练？代表
① 可学习折叠context 操作进动作空间branch / fold 等特殊动作，端到端优化"什么占据工作记忆"是（SFT 或 RL）Context-Folding + FoldGRPO；AgentFold
② RL 自总结总结作为 rollout 链路的一环多段生成由模型自产摘要串联，outcome reward 穿透整条链是（与策略联训）Cursor Composer
③ 外挂摘要 / 管理器不动 agent 权重专用摘要模型周期压缩，或外部可学习 context 管理器摘要器 / 管理器可单独训ReSum + ReSum-GRPO；AdaCoM
④ 硬截断超长即弃截断 + 失败惩罚 / 早停否早期 Search-R1 系（§3.3 ④）

①→④ 是当前的经验效果排序（Context-Folding 论文中折叠 agent 显著优于 summarization-based 基线），训练成本排序与之相同—— 效果与投入在这条谱系上基本是单调交换。

10.2 Context-Folding + FoldGRPO：把"折叠"放进动作空间

Context-Folding（arXiv:2510.11967）给 agent 增加两个特殊工具： branch(description, prompt) 为子任务开一条独立工作上下文的分支； return(message) 结束分支——把分支内全部中间步骤折叠掉，主线程只保留 outcome 摘要。 活跃上下文 \(|c_t|\) 不再随 turn 单调增长，而是"主线 + 已折叠子任务摘要"的紧凑形态。

让模型学会何时分支、何时折叠的是 FoldGRPO：在 GRPO 轨迹级优势之上叠加 token 级过程信号 \(Q_{i,t}\)：

过程信号触发条件取值设计意图
Unfolded token penalty活跃上下文超过工作上限的 50%主线程全部 token \(Q_{i,t}=-1\)逼迫 token 密集的操作进分支、及时折叠
Out-of-scope penaltyLLM judge（GPT-5-nano）判定分支行为偏离 branch 声明的子任务\(Q_{i,t}=-0.2\)防止分支变成"什么都干"的逃逸舱
Failure penalty工具调用失败该 turn 全部 token \(Q_{i,t}=-1\)常规工具卫生

结果（基座 Seed-OSS-36B-Instruct）：BrowseComp-Plus pass@1 62.0%、SWE-bench Verified 58.0%， 活跃上下文比长上下文 ReAct 基线小 10×，并显著优于外挂摘要式管理。 值得注意的视角转换：这本质上是 §5 的 credit assignment 机制在 context 维度的复用——用过程奖励直接 shaping"什么值得占据工作记忆"。

10.3 AgentFold：多尺度折叠，SFT 就能学会

AgentFold（arXiv:2510.24699, Tongyi）把折叠做成多尺度决策：每步可选"保留细节的轻量 granular condensation"， 也可选"把一整段多步子任务收拢成一条结论的 deep consolidation"——context 被当作"主动雕刻的认知工作空间"而非 append-only 日志（灵感来自人类的 retrospective consolidation）。 关键证据：AgentFold-30B-A3B 纯 SFT（无 RL、无 continual pretraining）即在 BrowseComp 拿 36.2%、BrowseComp-ZH 47.3%， 超过 DeepSeek-V3.1-671B-A37B 与 o4-mini。

实务结论：折叠行为先 SFT、后 RL
AgentFold 说明折叠的数据格式与行为模式本身就值一大截分——RL 不是第一步。 推荐路径：先用带折叠结构的轨迹 SFT 教会 branch / fold 行为，再视任务上 FoldGRPO 式过程奖励精修。 直接在不会折叠的模型上做折叠 RL，探索阶段几乎采不到有效折叠行为。

10.4 RL 自总结与可学习外挂管理器

  • RL 自总结（Cursor Composer）：一条 rollout 由多段生成组成、段间用模型自产摘要串联， 最终 outcome reward 作用于链上全部 token——"保住关键信息的摘要被强化，丢上下文的摘要被打压"， 无需为摘要单独设计 reward，credit assignment 由 outcome 隐式完成。工程上与 §9 的快照机制天然配合（摘要点即环境检查点）。 这是已有成熟 RL 管线时改动最小的可学习方案。
  • 外挂摘要 + 策略适应（ReSum, arXiv:2509.13313, Tongyi）：周期性调用专用摘要模型 ReSumTool-30B 压缩交互历史， 再用 ReSum-GRPO 让策略学会"从摘要状态继续推理"。对 ReAct 平均 +4.5%，ReSum-GRPO 后再 +8.2%； WebSailor-30B + ReSumTool-30B 在 BrowseComp-en 达 16.0% pass@1（同期 Claude-4-Sonnet 12.2%）。
  • 可学习外部管理器（AdaCoM, arXiv:2605.30785）：当 agent 权重不可动（闭源模型、多 agent 共享底座）时， 训练一个外部 LLM 控制器，用灵活的 context 修改动作 + 端到端 RL 管理冻结 agent 的上下文。 其"Fidelity-Reliability 权衡"值得记住：强 agent 受益于保高保真上下文，弱 agent 反而需要激进压缩才稳定； 学到的管理策略在能力相近的 agent 间迁移最好。

10.5 Interleaved thinking 的部署铁律：不可丢弃历史 think 块

Interleaved thinking 指"在显式推理与工具调用间交替，并把推理跨步携带"（MiniMax 定义）。 对这类模型（MiniMax M2 系等），多轮 agent 对话中绝不能丢弃历史 <think> 块。MiniMax 官方实测（丢弃历史 reasoning vs 完整保留）：

机理就是本文反复出现的 train-test mismatch（§3.3、§4），只是这次错在部署侧：模型在 RL 中是带着完整 reasoning 历史 rollout 的， 推理期把 think 剥掉等于人为制造分布偏移——MiniMax 总结为"cumulative understanding breaks down, state drift increases, self-correction weakens, and planning degrades"。 这个坑的高发原因是 OpenAI Chat Completions API 默认不支持回传 reasoning content： 解法是改用带 reasoning_details 回传的接口，或 Anthropic 兼容协议（把含 thinking 块的完整输出 append 进 messages）。 注意规则的准确表述不是"永远保留 think"，而是部署时的 think 处理必须与训练时一致——Qwen3 系默认 chat template 在多轮中剥离历史 think，照模板部署同样正确。

10.6 未决争议：折叠 / 压缩 vs 检索原始信息

反方证据：Goal-Directed Search Outperforms Goal-Agnostic Memory Compression in Long-Context Memory Tasks（arXiv:2511.21726） 训练 RL agent（SUMER）直接在原始未压缩信息上做目标导向检索，在 LoCoMo 记忆基准上比此前方法提升 43%。 论点：压缩天然有损，任务无关的预压缩无法适配多样的未来查询——"留住原始数据、按需检索"优于"提前压缩"。

与折叠路线的关系并非全面对立：folding 是 goal-directed 的（模型自己决定折叠什么、何时折叠），这波批评的火力主要打 ③ 档的任务无关外挂摘要。 但它点出了折叠的真实软肋——折掉的信息不可恢复。"fold + 完整历史落盘 + 给模型检索工具兜底"的组合 2026 年年中尚无定论性研究，是值得做的开放方向。

何时用什么：决策清单
  • horizon ≤ 数十 turn 且 context 预算充足 → 什么都不用，完整历史最稳（§10.5 铁律照常适用）
  • 自训模型、horizon 数百 turn（deep research / SWE）→ ①：先 SFT 教折叠行为，再 FoldGRPO 过程奖励精修；已有成熟 RL 管线时 ② 自总结改动最小
  • agent 权重不可动 → ③：ReSum 式外挂摘要（再用 ReSum-GRPO 让策略适应），或 AdaCoM 式可学习外部管理器
  • 跨 episode 记忆型任务（长期对话问答）→ 优先保留原始历史 + 检索（arXiv:2511.21726），慎用任务无关压缩
  • 任何情况：interleaved thinking 模型按训练时的 think 保留策略部署；④ 硬截断只作为兜底保险而非方案

SECTION 11

开放问题与未来方向

多轮 tool use 是 2026 年 agentic RL 最活跃也最未解决的子领域。这里列出 10 个值得追踪的问题。

① 长 horizon credit assignment
100+ turn 下 outcome-only 信号被严重稀释。SWEET-RL / GiGPO / Tree-GRPO 等 turn / step 级方法提供了初步答案（§5），但理论保证还很弱；47 方法谱系见综述 arXiv:2604.09459。
② 不可重置环境
真实搜索、真实网页、真实 git 仓库无法 reset → trajectory 不可复现。如何在 RL 训练框架中刻画这种非平稳性？（快照 / fork 的工程缓解见 §9.1，但"活"环境的非平稳性本身仍无理论刻画）
③ Pass@k preservation
Limit of RLVR 指出 RL 在 pass@1 提升的同时损失 pass@k。多轮场景下这个问题更严重，因为每个 turn 都在缩窄探索空间。
④ Tool failure recovery
模型很少能从工具错误中优雅恢复。Agent-R 的 MCTS 引导是一条路，但成本太高。能否设计廉价的 recovery 奖励？
⑤ 异步训练-推理一致性
vLLM 推理与 Megatron 训练使用不同 kernel → 同一权重下 logprob 微小差异，重要性采样的方差被放大。CISPO 的 stop-grad 部分缓解。
⑥ 工具组合的组合爆炸
单工具 RL 已较成熟，但 5+ 个工具同时可用时，模型如何选择？Tool-Star、ToolRL 是起点，还远未解决。
⑦ Memory as RL
把 memory 操作当成 tool 来 RL 训练（Memory-R1 / MemAgent）是个新方向。但 memory 内容是未来策略的隐式状态，credit assignment 跨越 episode 极难。episode 内的同谱系问题（可学习折叠 / 自总结）见 §10。
⑧ Self-play with Tools
Multi-Agent Evolve（proposer/solver/judge）+ tool use 是否能 zero-data 自演化？Agent0 的初步尝试但只到 single-tool。
⑨ Inference-aware training
训练时已知部署时的 tool 集合 / 调用预算 → 学习自适应工具调用策略。L1 / BudgetThinker 在单轮上做了，多轮尚未。
⑩ Reward 设计的科学化
当前 reward shaping 多靠经验：format reward 系数多少、length penalty 何时启动、tool cost 怎么权衡——缺乏系统的理论框架。

11.1 给研究者的"切题"建议

2026 年最值得做的方向（个人观点）
  1. Turn-level credit assignment 的理论化：把 Tree-GRPO / GiGPO / Turn-PPO 用 PAC-Bayes 或 variance-reduction 框架统一（arXiv:2604.09459 的两维谱系是起点）
  2. 多工具协作的 RL 范式：Tool-Star 是起点，但 5+ 工具时的探索-利用平衡是真正难题
  3. Memory-as-RL 与 long-horizon agent 的结合：MemAgent 8K → 3.5M context 已经证明可行，但是 RL 上的优化还很初步
  4. Pass@k preserving 的多轮版本：把 Pass@k Training 推广到多轮，保持工具调用策略的多样性
