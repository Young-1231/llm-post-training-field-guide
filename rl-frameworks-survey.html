一、引言：RL 后训练范式的崛起

2022 年 ChatGPT 横空出世，将 RLHF（Reinforcement Learning from Human Feedback）从一个学术概念推向了工业核心。2024 至 2026 年间，以 DeepSeek-R1、OpenAI o1/o3、Doubao-1.5-Pro、Qwen3 为代表的一批"推理模型（Reasoning Models）"进一步证明：大规模 RL 后训练（特别是 RLVR，即基于可验证奖励的强化学习）已经成为继预训练（Pre-training）和监督微调（SFT）之后的第三条扩展曲线（the third scaling curve）。

这股浪潮的直接技术后果，是 RL 训练框架的爆发式繁荣。从 HuggingFace TRL 的"开箱即用"，到字节 veRL 的"工业级 671B 扩展"，再到蚂蚁 AReaL 的"全异步范式"，整个生态在短短两年内涌现出十余个生产级框架。它们在算法支持、分布式架构、推理-训练协同、异步并发、Agent / 多轮交互等维度上呈现出高度多元化的设计取向。

本综述目标：系统盘点 2026 年主流 RL 训练框架，覆盖其架构原理、算法支持、性能特征、社区生态与典型使用者；并结合实际工程经验给出从 7B 到 671B、从单卡研究到千卡集群、从单轮 RLHF 到多轮 Agent 的分层选型建议。

二、TRL（HuggingFace）—— RLHF 的事实教学标准

⚠️ 注意：下文及对比表中的 GitHub Star 数（如 18.4k★、21.2k★、9.5k★ 等）均为 2026.05 时点近似值，会随时间持续变化，请勿视为精确数字。

TRL: Transformer Reinforcement Learning

★ 18.4k v1.4.0 (2026.5) HuggingFace Apache-2.0 SFT / DPO / GRPO / RewardModel

起源与定位

TRL 项目最早源于 Leandro von Werra 的 trl（"Transformer Reinforcement Learning"），起初是一个小型的 PPO 训练包，专门用于在 HuggingFace Transformers 之上做 GPT-2 级别的 RLHF。2023 年被 HuggingFace 正式纳入官方组织后，其定位逐步演变为"Hugging Face 生态下统一的 Post-Training 入口"。截至 2026 年 5 月，TRL 已经成为整个开源社区引用最多、文档最完善、学习曲线最低的 RL 后训练库。

架构

TRL 在架构上是一个深度依赖 Transformers + Accelerate + PEFT 的包装层。它没有自己独立的分布式调度器，而是复用 Accelerate 与 DeepSpeed / FSDP 作为多卡 backbone。其训练 API 围绕 XXXTrainer 模式组织：

  • SFTTrainer —— 监督微调（含 PEFT/LoRA/QLoRA 支持）
  • DPOTrainer —— Direct Preference Optimization
  • GRPOTrainer —— DeepSeek-R1 同款 GRPO（v0.14.0 于 2025 年 1 月引入）
  • RewardTrainer —— Reward Model 训练
  • PPOTrainer —— 经典 PPO（在 v0.10 之后大幅重写）
  • KTOTrainer / ORPOTrainer / CPOTrainer / BCOTrainer / NashMDTrainer —— 各类偏好优化变体
  • OnlineDPOTrainer / XPOTrainer —— 在线偏好优化

关键特性（2026 最新）

Unsloth 内核集成提供 2-3x 训练加速（特别是 LoRA / GRPO 场景）。
vLLM 在线生成v0.14（2025.1）引入 GRPOTrainer 时即同步支持 vLLM 快速生成，后续版本扩展出 server / colocate 两种集成模式。
多模态 SFT/DPO支持视觉-语言模型如 Qwen-VL、Llama-Vision。
CLI 工具trl sft、trl dpo 无需写代码即可微调。

适用场景

  • 研究与教学：算法迭代快，文档丰富，最适合用于复现 paper 和算法 ablation。
  • 中小模型（≤ 13B）后训练：DPO / GRPO + LoRA 是最常见组合。
  • 生产 SFT / DPO 流水线：与 HuggingFace Hub、datasets 无缝衔接。

局限

  • 对于 ≥ 70B 的全参 RL，TRL 的 PPO/GRPO 实现并不是最优选择，吞吐量明显逊于 OpenRLHF / veRL。
  • Agent 多轮 RL 支持薄弱，缺乏内置 trajectory-level 工具。
  • Ray-based 异步生成支持有限（强依赖 Accelerate）。

三、OpenRLHF —— Ray + vLLM 工业级 RLHF 范本

OpenRLHF

★ 9.5k v0.10.3 (2026.5) OpenRLHF 社区 PPO / GRPO / DAPO / REINFORCE++ / RLOO / DPO Ray + vLLM + DeepSpeed

起源

OpenRLHF 由 Jian Hu 等人于 2023 年底发起，是最早将 Ray 分布式调度 + vLLM 加速 rollout 这一现代范式落地的开源 RLHF 框架。2024 年发表在 arXiv 上的 "OpenRLHF: An Easy-to-use, Scalable and High-performance RLHF Framework"（2405.11143）使其在工业界迅速流行，已被多个会议 demo track 引用（具体 venue 待核实）。

架构核心

OpenRLHF 采用 多角色（multi-role）Ray 调度：

  • Actor（策略模型，由 DeepSpeed ZeRO-3 训练）
  • Critic（价值模型，可选，DPO/GRPO 时关闭）
  • Reward（奖励模型 / 规则奖励 / Agent 评分）
  • Reference（参考模型，用于 KL）
  • vLLM Engines（多个 vLLM 实例，承担约 80% 训练时长的 rollout）

每个组件以独立的 Ray Actor 启动，OpenRLHF 通过 RPC + NCCL Group 在它们之间同步权重和数据。其 Hybrid Engine 调度允许 Actor 与 vLLM 共享同一组 GPU，最大化利用率。

关键特性

Async RL开启 --train.async_enable（v0.10 起的嵌套参数风格；v0.9 及更早为 --async_train）后生成与训练真正并行，off-policy 程度由 --train.async_queue_size 控制（默认 1，越大越 off-policy）。
DAPO 动态采样对每个 prompt 多次采样并按 reward 过滤。
TIS（Truncated Importance Sampling，截断重要性采样）修正 vLLM rollout 与训练引擎间的概率失配（off-policy 校正）。
VLM 支持Qwen-VL 等图文输入的端到端 RL。
多轮 Agentreset/step 接口，token 级 trajectory，与所有 RL 算法兼容。
70B+ 全参官方口径：16 张 A100 80G 即可做 70B+ 全参 RLHF 训练。

性能

根据 OpenRLHF 论文自身评测（arXiv:2405.11143，单方口径），其 PPO 训练吞吐相对深度调优的 DeepSpeed-Chat 在 7B-70B 区间约有 1.8-2.5x 优势（论文未与 TRL 直接对比）；而 HybridFlow（veRL）论文在另一组设置下的结论则是 veRL 快于早期版本的 OpenRLHF——两边均为各自口径，请以自身负载实测为准。代码量上（~8.5k 行）远小于 veRL（~32k 行）与 TRL（~19k 行），保持了较好的可读性与黑客性。

使用者

官方 adopters 名单包括 Google、字节跳动、腾讯、阿里巴巴、百度、中国电信、vivo、Allen AI（Ai2）、NexusFlow、Berkeley Starling 团队等公司与机构；GitHub Issues 与 PR 活跃度位列开源 RLHF 框架前三。

四、veRL（字节 Volcano Engine RL / HybridFlow）—— 工业旗舰

verl: Volcano Engine Reinforcement Learning for LLMs

★ 21.2k v0.7.1 (2026.3) ByteDance Seed / 火山引擎 PPO / GRPO / GSPO / DAPO / DrGRPO / PRIME / ReMax / RLOO FSDP / Megatron / vLLM / SGLang

起源与论文

veRL 是字节跳动 Seed 团队 2024 年发布的 HybridFlow 论文的开源实现，并于 2025 年正式入选 EuroSys 2025。截至 2026 年 5 月，veRL 已经是 GitHub stars 最高（21.2k）的 RL 后训练框架，被业界视为大规模 RL 的"工业旗舰"。

HybridFlow 架构

veRL 的核心设计哲学是 解耦"控制流"与"计算引擎"：

  • Single-Controller + Multi-Worker：算法层用 single-process 控制器表达（如 GRPO 主循环），数据流定义只需几行 Python；计算引擎以 Ray Worker Group 形式部署。
  • 3D-HybridEngine：通过精心设计的权重重切分（actor model resharding）算法，使训练（FSDP/Megatron）与生成（vLLM/SGLang）之间可以在无内存冗余 + 极低通信开销下切换。
  • 多 backend 自由组合：FSDP + vLLM、Megatron + SGLang、FSDP2 + SGLang 等任意搭配。

性能

veRL 官方报告其相对 DeepSpeed-Chat / OpenRLHF v0.2.5 / NeMo-Aligner v0.2.0 在 7B-70B 区间获得 1.5-20x 的吞吐提升（A100 集群）。最近的 v0.7 相对早期 v0.3 又获得约 1.4x 加速。

R1 复现与生态

veRL 是 2025 年初 DeepSeek-R1 开源风潮中复现频率最高的框架。诸如 simpleRL-reason、TinyZero、Open-R1-veRL、EasyR1 等多个广受关注的 R1 复现项目都基于 veRL。据 veRL 官方 README，字节自家 Doubao-1.5-Pro 的 RL scaling 预览版（深度思考模型，而非主线版本）基于 veRL 训练，在 AIME 上达到 70.0 pass@1 的 OpenAI o1-level 表现（veRL 项目方口径）。

规模能力

  • 已验证可扩展到 DeepSeek-V3 / R1 级别的 671B MoE 模型（通过 EP + PP + TP + ZeRO 组合）。
  • 原生支持 multi-turn tool calling RL（v0.6+）。
  • 支持 SLURM 与 K8s 部署。

局限

  • 代码库较大（~32k 行 Python），学习曲线相对陡峭。
  • 对小规模研究者偏"重"，调试需要熟悉 Ray + Megatron。

五、AReaL（蚂蚁 / 清华 IIIS）—— 全异步 RL 新范式

AReaL: Async Reinforcement Learning System

★ 5.2k v1.0.4 (2026.5) 蚂蚁 Ant Group · Tsinghua IIIS（仓库 inclusionAI/AReaL） GRPO / PPO / DAPO / REINFORCE / RLOO vLLM / SGLang / Async

论文与定位

AReaL 对应论文 "AReaL: A Large-Scale Asynchronous Reinforcement Learning System for Language Reasoning"（arXiv:2505.24298），由清华 IIIS 与蚂蚁集团联合发布。它最核心的贡献是把 "完全异步 RL" 作为一等设计原则，从根本上重构了传统同步 RLHF 的 generation-wait-train 流水线。

核心设计

  • 完全解耦的 Rollout Worker 与 Training Worker：Rollout Worker 持续生成新样本，从不等待训练侧；Training Worker 一旦凑齐一个 batch 就更新模型。
  • Staleness-Enhanced PPO：为了在异步带来的"数据陈旧（off-policy）"下保持稳定，引入了基于陈旧度的 PPO 变体（修正 importance ratio + clip）。
  • 负载均衡器：动态调节 rollout 与 train 比例，将 staleness 控制在可接受范围。
  • v1.0 起更名为"RL Bridge for Agent Applications"，定位向 agentic RL 倾斜，支持搜索、客服、代码、数学等任务的多轮 RL。

性能

论文报告：在数学/代码推理基准上，AReaL 相对同步系统在同等 GPU 数下取得 up to 2.77x 训练加速，最终性能持平或更优。

典型应用

AReaL 的设计天然适合 长 CoT 推理 与 多轮 Agent：在这些场景下生成时间分布极度长尾，传统同步范式 GPU 利用率往往不到 30%；AReaL 异步设计使训练 GPU 长期保持高利用。蚂蚁 boba² 系列与多个 Agent 应用基于 AReaL 训练。

六、NeMo-RL / NeMo-Aligner（NVIDIA）—— Megatron 原生 RL

NeMo-Aligner（已归档）

★ 854 v0.7.0 (2025.3) NVIDIA Archived 2025.11

NeMo-Aligner 是 NVIDIA 早期的 RLHF 工具包，支持 SFT、DPO、PPO、SteerLM、SPIN、REINFORCE、RM。深度集成 Megatron-LM 的 3D 并行能力。2025 年 11 月已归档，官方推荐迁移到新一代 NeMo-RL。

NeMo-RL（继任者）

★ 1.6k v0.6.0 (2026.4) NVIDIA GRPO / GSPO / DAPO / SFT / DPO / RM

2025 下半年发布的下一代框架，提供：

  • 与 Megatron-Core 深度集成，支持 6D parallelism。
  • HuggingFace 模型直读 + Megatron 训练 + SGLang 推理无缝衔接。
  • FP8 训练、speculative decoding、YaRN 长上下文支持。
  • VLM 训练、多轮 RL、sequence packing。

定位

NVIDIA NeMo 系产品的最大优势是 Megatron 原生 + FP8 + 千卡级稳定性，是在纯 NVIDIA 集群（H100 / B200 / GB200）上训练 100B+ 大模型的首选之一。但其代码栈对非 NVIDIA 用户（如 AMD、华为昇腾）友好度较低。

七、ART（OpenPipe）—— 简化的 Agent RL

ART: Agent Reinforcement Trainer

★ 9.5k v0.5.17 (2026.3) OpenPipe GRPO + SFT warmup

定位

ART 由 SaaS 公司 OpenPipe 维护，是"serverless agent RL"的代表。它的核心理念：让应用开发者通过 Python client 调用远程训练服务（可托管也可自部署），无需关心底层 Ray / vLLM / GPU 调度。

架构

  • Client-Server：用户写 agentic workflow（如 LangGraph、MCP agent），ART client 自动记录 trajectory，远程 server 完成 GRPO 训练。
  • 原生集成 W&B、LangGraph、MCP、Langfuse 等 agent 可观测性栈。
  • 支持 Qwen3、Llama 系列等 HuggingFace 兼容模型。

典型应用

邮件搜索 agent、工具使用、文档摘要、游戏（2048、Tic Tac Toe）。OpenPipe 宣称相对 baseline 训练成本降低 40%、训练快 28%。

局限

聚焦 Agent 场景，不适用于通用 RLHF 与大模型规模训练（>30B 全参较困难）。

八、其他重要框架

8.1 slime（THUDM）—— SGLang-Native 训练框架

★ 5.7k v0.2.4 (2026.3) Zhipu AI / THUDM（仓库 THUDM/slime） GRPO / PPO / DAPO / 多种 RLVR Megatron + SGLang

slime 是智谱 AI（GitHub 组织 THUDM）开源的 "Megatron + SGLang 原生"RL 训练框架。其设计哲学是：训练用 Megatron 拿到最优 throughput，rollout 用 SGLang 拿到最优生成速度（特别是 prefix cache 重用）。

  • 三模块架构：Training（Megatron）+ Rollout（SGLang + router）+ Data Buffer。
  • 原生异步、解耦训练与推理。
  • 广泛支持 Qwen3 全系（含 MoE）、DeepSeek-V3/R1、Llama3、GLM-4.5/4.6/4.7/5/5.1。
  • 官方 README 称其为 GLM-4.5 / 4.6 / 4.7 / GLM-5 / 5.1 全系背后的 RL 训练框架。

8.2 ROLL（阿里淘天 + 阿里 AI Engine）

★ 3.1k v0.2.1 (2026.3) Alibaba TAOBAO & TMALL PPO / GRPO / Reinforce++ / TOPR / RAFT++ / GSPO（20+）

ROLL（Reinforcement Learning Library）由阿里淘天集团联合 AI Engine 团队开源，强调大规模 RLVR + Agentic RL + 多模态三位一体：

  • Ray-based 多角色架构，支持千卡级集群。
  • 训练 backend 同时支持 DeepSpeed 与 Megatron-Core；推理支持 vLLM 与 SGLang。
  • 独特卖点：样本级异步 rollout、FP8 rollout、极致 offload/reload、原生 LoRA RL。
  • 提供 20+ RL 算法策略选择，是目前算法覆盖最广的开源框架之一。

8.3 DeepSpeed-Chat（Microsoft）—— 早期开拓者

Microsoft DeepSpeed 阶段化 RLHF（SFT → RM → PPO）

2023 年 4 月发布，是最早提供端到端 RLHF 三阶段（SFT、RM、PPO）的开源实现之一，奠定了 RLHF 在工程社区的普及基础。优势在于深度集成 DeepSpeed ZeRO-3 / Hybrid Engine。2025 年起更新缓慢，新算法（GRPO、DAPO）支持滞后于 OpenRLHF / veRL，目前主要作为 DeepSpeed 教学示例存在。

8.4 ColossalChat（HPC-AI / Colossal-AI）

HPC-AI Tech 完整 RLHF 流水线

基于 Colossal-AI 引擎的 RLHF 实现，主要面向 LLaMA-7B/13B 等中小模型。包含 SFT、RM、PPO + PTX（防遗忘）。早期（2023）有较大影响力，2026 年活跃度不高，但仍是 Colossal-AI 生态内训练 RLHF 的官方途径。

8.5 EasyR1（hiyouga）—— veRL 多模态衍生

★ 4.9k v0.3.2 (2025.9) hiyouga（LLaMA-Factory 作者） GRPO / DAPO / Reinforce++ / RLOO / GSPO / CISPO

EasyR1 是基于 veRL 的清洁 fork，专攻多模态（VLM）RL。原生支持 Qwen2-VL / Qwen2.5-VL / Qwen3-VL，并提供 padding-free training、LoRA RL、断点续训。作为最受欢迎的 veRL 衍生分支，已被多个研究项目（如 MM-UPT）使用。

8.6 RAGEN（Northwestern MLL Lab）—— Agent 自我进化

★ 2.7k RAGEN-2 (2026.4, arXiv:2604.06268) Northwestern MLL Lab StarPO (PPO/GRPO 变体)

RAGEN 是多轮 reasoning agent RL的代表性研究框架。核心算法 StarPO（State-Thinking-Actions-Reward Policy Optimization）面向轨迹级优化，内置 10 个环境（Sokoban、WebShop、FrozenLake、SearchQA、Lean、Bandit、Countdown、MetaMathQA、DeepCoder、Sudoku）。原论文（arXiv:2504.20073）识别出多轮 agent RL 的 Echo Trap 失稳模式（reward 方差塌缩 + 梯度尖峰），并以 StarPO-S 应对：基于 reward 方差的轨迹过滤、引入 critic、梯度稳定化。RAGEN-2（arXiv:2604.06268）进一步提出 template collapse 诊断（熵指标看不出的"固定模板"失稳，用互信息代理在线监测）与 SNR-Aware Filtering（按 reward 方差筛选高信号 prompt），缓解 agent RL 的稳定性问题。

8.7 Agent-R1（端到端 Agent RL）

~ 1.5k 0russwest0 / AgentR1 社区

面向工具增强 Agent 的端到端 RL 框架。核心理念是"Step-level MDP"：把 agent 的每一步交互（state, action, reward, next-state）形式化为严格 RL transition，规避基于纯文本的 retokenization drift 问题。论文 arXiv:2511.14460。

8.8 AgentFlow（in-the-flow agentic optimization）

Stanford 等 · Pan Lu 团队（仓库 lupantech/AgentFlow）

9.5 MoE RL 稳定性：三条路线

MoE 模型的 RL 后训练比 dense 模型脆弱得多，根因不在算法而在 router：top-k 专家选择是一个对数值差异极端敏感的离散决策。训练引擎（Megatron / FSDP）与推理引擎（vLLM / SGLang）的 kernel 实现、归约顺序与精度路径不同，在 dense 模型上只表现为 logits 的微小偏移，在 MoE 上却会让同一个 token 被送进完全不同的专家——微小数值 mismatch 被放大成行为级差异，名义上的 on-policy 训练实际在用另一个策略的样本算梯度。

问题有多严重（实测量化）：
  • R3 论文（Qwen3-30B-A3B，SGLang rollout + Megatron 训练）：同一输入下约 10% 的 router 在训推两侧选出不同专家，94% 的 token 至少有一层专家选择不一致。
  • GSPO 论文（Qwen3-30B-A3B-Base，48 层）：每做一次梯度更新，同一 rollout 样本就有约 10% 的激活专家发生翻转，且模型越深越严重——token 级 importance ratio 因此剧烈失真。
  • 训推 per-token KL：MoE 裸跑约 1.5×10⁻³，是 dense 模型（Qwen3-8B，6.4×10⁻⁴）的两倍以上；这个差值足以让裸 GRPO 在几十步内崩溃（R3 实验中 GRPO 第 60 步崩、GRPO+TIS 第 105 步崩）。
路线代表工作核心机制代价 / 限制生产使用者
① 系统侧对齐R3 / Rollout Routing Replay（arXiv:2510.11370）；DeepSeek Keep Routing + Keep Sampling Mask（V3.2 技术报告）记录推理引擎的专家路由（与采样截断掩码），训练时强制重放，从根上消除训推路由分歧需要推理引擎把 routing metadata 随轨迹回传——对现有框架是 breaking change；Megatron-LM 无原生支持DeepSeek（自 V3-0324 起用于生产 RL）
② 算法侧绕过GSPO（arXiv:2507.18071）importance ratio 升到序列级（长度归一化的几何平均），序列似然对个别 token 的路由翻转天然鲁棒，免 Routing Replayclip ε 必须比 token 级小 2-3 个数量级；clip 粒度粗，一刀切掉整条轨迹Qwen3 全系
③ 退回朴素 policy gradientStabilizing RL with LLMs（arXiv:2512.01374）严格 on-policy + IS 修正的 basic policy gradient；一阶分析给出 token 级代理目标的适用条件放弃 off-policy 样本复用的吞吐收益，用算力换稳定Qwen（30B MoE、数十万 GPU 时消融）

路线一：系统侧对齐 —— R3 / Keep Routing（DeepSeek 生产路线）

R3（Rollout Routing Replay，2025-10）在 rollout 时缓存推理引擎每层的二值专家选择掩码 Iinfer，训练时强制按它选专家，但 gating 权重仍由训练侧 router 的 logits 经 softmax 计算——所以梯度照常流向 router，router 仍然在学，只是"选谁"这个离散决策被钉死为与推理一致：

# Rollout 阶段（推理引擎：SGLang / vLLM）
for layer in moe_layers:
    mask[layer] = topk(router_infer(x))        # 二值专家选择掩码 I_infer
trajectory.metadata += mask                    # 随轨迹回传训练侧

# 训练阶段（训练引擎：Megatron / FSDP）
for layer in moe_layers:
    logits = router_train(x)                   # 正常前向：梯度仍流向 router
    gate   = softmax(logits[mask[layer]])      # gating 权重用训练侧 logits 计算
    y      = Σ gate_e · expert_e(x)  for e in mask[layer]   # 专家选择强制与推理一致

效果（Qwen3-30B-A3B）：训推 per-token KL 从 1.5×10⁻³ 降到 7.5×10⁻⁴，基本回到 dense 水平（6.4×10⁻⁴）；单 mini-step 设定下裸 GRPO 第 60 步崩、GRPO+TIS 第 105 步崩，GRPO+R3 跑满 180 步不崩，平均分比 TIS 高 5.59 分（71.83 vs 66.24）。R3 与算法正交：GSPO+R3（69.00）仍优于纯 GSPO（66.76）。R3 还可与 prefix KV cache 集成（routing mask 随 KV 一起缓存），适配多轮 agent 的增量 rollout。

DeepSeek 的同款机制叫 Keep Routing：V3.2 技术报告原话是"保存推理框架采样时使用的专家路由路径，训练时强制使用相同路径，确保优化的是同一批专家参数"，并明确"自 DeepSeek-V3-0324 起就一直用于我们的 RL 训练管线"——也就是说，在 GSPO 宣布"序列级 IS 可以免除 Routing Replay"的前后，DeepSeek 一直把路由重放当作生产必需品，系统侧对齐是被旗舰模型持续验证的独立路线，而非被算法取代的过渡 hack。V3.2 还配套 Keep Sampling Mask：top-p/top-k 采样会截断词表，使 πold 与 πθ 的动作空间不一致、违反 importance sampling 前提；解法是把采样时的截断掩码一并保存，训练时施加到 πθ 上，保证新旧策略共享同一动作子空间。

生态现状（这条路线最大的坑）：Megatron-LM 至今没有原生的 router replay 支持——NVIDIA/Megatron-LM issue #4168（2026-04 提交的 feature request，请求在 megatron.core.transformer.moe 中接受外部专家选择掩码）截至本文更新仍处 open 状态、无人认领。Hugging Face 2026-03 对 16 个开源 RL 库的普查同样确认：没有任何开源异步 RL 库实现了 Keep Routing / Keep Sampling Mask，因为两者都要求推理服务器把 routing/mask metadata 随响应回传，对每个框架的数据流都是 breaking change。想走路线一，目前基本意味着自己改训练引擎的 MoE forward。

路线二：算法侧绕过 —— GSPO 序列级 IS（Qwen3 路线）

GSPO 的判断是：既然 token 级 importance ratio 会被路由翻转污染，那就不要在 token 级算 ratio。它把 clip 对象换成长度归一化的序列级 ratio：

s_i(θ) = ( π_θ(y_i|x) / π_θold(y_i|x) )^(1/|y_i|)
       = exp( 1/|y_i| · Σ_t log[ π_θ(y_i,t | x, y_i,<t) / π_θold(y_i,t | x, y_i,<t) ] )

优化信号只依赖序列似然——个别 token 被路由进不同专家时，整条序列的似然基本稳定——于是 MoE 路由抖动不再直接进梯度，Qwen3 全系（含 235B MoE）借此免除了 Routing Replay。关键实操细节：1/|y| 的几何平均把序列 ratio 压缩到 1 附近极窄的区间，GSPO 原论文的 clip 区间是 left 3e-4 / right 4e-4，而 GRPO 是 0.2 / 0.27——相差约三个数量级。把 token 级的 ε=0.2 照搬到序列级，clip 将形同虚设，trust region 完全失效。

路线三：on-policy + IS 修正的朴素 policy gradient（Qwen 一阶分析收束）

Qwen《Stabilizing Reinforcement Learning with LLMs: Formulation and Practices》（2025-12）用一阶近似给这场争论立了统一标尺：token 级代理目标（PPO/GRPO 家族）能优化序列级奖励，仅当训推差异与 policy staleness 都足够小。这同时解释了三类手段各自在压什么——IS 修正压训推概率失配、clipping 控 staleness 漂移、Routing Replay 消 MoE 路由分歧。其 30B MoE、数十万 GPU 时的大规模消融结论：on-policy 场景下，basic policy gradient + IS 修正的训练稳定性最高；若要用 off-policy 复用样本提吞吐，clipping 与 Routing Replay 仍是必需品。换言之，路线三不否定路线一/二，而是给出了它们各自必要性的适用条件。

选型决策依据：
  1. 自研全栈、能改训练引擎 MoE forward（DeepSeek / Moonshot 式）→ 路线一最彻底：从根上消除路由失配，与任何算法正交可叠加；rollout 用 top-p/top-k 时加 Keep Sampling Mask。
  2. 用开源 Megatron / verl / slime 栈、改不动引擎 → 路线二 GSPO 是现实选择：纯算法侧、主流框架已内置开关；记得把 ε 调到 1e-4~1e-3 量级。
  3. 能接受同步 on-policy、吞吐换稳定 → 路线三：basic policy gradient + TIS/MIS 式 IS 修正。
  4. 无论哪条路线，监控必加两项：训推 per-token KL（MoE 健康参考线 ~7×10⁻⁴ 量级，冲到 10⁻³ 以上即危险）与 router 翻转率。MoE RL 崩溃排查顺序：先查路由一致性与训推 mismatch，再怀疑算法超参。

参考来源：Stabilizing MoE RL by Aligning Training and Inference Routers / R3（arXiv:2510.11370）；Group Sequence Policy Optimization（arXiv:2507.18071）；DeepSeek-V3.2 Technical Report（arXiv:2512.02556）；Stabilizing RL with LLMs: Formulation and Practices（arXiv:2512.01374）；Megatron-LM issue #4168；HF: Keep the Tokens Flowing（16 库普查）。

9.6 权重同步工程：从反模式到标准化协议

每个训练步之后，训练引擎的新权重必须灌回推理引擎才能继续 rollout。这条回灌链路是各框架性能差异的真正来源之一：做错了，几百亿参数一次同步要几十秒、吃掉整步时间的大头；做对了，万亿参数全量更新约 20 秒，常规规模可以压到几十毫秒，LoRA 场景亚毫秒。

反模式与量级表

头号反模式是逐参数 NCCL broadcast：对动辄数百上千个 weight tensor 逐个发起 broadcast 调用，per-call 启动开销远超数据传输本身。正确做法是 packed bucketed broadcast：把参数打包进约 1GB 的 uint8 桶，双 buffer + 双 CUDA stream 让"打包下一桶"与"广播当前桶"重叠流水，并用独立于训练通信的专用 NCCL communicator 避免与梯度通信互相阻塞。

同步路径机制典型延迟量级适用场景
CUDA IPC训推 colocate 同卡时共享显存句柄，零网络传输亚毫秒~毫秒colocate 部署 / 单机
packed bucketed NCCL broadcast~1GB uint8 桶 + 双 buffer 双 stream + 专用 communicator分桶后 ~20ms；未分桶 NCCL ~100-500ms（HF 16 库横评口径）分离式部署的主流路径
LoRA adapter-only 同步只传 adapter（rank 32 约 50MB），base 权重不动亚毫秒LoRA RL，热切换不停推理
checkpoint-engine 三段流水线H2D → Broadcast → Reload，按桶重叠~20s（1T 参数全量）万亿参数 / 千卡集群
文件系统落盘 + 重载（反模式）save checkpoint → 推理引擎重新加载秒~分钟级仅 debug / 容错兜底

万亿参数怎么办：Kimi checkpoint-engine 三段流水线

Moonshot 在 Kimi K2（1T 参数 MoE）的 colocate 同步 RL 中给出了事实标准实现：K2 技术报告披露其分布式 checkpoint engine 把 1T 全量权重更新做到 30 秒以内；开源的 checkpoint-engine 中间件官方口径是数千卡集群上约 20 秒完成 1T 参数更新。流水线分三段：

训练引擎（sharded weights，CPU/DRAM）
   │  ① H2D：把分片权重按桶搬上 GPU
   ▼
checkpoint-engine workers
   │  ② Broadcast：worker 间广播 → 写入与推理引擎共享的 CUDA IPC buffer
   ▼
推理引擎（vLLM / SGLang，并行切分可与训练侧完全不同）
   │  ③ Reload：各实例只拷贝自己 shard 需要的权重子集
   ▼
继续 rollout    （三段以桶为粒度流水重叠；先 gather 全部参数 metadata
                  决定桶大小并生成执行计划，ZeroMQ socket 控制传输；
                  显存不足时自动回退到串行执行）

官方 benchmark：Kimi-K2-Instruct（FP8）在 256×H20（TP16）上 broadcast 阶段 16.04s（8GiB 桶）、metadata gather 1.22s；GLM-4.5-Air（BF16，8×H800）3.47s；DeepSeek-V3.1（FP8，16×H20）10.19s。两个设计决策值得注意：其一，broadcast-everything 优于 transfer-what-you-need——K2 报告明言宁可多传数据，换取更低的同步协调开销与更简单的工程实现（全量广播 + 各实例自取所需）；其二，weight sync 机制被复用成容错与弹性机制：P2P 模式（基于 mooncake-transfer-engine）让故障重启或弹性扩容的新推理实例直接从现有实例的 CPU 内存拉权重入伙，不打扰在跑流量。

LoRA 场景：adapter-only 同步

LoRA RL 只训 adapter，同步也只该传 adapter：rank 32 的 adapter 约 50MB，相比广播数百 GB 的 base 权重，同步降到亚毫秒级且可热切换、不暂停推理——staleness 问题中"同步开销"这一分量近乎消失，这也是 LoRA RL 在中小规模 agentic 场景流行的隐性原因之一。HF 2026-03 的 16 库普查显示，支持 LoRA 的 13 个库中有 8 个实现了 adapter-only 同步；选型时这是一个值得直接询问的 checklist 项。

标准化：vLLM Native RL APIs 四阶段协议（2026-05）

长期以来每个 RL 框架都在 vLLM / SGLang 上自造 weight sync 轮子（monkey-patch、私有 collective_rpc）。2026-05-28 vLLM 发布 Native RL APIs，把这条链路收编为官方协议，核心是 weight transfer engine 抽象 + 四阶段同步协议：

阶段API职责
① 初始化init_weight_transfer_engine训练开始前建立 trainer ↔ inference worker 的通信组
② 开始更新start_weight_update每个训练步后通知 vLLM worker 准备接收权重
③ 传输update_weights传输全量或部分权重，可分块多次调用
④ 收尾finish_weight_update后处理（如在线量化到 FP8）

提供双后端：NCCL backend（训推在不同 GPU，走 packed bucketed broadcast）与 CUDA IPC backend（训推同卡，共享显存句柄），均支持 packed 实现以消除逐参数序列化开销。配套的 pause/resume keep mode 解决异步 RL 的经典两难：权重更新时既不 abort 在飞请求（浪费已生成 token）也不等它跑完（阻塞训练），而是保留请求状态原地暂停；针对 DP+EP 部署还给出两阶段防死锁协议——第一阶段 local pause（各引擎暂停调度新请求，但继续响应 DP wave 消息、参与集体 forward），第二阶段每 32 步通过 all-reduce 协调进入 global pause，保证所有 rank 一致地完成状态切换。落地验证：SkyRL 已用原生 API 跑通异步训练（Qwen3-1.7B）；Prime-RL 在 16 个 8×H200 节点的 P/D 分离部署上用 GLM-5.1-FP8 验证 100+ 训练步稳定收敛。这标志着 weight sync 从"各框架自造轮子"走向标准化协议，类似当年推理侧收敛到 OpenAI-compatible API。

权重同步工程检查清单：
  • colocate 同卡 → CUDA IPC；分离式部署 → packed bucketed NCCL broadcast（先确认所用框架版本不是逐参数 broadcast）。
  • 百亿级全量 → bucketed broadcast 数百 ms 可达；万亿级 → checkpoint-engine 式三段流水线，预留桶传输所需的额外显存（不足会回退串行、时间显著拉长）。
  • LoRA RL → 确认框架支持 adapter-only 同步，避免无谓的 base 权重广播。
  • 异步 + 在飞请求 → 用 vLLM keep mode / pause-resume，而非 abort；注意 prefix cache 中上一版策略算出的旧 KV 跨权重更新复用的正确性问题。
  • 把"同步耗时占整步比例"纳入监控：超过 ~10% 就该查桶大小、通信拓扑与是否误走了落盘路径。

十一、实践选型建议

11.1 按规模选

① 小规模研究 / 单机 / ≤ 7B

首选：TRL

  • 1 张 80G H100 + LoRA + GRPO 可跑 Qwen2.5-7B
  • Unsloth 内核 2-3x 加速
  • 文档最全，几十行代码完成 GRPO/DPO 实验

备选：ART（Agent 场景）、Axolotl（社区流水线）。

② 中等规模 / 7B-70B / 多机 8-64 卡

首选：OpenRLHF 或 veRL

  • OpenRLHF：代码量小、学习曲线友好、Ray 调度成熟、Async / Agent 强。
  • veRL：性能与算法覆盖最强，但代码量大。

VLM 场景：EasyR1（基于 veRL 的多模态分支）。

需要异步：AReaL（全异步 PPO 范式）。

③ 大规模 / ≥ 70B 全参 / MoE / 千卡

首选：veRL + Megatron

  • 已验证可扩展到 DeepSeek-V3 / R1 级 671B MoE
  • EP + PP + TP 全套并行，FSDP2 hybrid 也成熟

NVIDIA 纯 H100 / B200 集群：NeMo-RL（FP8 + 6D parallelism）。

SGLang 路线：slime（智谱 GLM 系列实战检验）、ROLL。

④ Agent / Multi-turn RL

研究 / 复杂环境：RAGEN（StarPO + 10 内置环境）、Agent-R1（step-level MDP）、AgentFlow（Planner 优化）。

工程落地：OpenRLHF（多轮 agent 接口）、AReaL（异步对长 trajectory 友好）、ART（client-server，省运维）。

提示：Agent RL 训练对 rollout 长尾极敏感，必选 async rollout，否则同步等待会让 GPU 利用率长期低于 40%。

11.4 常见坑

  • 选 framework 不等于性能保证：相同框架，参数（micro batch、rollout n、KL 系数、advantage 归一化）配置错误，吞吐与最终效果可能差 5x 以上。先看官方 recipe。
  • 不要在 2026 年新建项目还用 DeepSpeed-Chat / ColossalChat：算法滞后，性能落后于 OpenRLHF / veRL 5-20x。
  • SGLang vs vLLM 不绝对：长 prompt + 多 sample → SGLang；常规 → vLLM。veRL / ROLL 支持两个 backend，建议两者都 benchmark。
  • Async RL 引入 staleness，需要做 importance correction（如 TIS、staleness-aware PPO），否则会发散。AReaL 与 OpenRLHF 都提供了开箱即用的方案。
  • LoRA RL 的合并陷阱：在多个角色（actor、ref）共享 base + 切 adapter 时，注意 vLLM 的 LoRA 加载速度，否则会成为新瓶颈。
  • MoE RL 崩溃先查路由而非超参：训推引擎对同一输入约 10% 的 router 会选出不同专家，裸 GRPO 跑 MoE 可能几十步就崩。三条对策（Keep Routing/R3、GSPO、on-policy+IS）与决策依据见 9.5 节；监控训推 per-token KL 与 router 翻转率。
  • 权重同步是隐形瓶颈：逐参数 broadcast 是反模式，注意确认框架走的是 packed bucketed broadcast / CUDA IPC / adapter-only 路径；机制与量级详见 9.6 节。

结语

2026 年的 RLHF / RL 后训练框架生态，已经从 2023 年 DeepSpeed-Chat / TRL 的"单点突破"演化为百花齐放、各擅胜场的格局。如果说要用一句话概括当前的技术地形：

"TRL 用于学习与小规模、OpenRLHF 用于优雅工程、veRL 用于规模与极限、AReaL 用于异步与 Agent、NeMo-RL / slime 用于纯 NVIDIA / SGLang 路线、ROLL / EasyR1 用于多模态与算法广度、ART / RAGEN 用于 Agent 创新。"

从更宏观的视角看，几个明确的技术趋势正在形成：

  1. 异步范式正在成为默认。同步 RL 在长 CoT 与多轮 agent 场景下浪费严重，2.5-3x 的异步加速对每一个生产团队都难以拒绝。
  2. 训练-推理协同进入"零拷贝时代"。3D-HybridEngine、Megatron-Core 与 vLLM/SGLang 的权重热同步将训练效率推到一个全新平台；2026 年 vLLM Native RL APIs 进一步把权重同步收编为标准化协议（机制详见 9.6 节）。
  3. 算法民主化。GRPO 之后，DAPO / GSPO / DrGRPO / TIS 等改进算法在主流框架中以"开关"形式提供，研究门槛持续下降。
  4. Agent RL 成为新战场。从单轮 RLHF 到多轮 tool-use、search、code agent，整个范式正在重塑。
  5. 开源中国队的领导地位。veRL（字节）、AReaL（蚂蚁/清华）、slime（智谱）、ROLL（阿里）、EasyR1（hiyouga）—— 五大头部国产框架在算法、规模、性能上已经处于全球最前沿。

对于任何在 2026 年准备进入 RL 后训练的团队或个人研究者：选择框架的决策永远是"团队规模 × 模型规模 × 算法路线 × 集群类型"四维联合的结果。希望这份综述能成为你绘制自己技术地图的起点。
